4. Async Materialization

The AsyncExecutor is the heart of The parallel materialization system. It transforms the original sequential Executor (which resolved inputs one-by-one) into a concurrent engine that exploits DAG structure for parallelism. This section describes three key mechanisms: broadcast-channel deduplication, deadlock-free semaphore placement, and input assembly for deterministic ordering.

4.1 The Materialization Pipeline

When a client requests Get(addr), the AsyncExecutor follows an 11-step pipeline. The steps are not all executed for every request — the first three are early-exit checks that short-circuit the common cases (cache hits, leaf blobs, and duplicate in-flight requests). Only when all three checks miss does the executor enter the full producer path (steps 4–11).

The pipeline is structured around three phases:

  • Fast paths (steps 1–2): Cache and leaf checks return immediately with zero computation. These handle the majority of requests in a warm system.
  • Deduplication (step 3): If another task is already computing this address, the current task subscribes to a broadcast channel and waits — no redundant work is performed.
  • Producer path (steps 4–11): The task registers itself as the producer, resolves inputs in parallel, acquires a semaphore for compute, executes the recipe function, broadcasts the result to any waiting subscribers, and caches the output. The critical ordering constraint: input resolution (step 6) happens before semaphore acquisition (step 7), which prevents deadlocks in diamond-shaped DAGs (§4.3).

4.2 Broadcast-Channel Deduplication

When multiple clients request the same unmaterialized CAddr concurrently, only one task should compute it — the others should wait for the result. Without deduplication, N concurrent requests for the same address would trigger N independent compute tasks, wasting CPU and potentially producing redundant I/O against the content store.

The deduplication mechanism uses a shared in_flight map that tracks which addresses are currently being computed. The map's values are Tokio broadcast::Sender handles — multi-producer, multi-consumer channels that clone the result to every subscriber:

  • First request (producer): Finds no entry in in_flight, creates a broadcast channel, inserts it into the map, and proceeds to compute. When the result is ready, it sends once on the channel — all subscribers receive a clone.
  • Subsequent requests (subscribers): Find an existing entry in in_flight, call tx.subscribe() to get a receiver, drop the lock, and await the result. No computation is performed — the subscriber simply waits for the producer to finish.

A subtle but important detail: the lock on in_flight is held only for the duration of the map lookup (step 3) or insertion (step 4), then immediately dropped. Subscribers never hold the lock while waiting for the result — this prevents the lock from becoming a bottleneck under high concurrency. 

// In AsyncExecutor struct
in_flight: Arc<Mutex<HashMap<CAddr, broadcast::Sender<Result<Bytes>>>>>

// Step 3: Deduplication check
{
    let in_flight = self.in_flight.lock().await;
    if let Some(tx) = in_flight.get(&addr) {
        // Another task is computing this addr — subscribe to its channel
        let mut rx = tx.subscribe();
        drop(in_flight);  // Release lock before waiting
        
        let result = match rx.recv().await {
            Ok(result) => result,
            Err(_) => Err(DerivaError::ComputeFailed(
                "producer task failed or was cancelled".into()
            )),
        };
        return result;  // Return result from producer
    }
}

// Step 4: Register as producer
let (tx, _rx) = broadcast::channel(self.config.dedup_channel_capacity);
{
    let mut in_flight = self.in_flight.lock().await;
    in_flight.insert(addr, tx.clone());
}

// ... compute ...

// Step 9: Broadcast result to all subscribers
let _ = tx.send(result.clone());

// Step 10: Cleanup
self.in_flight.lock().await.remove(&addr);

Why broadcast channels? Tokio's broadcast channel supports multiple receivers and clones the message to each. This is a natural fit for deduplication: the producer sends once, and all subscribers receive a copy. Alternative designs using oneshot channels would require N channels for N subscribers, complicating both allocation and cleanup. With broadcast, the subscriber count is unbounded and dynamic — new subscribers can attach at any point before the send.

Capacity tuning: The channel capacity (default: 16) determines how many results can be buffered if subscribers are slow to receive. In practice, subscribers are fast (they clone a Bytes handle, which is a reference-counted pointer — not a deep copy), so a small capacity suffices. If the channel fills, send() drops the oldest message, but this is harmless because the producer also returns the result directly to its own caller.

Error propagation: If the producer task panics or is cancelled, the broadcast channel is dropped, and all subscribers receive a RecvError. This is converted into a DerivaError::ComputeFailed, ensuring that subscribers never hang indefinitely — they either receive the result or an explicit error.

4.3 Deadlock-Free Semaphore Placement

A semaphore limits concurrent compute operations to prevent resource exhaustion (CPU, memory). The critical design question is not whether to use a semaphore, but when to acquire it. The ordering of semaphore acquisition relative to input resolution determines whether the system can deadlock.

The problem arises in diamond-shaped DAGs — the most common non-trivial topology. Consider recipe D = h(B, C) where both B and C depend on leaf A. Three tasks are spawned: T1 for B, T2 for C, and T3 for D. If the semaphore is acquired before input resolution, T3 holds a permit while waiting for B and C to complete. But B and C need permits to compute. With max_concurrency=2, all three permits are consumed (T1, T2, T3), and T1/T2 cannot start because T3 holds the last permit — classic circular wait.

The solution is to acquire the semaphore after input resolution. A task waits for all its inputs without holding any permit, then acquires a permit only for the brief compute step. This eliminates circular wait: permits are held only during actual computation, never during dependency resolution.

The _permit variable holds the semaphore permit via RAII — when it goes out of scope (after step 8), the permit is automatically released. This ensures permits are held for the minimum duration: only during compute, never during input resolution, broadcasting, or cleanup. The guarantee is structural, not convention-based — Rust's ownership system enforces it at compile time.

4.4 Parallel Input Resolution

Step 6 is where the materialization pipeline extracts parallelism from the DAG. Each recipe declares its inputs as a list of content addresses. The sequential executor resolved these one at a time — a for loop that blocked on each materialize() call before starting the next. For a recipe with N independent inputs, this serializes N independent computations into a chain, wasting all available concurrency.

The parallel executor replaces this loop with try_join_all: all N input futures are spawned simultaneously, and the executor waits for the entire batch to complete. The key insight is that inputs to a single recipe are always independent — they share no mutable state and cannot depend on each other (the DAG is acyclic). This makes parallel resolution a pure win: no synchronization overhead, no ordering constraints, just concurrent execution. 

let futures: Vec<_> = recipe.inputs.iter()
    .map(|input_addr| self.materialize(*input_addr))
    .collect();
let input_bytes = futures::future::try_join_all(futures).await?;

try_join_all provides fail-fast semantics: if any input fails, the entire join returns immediately with that error, and remaining futures are dropped (canceled). This avoids wasting compute on inputs that will never be used. The alternative — join_all without the try_ prefix — would wait for all futures even after a failure, collecting errors into a Vec<Result>. Since a single failed input makes the entire recipe uncomputable, fail-fast is the correct choice.

For wider fan-in patterns, the speedup scales linearly. A recipe with 10 independent inputs, each taking 1 second, completes in ~1 second with parallel resolution versus 10 seconds sequentially — a 10× improvement. This is common in aggregation recipes that combine many leaf computations.

The fail-fast behavior of try_join_all is equally important for error handling. If input 3 of 10 fails after 0.5 seconds, the remaining 9 futures are dropped immediately rather than running to completion. This prevents wasted work: since a single failed input makes the recipe uncomputable, there is no value in resolving the remaining inputs. The alternative — join_all without the try_ prefix — would wait for all 10 futures and collect errors into a Vec<Result>, wasting up to 9 seconds of compute on inputs that will never be used.

4.5 Input Assembly: Deterministic Ordering

Parallel input resolution introduces a subtle correctness requirement: the compute function must receive its inputs in the same order every time, regardless of which input finishes first. In a content-addressed system, the output address is a hash of the computation — if the same recipe produces different byte orderings on different runs, it produces different content addresses, breaking the fundamental invariant that identical computations yield identical addresses.

try_join_all solves this through positional slot collection. Internally, it allocates a Vec with one slot per future, indexed by position. As futures complete (in any order), each result is placed into its pre-assigned slot. The returned Vec<Bytes> always matches the order of the input iterator, not the order of completion. This is a property of the combinator itself — no additional sorting or reordering is needed.

The following visualization demonstrates this on D = h(B, C). Two runs resolve B and C in different orders, but the assembled input vector is identical. Click to toggle between scenarios:

This guarantee is what makes parallel resolution safe in a content-addressed system. If h(bytes_B, bytes_C) and h(bytes_C, bytes_B) produced different outputs (as most functions do), parallel execution would be non-deterministic — the same recipe would produce different content addresses depending on scheduling. Positional slot collection eliminates this: the compute function always sees [bytes_B, bytes_C], producing the same CAddr regardless of which future completed first.

4.6 Comparison with Original Executor

The original Executor was synchronous and single-threaded. It resolved inputs with a blocking for loop — each materialize() call blocked the thread until that input was fully computed before starting the next. There was no deduplication: if two recipes shared an input, it was computed twice. There was no concurrency limit: the call stack depth was the only bound on resource usage. 

// Original: Sequential, blocking, no dedup
let mut input_bytes = Vec::new();
for input_addr in &recipe.inputs {
    let bytes = self.materialize(*input_addr)?;  // Blocking
    input_bytes.push(bytes);
}
let output = func.execute(&input_bytes, &recipe.params)?;

The AsyncExecutor replaces every aspect of this design. Inputs are resolved concurrently via try_join_all. Broadcast channels deduplicate concurrent requests to the same address. A semaphore bounds resource usage. The compute step runs on a blocking thread pool via spawn_blocking to avoid starving the async runtime. 

// Current: Parallel, async, deduplicated, bounded
let futures: Vec<_> = recipe.inputs.iter()
    .map(|input_addr| self.materialize(*input_addr))
    .collect();
let input_bytes = futures::future::try_join_all(futures).await?;
let _permit = self.semaphore.acquire().await?;
let output = self.execute_verified(&addr, func, input_bytes, &recipe.params).await?;

The speedup from parallel input resolution scales with fan-in width: for N inputs each taking T seconds, the original executor takes N×T while the async executor takes T (the slowest input). But the comparison extends beyond raw speed. Broadcast-channel deduplication eliminates redundant computation across concurrent callers. The semaphore prevents resource exhaustion that the original executor was vulnerable to (deep DAGs could exhaust the call stack). And spawn_blocking isolates CPU-bound compute from the async runtime, preventing a long-running recipe from starving I/O tasks.

4.7 Deduplication Effectiveness

Broadcast-channel deduplication is most effective when multiple clients request the same CAddr within a short time window (milliseconds to seconds). Real-world scenarios:

  • Batch requests: A web service receives 100 requests for the same derived dataset (e.g., "latest sales report"). Without dedup, 100 compute tasks execute. With dedup, 1 compute task executes, 99 subscribe.
  • Retry storms: A client retries a failed request 10 times in rapid succession. Without dedup, 10 compute tasks. With dedup, 1 compute task (if the first is still in-flight when retries arrive).
  • Shared dependencies: Two recipes depend on the same expensive input. If both recipes are materialized concurrently, the shared input is computed once.

Evaluation (§9) measures deduplication effectiveness: for N concurrent requests to the same CAddr, compute invocations should be exactly 1 (100% dedup rate).

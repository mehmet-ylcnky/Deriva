2. Background

Parallel execution of dependency graphs is a well-studied problem in build systems, dataflow engines, and task schedulers. However, existing systems treat storage as external — they parallelize computation but don't own the data. This section positions Deriva's approach against three categories of related work and explains why DAG-aware parallelism within the storage layer is fundamentally different.

2.1 Critical Path Analysis in Dependency Graphs

A dependency DAG defines a partial order: task T must complete before task U if U depends on T's output. The critical path is the longest chain of dependent tasks — it determines the minimum wall-clock time even with infinite parallelism. Consider this diamond DAG:

The key insight: independent branches can be computed concurrently. When D requests inputs B and C, and both are unmaterialized, the system should spawn two parallel tasks rather than computing them sequentially. This is structural parallelism — the DAG topology dictates the schedule.

2.2 Build Systems: Bazel and Ninja

Bazel constructs an action graph where each action (compile, link, test) depends on input files and produces output files. Bazel's scheduler detects independent actions and executes them in parallel, respecting the action graph's partial order [1]. However:

  • External cache: Bazel's remote cache is a separate service (typically backed by S3 or a dedicated cache server) [1]. When a source file changes, Bazel must recompute all transitive dependents — it has no way to query the cache for "which cached actions depend on this file?" The cache is a black box that stores (action_hash → output_bytes) mappings.
  • No GC: Cached actions accumulate indefinitely. Bazel provides bazel clean to wipe the entire cache, but there's no selective GC based on action reachability [2]. If an action's inputs are deleted, the cached output becomes orphaned but remains in storage.
  • Parallelism without ownership: Bazel parallelizes action execution but doesn't own the storage layer. It can't perform structural invalidation (walk reverse edges to find dependents) or structural GC (mark-and-sweep based on action reachability) because the cache doesn't expose the action graph.

Ninja is similar but more minimal: it reads a build graph from a .ninja file and executes commands in parallel [3]. Ninja has no remote cache — it relies on filesystem timestamps for incremental builds. When a source file's mtime changes, Ninja recomputes all transitive dependents by walking the build graph. This is structural invalidation, but it's limited to the local filesystem and doesn't apply to distributed caches.

2.3 Dataflow Engines: Spark, Dask, Ray

Apache Spark represents computations as RDD (Resilient Distributed Dataset) lineage graphs. Each RDD transformation (map, filter, join) creates a new RDD with a pointer to its parent. When an action (collect, count) is triggered, Spark's DAG scheduler constructs a task graph and executes stages in parallel [4]. However:

  • Ephemeral lineage: RDD lineage exists only for the duration of a Spark session. When the session ends, the lineage is lost. If an input DataFrame changes, Spark has no way to identify which cached RDDs are stale — the user must manually call unpersist() or restart the session [5].
  • External storage: Spark's cache() stores RDDs in memory or on disk, but this cache is separate from the input data (which lives in HDFS, S3, etc.). Spark can't perform structural invalidation because it doesn't own the input storage — it can't detect when a Parquet file in S3 has been updated.
  • No content-addressing: Cached RDDs are identified by RDD ID (an integer), not by their computation. If two Spark jobs compute the same transformation on the same input, they produce separate cached RDDs — no deduplication.

Dask and Ray have similar limitations: they parallelize task graphs but treat storage as external. Dask's task graph is ephemeral (rebuilt on each compute() call) [6], and Ray's object store is a distributed shared memory without computation awareness — objects are identified by random IDs, not by how they were computed [7].

2.4 Cache Invalidation: Redis and Memcached

Traditional caches (Redis, Memcached) provide SET/GET/DELETE operations but have no notion of dependencies [8]. When a cached value becomes stale, the application must manually delete it. For transitive invalidation (e.g., "delete all cached values that depend on key X"), the application must maintain a separate dependency graph and walk it to find dependents. This is error-prone and requires external orchestration.

Some systems (e.g., Varnish, Squid) support cache tags: each cached entry is tagged with one or more labels, and invalidating a tag clears all entries with that tag [9]. However, tags are flat — they don't capture transitive dependencies. If A depends on B and B depends on C, invalidating C requires manually tagging both B and A with C's tag, which doesn't scale to deep DAGs.

2.5 Storage GC: Git and HDFS

Git's gc performs mark-and-sweep on the object store: it marks all objects reachable from refs (branches, tags), then sweeps unreachable objects [10]. However, Git's objects are content-addressed (by SHA-1 hash of raw data), not computation-addressed. Git has no notion of "this blob was derived from these inputs via this function" — it only knows "this blob has this hash."

HDFS block scanner periodically verifies block checksums and replicates under-replicated blocks [11]. It's a replica-aware GC (ensures replication factor is maintained) but not computation-aware (doesn't know which blocks are derived vs raw, or which blocks are orphaned because their computation recipe was deleted).

2.6 The Gap: No System Combines DAG-Aware Parallelism with Storage Ownership

The common thread: existing systems parallelize computation but don't own the storage. This creates three fundamental limitations:

The table above highlights Deriva's position: among the surveyed systems, none combines all five capabilities. Build systems (Bazel, Ninja) have parallel execution and some form of caching but lack structural invalidation and GC. Dataflow engines (Spark, Dask) have parallel execution but ephemeral caches. Caches (Redis) have persistence but no computation or dependency awareness. Content-addressed stores (Git) have content addressing and GC but no computation. Deriva unifies all five by making the storage layer own the dependency DAG.

Why External Storage Breaks Structural Operations

  1. No structural invalidation: The cache doesn't know the dependency graph, so it can't walk reverse edges to find dependents. Invalidation requires external orchestration (the application must maintain a separate dependency graph and manually delete stale entries).
  2. No structural GC: The storage system doesn't know which data is derived vs raw, or which derived data is reachable from active computations. GC is either time-based (delete after N days) or manual (user runs a cleanup script).
  3. No deduplication across sessions: Build systems and dataflow engines rebuild the dependency graph on each run. If two runs compute the same transformation on the same input, they produce separate cached results — no sharing.

Deriva's approach is fundamentally different: the storage layer owns the dependency DAG. The persistent DAG is not an external metadata store — it's the primary index for both recipes and cached materializations. This enables three structural operations that no external orchestrator can provide:

  • Parallel materialization (§4): When resolving a recipe's inputs, the executor queries the DAG to detect independent branches and spawns concurrent tasks. The DAG is the source of truth for "which inputs are unmaterialized and can be computed in parallel."
  • Cascade invalidation (§6): When a leaf changes, the invalidator walks the DAG's reverse edges (dependents) to find all transitive consumers. No external dependency tracking needed — the DAG is the dependency graph.
  • Garbage collection (§7): The GC computes the live set as dag.live_addr_set() (all CAddrs reachable from recipes) ∪ pinned addrs. Orphaned blobs are those in BlobStore but not in the live set. This is computation-aware GC — the system knows which blobs are derived and which are orphaned.

The rest of this paper describes how these three features are implemented on top of the persistent DAG. The key insight: the DAG is not just a performance optimization — it's the structural foundation that makes computation-addressed storage viable for real-world workloads.

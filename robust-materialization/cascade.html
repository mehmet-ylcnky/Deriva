6. Cascade Invalidation

When a leaf changes (e.g., new data pushed with the same semantic meaning but different content), all recipes that depend on it — directly or transitively — become stale. The original invalidate(addr) RPC removed only the specified address from cache, leaving transitive dependents stale. This caused silent correctness violations: clients would receive cached results computed from old data. Cascade invalidation solves this by traversing reverse DAG edges to find all dependents and evicting them atomically.

6.1 The Problem: Silent Stale Data

When a leaf changes — new data pushed with the same semantic key but different content — every recipe that consumed the old content becomes stale. In a content-addressed system, the cache maps CAddrs to computed bytes; if the input bytes change, the cached output is no longer valid. The danger is not that the cache returns an error, but that it returns the wrong answer silently. The cached bytes are perfectly valid bytes — they are simply the result of a computation that no longer reflects the current inputs.

The original invalidate(addr) RPC removed only the specified address from cache. This was correct for the single node but catastrophically incomplete for the graph: any recipe that depended on the invalidated leaf — directly or through any chain of intermediate recipes — continued to serve cached results computed from the old data. Clients received stale outputs with no indication that anything had changed. In data pipelines, this class of bug is particularly insidious because the outputs look correct — they have the right schema, the right types, plausible values — but they reflect a state of the world that no longer exists.

This is fundamentally broken for any real pipeline. Without cascade invalidation, developers must manually track all transitive dependents and invalidate them individually — error-prone for small DAGs, impractical for large ones. The failure mode is particularly dangerous because it is silent: no error is raised, no cache miss occurs, the system simply returns confidently wrong data. The next sections describe how cascade invalidation solves this by traversing reverse DAG edges to find and evict all affected nodes atomically.

6.2 Cascade Invalidation Architecture

The CascadeInvalidator is structured as a three-stage pipeline: traverse, plan, evict. The DAG Reader walks reverse edges from the invalidation root to discover all transitive dependents. The Invalidation Planner applies the configured policy — immediate eviction or dry-run analysis — to the discovered set. The Cache Writer performs batch removal of all affected entries. This separation keeps each stage independently testable and allows the dry-run mode (§6.9) to share the traversal and planning logic without touching the cache.

The result of every cascade operation is an InvalidationResult containing the eviction count, traversal count, maximum depth reached, bytes reclaimed, the full list of evicted CAddrs, and the wall-clock duration. These metrics serve two purposes: operational observability (how large was the blast radius, how long did it take) and correctness auditing (which specific addresses were evicted, allowing verification that the cascade reached all expected dependents).

pub struct InvalidationResult {
    pub evicted_count: u64,
    pub traversed_count: u64,
    pub max_depth: u32,
    pub bytes_reclaimed: u64,
    pub evicted_addrs: Vec<CAddr>,
    pub duration: Duration,
}

The separation into three stages — traverse, plan, evict — keeps each concern independently testable. The DAG Reader can be verified against known graph structures without touching the cache. The Planner can be tested for policy correctness without performing any I/O. The Cache Writer can be benchmarked for batch eviction throughput in isolation. This decomposition also enables the dry-run mode described in §6.9: the same traversal and planning logic runs, but the Cache Writer stage is skipped entirely, producing an InvalidationResult with evicted_count reflecting what would be evicted without actually mutating the cache.

6.3 Invalidation Policies

The cascade behavior is controlled by a CascadePolicy enum with three variants. This is a deliberate design choice over a boolean flag: three distinct policies are clearer than combinations of cascade: bool and dry_run: bool, and the enum prevents invalid states like "cascade off but dry-run on" which would be meaningless. 

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum CascadePolicy {
    /// Only invalidate the exact addr (original behavior)
    None,

    /// Traverse all transitive dependents and evict immediately
    Immediate,

    /// Traverse but don't evict — return what *would* be evicted
    DryRun,
}

None preserves the original single-address eviction behavior. This exists for cases where the caller knows there are no dependents — invalidating a leaf that was inserted but never consumed by any recipe, or evicting a single cache entry for space reclamation rather than correctness. Using None in these cases avoids the cost of reverse-edge traversal entirely.

Immediate is the recommended default for any invalidation triggered by data changes. It traverses reverse edges from the invalidation root, collects all transitive dependents into a set, and evicts them in a single batch operation. The batch eviction is important: removing entries one at a time would allow concurrent readers to observe partially-invalidated states where some dependents are evicted but others still serve stale data.

DryRun performs the identical traversal and produces the same InvalidationResult — including the full list of CAddrs that would be evicted, the traversal depth, and the estimated bytes — but skips the cache mutation step. This is used for impact analysis: before invalidating a heavily-depended-upon leaf, operators can preview the blast radius to understand how many downstream recipes will need recomputation. In large DAGs, this preview can be the difference between a routine cache refresh and an unexpected recomputation storm.

6.4 Reverse Edge Traversal with Depth Tracking

The persistent DAG (§3) stores reverse edges in a dedicated reverse tree, mapping each CAddr to the set of recipes that depend on it. This reverse index is maintained incrementally — every add_recipe call inserts both forward edges (recipe → inputs) and reverse edges (input → recipe) in the same transaction. The cost of maintaining reverse edges is paid at write time so that invalidation-time traversal is a simple lookup chain with no graph reconstruction.

The transitive_dependents_with_depth method performs BFS from the invalidation root, collecting every reachable dependent along with its depth (hop count from the root). The visited set ensures each node is processed exactly once, which is critical for diamond-shaped DAGs where multiple paths converge on the same dependent — without deduplication, the same node would be evicted multiple times and appear multiple times in the result. 

pub fn transitive_dependents_with_depth(&self, addr: &CAddr)
    -> (Vec<(CAddr, u32)>, u32)
{
    let mut result = Vec::new();
    let mut visited = HashSet::new();
    let mut queue = VecDeque::new();
    let mut max_depth: u32 = 0;

    // Seed queue with direct dependents at depth 1
    for dep in self.direct_dependents(addr) {
        if visited.insert(dep) {
            queue.push_back((dep, 1u32));
        }
    }

    // BFS traversal
    while let Some((current, depth)) = queue.pop_front() {
        max_depth = max_depth.max(depth);
        result.push((current, depth));

        for dep in self.direct_dependents(¤t) {
            if visited.insert(dep) {
                queue.push_back((dep, depth + 1));
            }
        }
    }

    (result, max_depth)
}

BFS is chosen over DFS for two reasons. First, it naturally produces depth information — the number of hops from the invalidation root to each dependent — which feeds into the max_depth field of InvalidationResult. This depth metric is valuable for operational monitoring: a cascade that reaches depth 15 in a DAG expected to have depth 3 suggests an unexpected dependency chain worth investigating. Second, BFS processes nodes in level order, which makes the eviction list more intuitive for debugging — dependents appear in order of distance from the root rather than in an arbitrary DFS traversal order. Both algorithms are O(V + E) in time and space, so the choice has no performance cost.

6.5 Cascade Invalidation Flow

The cascade invalidation flow involves four components communicating in sequence: the Client initiates a CascadeInvalidate RPC, the DerivaService delegates to the CascadeInvalidator, which queries the DAG for transitive dependents and then issues a batch eviction to the Cache. The entire flow is synchronous from the client's perspective — the RPC blocks until all evictions are complete and returns an InvalidationResult with the full accounting of what was removed.

This synchronous design is intentional. An asynchronous "fire and forget" invalidation would return immediately but leave a window where concurrent reads could still hit stale cache entries. By blocking until eviction completes, the caller has a strong guarantee: after the RPC returns, no subsequent get() call will return data computed from the old leaf value. The trade-off is latency — large cascades may take tens of milliseconds — but correctness is prioritized over speed for invalidation operations.

This is a stronger guarantee than eventual consistency — it is immediate consistency for the invalidated set. The InvalidationResult returned to the client contains the full eviction accounting, allowing the caller to verify that the cascade reached the expected scope and to log the blast radius for operational review.

6.6 Implementation: Sync and Async Variants

The CascadeInvalidator is implemented as a stateless struct with a single invalidate method that takes all dependencies as parameters — the DAG, the cache, the root address, and the policy. This stateless design means there is no invalidator instance to manage, no lifecycle to coordinate, and no risk of the invalidator holding stale references to a previous DAG or cache version. Each call is self-contained.

The method dispatches on the CascadePolicy enum. The None branch performs a single cache.remove(root) — the original behavior, preserved for backward compatibility and for cases where the caller knows no dependents exist. The Immediate branch follows the three-stage pipeline: traverse via transitive_dependents_with_depth, build the eviction list (optionally including the root itself), and call cache.remove_batch for atomic eviction. The DryRun branch performs the same traversal but replaces batch eviction with cache.contains checks, counting what would be evicted without mutating the cache. 

pub struct CascadeInvalidator;

impl CascadeInvalidator {
    pub async fn invalidate(
        dag: &Arc<PersistentDag>,
        cache: &Arc<SharedCache>,
        root: &CAddr,
        policy: CascadePolicy,
        include_root: bool,
        detail_addrs: bool,
    ) -> InvalidationResult {
        let start = Instant::now();

        match policy {
            CascadePolicy::None => {
                let was_present = cache.remove(root).await.is_some();
                InvalidationResult {
                    root: *root,
                    evicted_count: if include_root && was_present { 1 } else { 0 },
                    traversed_count: 0,
                    max_depth: 0,
                    bytes_reclaimed: 0,
                    evicted_addrs: Vec::new(),
                    duration: start.elapsed(),
                }
            }

            CascadePolicy::Immediate => {
                let (dependents, max_depth) =
                    dag.transitive_dependents_with_depth(root);
                let traversed_count = dependents.len() as u64;

                let mut to_evict: Vec<CAddr> =
                    dependents.into_iter().map(|(addr, _)| addr).collect();
                if include_root {
                    to_evict.insert(0, *root);
                }

                let (evicted_count, bytes_reclaimed, evicted_addrs) =
                    cache.remove_batch(&to_evict).await;

                InvalidationResult {
                    root: *root,
                    evicted_count,
                    traversed_count,
                    max_depth,
                    bytes_reclaimed,
                    evicted_addrs: if detail_addrs { evicted_addrs } else { Vec::new() },
                    duration: start.elapsed(),
                }
            }

            CascadePolicy::DryRun => {
                let (dependents, max_depth) =
                    dag.transitive_dependents_with_depth(root);
                let traversed_count = dependents.len() as u64;

                let mut would_evict = Vec::new();
                if include_root && cache.contains(root).await {
                    would_evict.push(*root);
                }
                for (addr, _) in &dependents {
                    if cache.contains(addr).await {
                        would_evict.push(*addr);
                    }
                }

                InvalidationResult {
                    root: *root,
                    evicted_count: would_evict.len() as u64,
                    traversed_count,
                    max_depth,
                    bytes_reclaimed: 0,
                    evicted_addrs: if detail_addrs { would_evict } else { Vec::new() },
                    duration: start.elapsed(),
                }
            }
        }
    }
}

Two parameters control the result granularity. include_root determines whether the invalidation root itself is evicted — set to true when a leaf's content has changed, false when only dependents need eviction (e.g., when a compute function is updated but the leaf data is unchanged). detail_addrs controls whether the full list of evicted CAddrs is included in the result; for large cascades this list can be substantial, so it is omitted by default and enabled only when the caller needs it for auditing or debugging.

The async variant shown here works with SharedCache (which wraps async I/O). A sync variant with identical logic exists for PersistentDag's own sled-backed cache, where all operations are blocking. Both variants share the same traversal logic — the only difference is whether cache operations use .await or block directly.

6.7 Batch Cache Eviction

The remove_batch method acquires the cache write lock once and iterates through all addresses in a single critical section. This is a deliberate choice over individual remove calls for two reasons: performance and atomicity. For performance, a cascade evicting 100 dependents makes one lock acquisition instead of 100 — on a contended cache with concurrent readers, this difference is substantial. For atomicity, holding the lock for the entire batch ensures that no concurrent reader can observe a partially-invalidated state where some dependents are evicted but others still serve stale data. 

// In SharedCache (async)
pub async fn remove_batch(&self, addrs: &[CAddr])
    -> (u64, u64, Vec<CAddr>)
{
    let mut cache = self.inner.write().await;
    let mut evicted_count = 0;
    let mut bytes_reclaimed = 0;
    let mut evicted_addrs = Vec::new();

    for addr in addrs {
        if let Some(bytes) = cache.remove(addr) {
            evicted_count += 1;
            bytes_reclaimed += bytes.len() as u64;
            evicted_addrs.push(*addr);
        }
    }

    (evicted_count, bytes_reclaimed, evicted_addrs)
}

The method returns three values: the count of entries actually evicted (which may be less than the input list if some addresses were not cached), the total bytes reclaimed, and the list of addresses that were present and removed. The distinction between "requested" and "actually evicted" matters because the DAG traversal finds all transitive dependents, but not all of them may have cached results — some may never have been materialized, or their cache entries may have already been evicted by TTL or memory pressure. The caller receives an accurate accounting of what actually changed.

The bytes_reclaimed counter provides immediate feedback for capacity monitoring. After a large cascade, operators can see exactly how much cache space was freed, which informs decisions about whether to pre-warm critical entries or let them be recomputed on demand.

6.8 Cascade Example: Diamond DAG

The diamond DAG is the simplest structure that exercises the critical properties of cascade invalidation: multiple paths from a single root to a shared dependent, requiring deduplication during traversal. Leaf A feeds into both B and C at depth 1, and both B and C feed into D at depth 2. When A is invalidated, the BFS traversal discovers B and C first (depth 1), then follows their reverse edges to D (depth 2). The visited set ensures D is collected only once despite being reachable through both B and C.

The diamond shape is significant because it is the minimal structure where naive traversal without deduplication would process D twice — once via B and once via C. The visited HashSet in the BFS ensures D appears exactly once in the eviction list, preventing redundant cache removes and producing an accurate evicted_count. In real DAGs with hundreds of nodes, diamond patterns are common wherever multiple transformations of the same source feed into a downstream aggregation, making deduplication essential for both correctness and performance.

6.9 DryRun for Impact Analysis

Cascade invalidation is irreversible — once cache entries are evicted, the only way to restore them is recomputation, which may be expensive for large subgraphs. The DryRun policy exists to answer the question "what would happen if I invalidated this leaf?" without committing to the eviction. This is particularly valuable for leaves that sit at the root of wide or deep dependency trees, where the blast radius may be larger than expected. 

# Check what would be invalidated
$ deriva cascade-invalidate --dry-run leaf_addr

Result:
  root: caddr(0xabc123...)
  would_evict: 47 addresses
  max_depth: 5
  estimated_bytes: 12.3 MB

# If acceptable, proceed with actual invalidation
$ deriva cascade-invalidate leaf_addr

Result:
  evicted: 47 addresses
  bytes_reclaimed: 12.3 MB
  duration: 15ms

The dry-run result uses the same InvalidationResult struct as the real operation, so the numbers are directly comparable. The evicted_count reflects addresses that are both transitively dependent and currently cached — addresses that were never materialized or whose cache entries expired are excluded. This means the dry-run count is always ≤ the traversal count, and the difference indicates how many dependents would need no action because they have no cached state to invalidate.

In operational practice, dry-run serves as a gate before large invalidations. An operator updating a foundational leaf in a 10,000-node DAG can preview the cascade scope, verify it matches expectations, and schedule the actual invalidation during a low-traffic window if the blast radius is large enough to cause a recomputation spike. Automated systems can use the same mechanism: if the dry-run evicted_count exceeds a configured threshold, the invalidation can be deferred or split into smaller batches.

6.10 Performance Characteristics

Cascade invalidation performance depends on DAG structure:

  • Traversal: O(D) where D = number of transitive dependents. BFS visits each dependent once.
  • Eviction: O(D) cache removals, batched under single write lock.
  • Total: O(D) time, O(D) space for visited set.
Benchmark results:

DAG Shape          Dependents    Traversal    Eviction    Total
─────────────────────────────────────────────────────────────────
Linear (depth 10)      10         0.1ms        0.2ms      0.3ms
Diamond (depth 5)      31         0.3ms        0.5ms      0.8ms
Wide (fanout 100)     100         1.2ms        2.1ms      3.3ms
Deep (depth 100)      100         1.5ms        2.0ms      3.5ms

For typical DAGs (10-100 dependents), cascade invalidation completes in single-digit milliseconds. The persistent DAG's reverse edge index makes traversal fast — no need to scan the entire DAG.

6.11 Comparison with Cache Invalidation Strategies

  • Redis/Memcached: No dependency tracking — invalidation is manual (delete by key pattern). Cascade requires application logic.
  • HTTP caching (Vary, Cache-Control): Time-based expiration or manual purge. No dependency-aware invalidation.
  • Build systems (Bazel, Ninja): Dependency-aware invalidation via forward edges (inputs → outputs). Deriva uses reverse edges (outputs → dependents) for cache eviction.
  • Dataflow engines (Spark, Dask): No persistent cache — recompute on every run. Invalidation is implicit (no cache to invalidate).

Deriva's cascade invalidation is unique: it combines persistent caching with dependency-aware eviction, enabled by the persistent DAG's reverse edge index.

11. Conclusion

This work transforms Deriva from a proof-of-concept into a robust computation-addressed store. By adding a persistent DAG, async execution, verification mode, cascade invalidation, garbage collection, and comprehensive observability, the three critical gaps identified previously are addressed: sequential execution, silent stale data, and unbounded storage growth.

11.1 The Persistent DAG as Single Enabler

The persistent DAG (§3) is the foundation that enables all other robustness features: 

Persistent DAG (sled-backed, bidirectional edges)
    │
    ├──▶ Forward edges (recipe → inputs)
    │    └──▶ Async materialization (§4)
    │         └──▶ Parallel input resolution
    │
    ├──▶ Reverse edges (input → dependents)
    │    ├──▶ Cascade invalidation (§6)
    │    │    └──▶ Transitive dependent eviction
    │    │
    │    └──▶ Garbage collection (§7)
    │         └──▶ Live set computation
    │
    └──▶ Restart recovery
         └──▶ DAG survives process restart

Without the persistent DAG, none of these features would be possible. The bidirectional edge structure (forward for materialization, reverse for invalidation/GC) is the key architectural insight. The 12 MB storage cost and 5ms insert latency are acceptable tradeoffs for the robustness gained.

11.2 Key Contributions

This work makes seven specific contributions to computation-addressed storage:

  1. DAG-aware parallelism: Async executor with broadcast-channel deduplication and deadlock-free semaphore placement. Achieves 1.5-8.5× speedup for DAGs with independent branches.
  2. Active determinism verification: Dual-compute mode with parallel execution and byte-for-byte comparison. Catches non-determinism that sandboxing misses (HashMap iteration, floating-point).
  3. Dependency-aware invalidation: Cascade invalidation via reverse edge traversal. Prevents silent stale data bugs that violate computation addressing invariant.
  4. Reachability-based GC: Mark-and-sweep with pin set protection. Achieves 97% storage savings over 1 year with 200ms weekly pause.
  5. Comprehensive observability: 40+ Prometheus metrics and structured tracing. Enables monitoring, alerting, and debugging in operational deployments.
  6. Persistent DAG store: Sled-backed bidirectional graph with transactional guarantees. Survives restarts and enables all robustness features.
  7. Pay-as-you-go robustness: All features are configurable (verification mode, GC frequency, max concurrency). Operators can tune overhead based on requirements.

11.4 Comparison with Related Systems

This work occupies a unique point in the design space:

  • vs Build systems (Bazel, Ninja): Deriva adds persistent caching, active verification, and runtime materialization. Build systems focus on hermetic builds and sandboxing.
  • vs Dataflow engines (Spark, Dask): Deriva adds persistent storage and computation addressing. Dataflow engines focus on distributed execution and fault tolerance.
  • vs Caches (Redis, Memcached): Deriva adds computation and dependency tracking. Caches focus on low-latency key-value storage.
  • vs Content-addressed stores (Git, IPFS): Deriva adds computation (recipes) and automatic invalidation/GC. CAS focus on immutable data distribution.

Deriva combines ideas from all four categories: build system dependency tracking, dataflow engine parallelism, cache persistence, and content-addressed immutability. The result is a system that can serve as both a build cache and a materialized view store.

11.5 Lessons Learned

Three key lessons from this implementation shaped the understanding of what it takes to build a robust computation-addressed store:

  1. Bidirectional edges are essential: Forward edges alone (as in the original design) are insufficient for operational robustness. Reverse edges enable invalidation and GC without scanning the entire DAG — a critical requirement when the DAG contains thousands of recipes. The storage cost (2× edge data, approximately 6 MB for the reverse tree with 10,000 recipes) is justified by the O(1) dependent lookup it provides. Without reverse edges, cascade invalidation would require a full DAG scan (O(N) per invalidation), making it impractical for interactive use. The lesson: invest in bidirectional indexing early — retrofitting reverse edges into an existing system is much harder than building them from the start.
  2. Async is worth the complexity: The async/await learning curve is steep, particularly in Rust where lifetime constraints interact with async boundaries in subtle ways. Challenges were encountered with Send bounds on futures, 'static lifetime requirements for spawned tasks, and the need to use Arc instead of references for shared state. However, the parallelism benefits are real and measurable (1.5-8.5× speedup). Tokio's ecosystem maturity (async-trait, tokio::join!, broadcast channels) makes async Rust practical for concurrent systems. The lesson: embrace async for I/O-heavy workloads — the complexity is manageable with modern tooling, and the performance gains are significant.
  3. Observability is non-negotiable: Without metrics and tracing, debugging operational issues is impossible. During development, Prometheus metrics were repeatedly relied upon to diagnose performance regressions, identify cache hit rate drops, and verify that GC was reclaiming the expected amount of storage. The 2% overhead is a small price for operational visibility. The lesson: instrument everything from day one — adding observability after the fact requires touching every code path, while building it in from the start is incremental and natural.

11.6 Limitations and Future Work

The current implementation is single-node only. Key limitations:

  • Scalability: Limited by single-machine CPU, memory, and storage. Cannot scale beyond ~100k recipes or ~1TB storage.
  • Availability: Single point of failure. Process crash loses in-flight materializations (though DAG persists).
  • Latency: All compute happens locally. Cannot offload expensive functions to remote workers.
  • Coordination: No multi-node coordination. Cannot share cache or distribute load across machines.

future distributed work (Distributed Deriva) will address these limitations:

  • Distributed DAG: Replicate DAG across nodes using Raft consensus. Enables horizontal scalability and fault tolerance.
  • Remote materialization: Execute recipes on remote workers. Enables compute offloading and load balancing.
  • Distributed cache: Shard cache across nodes. Enables cache sizes beyond single-machine memory.
  • Cross-node invalidation: Propagate invalidations across cluster. Maintains consistency in distributed setting.

The persistent DAG and async executor provide the foundation for distributed extension. The bidirectional edge structure will enable distributed invalidation. The async infrastructure will support remote RPC calls.

11.7 Broader Impact

Computation addressing has applications beyond build caching:

  • Materialized views: Database query results as recipes. Automatic invalidation when base tables change.
  • ML pipelines: Feature engineering as recipes. Reproducible model training with content-addressed datasets.
  • Data pipelines: ETL transformations as recipes. Incremental processing with dependency tracking.
  • Scientific computing: Simulation results as recipes. Reproducible research with verified determinism.

The robustness features presented here (verification, invalidation, GC) make Deriva suitable for real-world use in these domains. The persistent DAG enables long-running pipelines that survive restarts. The observability features enable operational monitoring.

11.8 Final Remarks

This work demonstrates that computation addressing is practical for concurrent systems. The persistent DAG is the key enabler: it provides the structural foundation for parallelism, invalidation, and garbage collection. The 12 MB storage cost and 5ms insert latency are acceptable for the robustness gained.

The evaluation shows that These features add minimal overhead (2-6%) while providing significant benefits: 1.5-8.5× speedup, determinism verification, automatic invalidation, and 97% storage savings. The pay-as-you-go design allows operators to tune overhead based on requirements.

This work transforms Deriva from a research prototype into a robust, deployable system. The next step is distributed scaling (future distributed work), which will enable horizontal scalability, fault tolerance, and multi-datacenter deployment. The persistent DAG and async executor provide a solid foundation for this extension.

Computation addressing is the future of reproducible, cacheable computation. This work proves it can be robust, fast, and practical.

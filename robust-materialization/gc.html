7. Garbage Collection

Without garbage collection, Deriva accumulates orphaned data indefinitely: leaf blobs that no recipe references, recipes whose inputs have been deleted, and stale cache entries. Over time, storage grows unbounded. This work adds mark-and-sweep GC that identifies live addresses (reachable from the DAG) and removes everything else, with support for pinning, dry-run, and configurable limits.

7.1 The Problem: Unbounded Storage Growth

Content-addressed storage never overwrites — every new leaf version, every new recipe, every cached result gets a unique CAddr and occupies its own storage slot. This is the foundation of immutability and cache correctness, but it has a cost: storage grows monotonically. Old leaf versions, abandoned experimental recipes, and superseded pipeline stages remain in storage indefinitely, even when no client will ever request them again. Without garbage collection, storage grows linearly with the cumulative history of all operations, not with the current working set.

Orphans arise through three common patterns. Leaf replacement creates orphans when a new data version supersedes an old one — the old leaf blob and any recipes that consumed it become unreachable. Experimental recipes create orphans when developers try multiple transformation functions and keep only the best result. Pipeline evolution creates orphans when intermediate stages are restructured — old intermediate recipes and their cached outputs become dead weight.

7.2 Mark-and-Sweep Architecture

The garbage collector follows the classic mark-and-sweep pattern adapted for content-addressed storage. The mark phase builds a live set — the set of all CAddrs that are reachable from the current DAG and the pin set. The sweep phase removes everything not in the live set from the blob store, cache, and DAG. The report phase returns detailed metrics about what was collected. This three-phase structure ensures that no reachable data is ever deleted: if a CAddr is in the live set, it survives; if it is not, it is garbage.

The root set has two sources. Recipe outputs and their transitive inputs form the primary roots — any CAddr that appears as a recipe output or as an input to any recipe is live, because it could be requested or needed for recomputation. The pin set provides explicit roots for addresses that must survive GC regardless of DAG reachability — for example, addresses that are referenced by external systems or that are being held for debugging.

The separation into mark and sweep phases is important for correctness. The mark phase is read-only — it traverses the DAG and pin set without modifying any state, producing a live set as a HashSet of CAddrs. The sweep phase then uses this snapshot to decide what to delete. This two-phase approach avoids the race condition where a concurrent write adds a new recipe referencing an address that the GC is about to delete — the live set is computed atomically before any deletions begin.

7.3 Live Set Computation

The live set is computed by iterating over the forward tree of the persistent DAG, which maps each recipe's output CAddr to its input CAddrs. Every address that appears as either a key (recipe output) or a value element (recipe input) is live — it is needed to materialize at least one recipe. This is a conservative definition: it keeps all addresses that could be needed, not just those that have been recently accessed. The conservatism is intentional — deleting an address that turns out to be needed would require recomputation from scratch, which may be impossible if the original leaf data is no longer available. 

pub fn live_addr_set(&self) -> HashSet<CAddr> {
    let mut live = HashSet::new();

    for (key, val) in self.forward.iter().flatten() {
        if let Ok(arr) = <[u8; 32]>::try_from(key.as_ref()) {
            live.insert(CAddr::from_raw(arr));
        }
        if let Ok(inputs) = bincode::deserialize::<Vec<CAddr>>(&val) {
            for input in inputs {
                live.insert(input);
            }
        }
    }

    live
}

The computation is O(E) where E is the total number of edges in the DAG — each recipe contributes one key lookup and one deserialization of its input list. For a DAG with 10,000 recipes averaging 3 inputs each, this is ~30,000 insertions into a HashSet, completing in single-digit milliseconds. The result is a snapshot: it reflects the DAG state at the moment of iteration. Any recipes added after the snapshot is taken will not have their addresses collected, but this is safe — new addresses cannot be orphans because they were just created. 

DAG state:
  Recipe X: output=0xabc, inputs=[0x111, 0x222]
  Recipe Y: output=0xdef, inputs=[0xabc, 0x333]

live_set = {0xabc, 0x111, 0x222, 0xdef, 0x333}

BlobStore contains: {0x111, 0x222, 0x333, 0x444, 0x555}

Orphaned blobs = {0x444, 0x555}  (not in live_set)

7.4 The Complete GC Algorithm

The full GC algorithm is a nine-step procedure organized into the three phases. The mark phase (steps 1–5) builds the live set from the DAG and pin set, enumerates all blobs in storage, identifies orphans by set difference, and applies the max_removals safety limit. The sweep phase (steps 6–9) removes orphaned blobs from the BlobStore, identifies recipes that became broken because their inputs were just removed, evicts cache entries for all removed addresses, and removes orphaned recipes from the DAG. The report phase constructs a GcResult with complete metrics.

The max_removals limit in step 5 is a safety mechanism for the first GC run on a system with a large backlog of orphans. Without it, a system that has accumulated 50,000 orphans over months would attempt to delete all of them in a single pass, potentially causing I/O pressure and latency spikes. The limit allows operators to drain the backlog gradually — for example, removing 1,000 orphans per run over 50 runs — while keeping the system responsive. 

pub async fn run_gc(
    dag: &PersistentDag,
    blobs: &BlobStore,
    cache: &SharedCache,
    pins: &PinSet,
    config: &GcConfig,
) -> Result<GcResult, DerivaError> {
    let start = Instant::now();

    // ========== MARK ==========

    // 1. Compute live set (recipe outputs + inputs)
    let mut live_set = dag.live_addr_set();

    // 2. Add pinned addresses (user-protected)
    for addr in pins.as_set() {
        live_set.insert(*addr);
    }

    // 3. Enumerate all blobs in storage
    let all_blobs = blobs.list_addrs()?;

    // 4. Identify orphaned blobs (in BlobStore but not in live set)
    let mut orphaned_blobs: Vec<CAddr> = all_blobs.iter()
        .filter(|a| !live_set.contains(a))
        .copied()
        .collect();

    // 5. Apply max_removals limit (safety mechanism)
    if config.max_removals > 0 && orphaned_blobs.len() > config.max_removals {
        orphaned_blobs.truncate(config.max_removals);
    }

    // ========== SWEEP ==========

    // 6. Remove orphaned blobs from BlobStore
    let (blobs_removed, bytes_reclaimed_blobs) = if config.dry_run {
        let mut bytes = 0u64;
        for addr in &orphaned_blobs {
            if let Ok(Some(data)) = blobs.get(addr) {
                bytes += data.len() as u64;
            }
        }
        (orphaned_blobs.len() as u64, bytes)
    } else {
        blobs.remove_batch_blobs(&orphaned_blobs)?
    };

    // 7. Find orphaned recipes (any input was just removed)
    let removed_set: HashSet<CAddr> = orphaned_blobs.iter().copied().collect();
    let mut orphaned_recipes = Vec::new();
    for addr in dag.all_addrs() {
        if pins.is_pinned(&addr) { continue; }
        if let Ok(Some(inputs)) = dag.inputs(&addr) {
            if inputs.iter().any(|i| removed_set.contains(i)) {
                orphaned_recipes.push(addr);
            }
        }
    }
    let recipes_removed = orphaned_recipes.len() as u64;

    // 8. Remove stale cache entries
    let mut all_removed: Vec<CAddr> = orphaned_blobs.clone();
    all_removed.extend_from_slice(&orphaned_recipes);

    let (cache_entries_removed, bytes_reclaimed_cache, _) = if config.dry_run {
        (0u64, 0u64, vec![])
    } else {
        cache.remove_batch(&all_removed).await
    };

    // 9. Remove orphaned recipes from DAG
    if !config.dry_run {
        for addr in &orphaned_recipes {
            let _ = dag.remove(addr);
        }
    }

    // ========== REPORT ==========

    Ok(GcResult {
        blobs_removed,
        recipes_removed,
        cache_entries_removed,
        bytes_reclaimed_blobs,
        bytes_reclaimed_cache,
        total_bytes_reclaimed: bytes_reclaimed_blobs + bytes_reclaimed_cache,
        live_blobs: (all_blobs.len() as u64).saturating_sub(blobs_removed),
        live_recipes: dag.len() as u64,
        pinned_count: pins.count() as u64,
        duration: start.elapsed(),
        removed_addrs: if config.detail_addrs { all_removed } else { vec![] },
        dry_run: config.dry_run,
    })
}

Step 7 — orphaned recipe detection — deserves attention. After blobs are removed, any recipe whose input list references a removed blob is now broken: it can never be materialized because its inputs no longer exist. Rather than leaving these broken recipes in the DAG (where they would cause errors on future get() calls), the GC proactively removes them. This cascading cleanup is analogous to cascade invalidation (§6) but operates on storage liveness rather than cache freshness.

The dry_run flag threads through every destructive operation. When set, the mark phase runs identically (computing the same orphan list), but the sweep phase replaces actual deletions with size measurements. This produces a GcResult that accurately predicts what a real GC run would do, allowing operators to preview the impact before committing.

7.5 GC Execution Flow

The GC execution flow involves six components communicating in sequence. The Client initiates a RunGc RPC, the DerivaService delegates to the GC module, which queries the DAG for the live set, queries the BlobStore for all stored addresses, computes the orphan set by difference, then issues batch removals to both the BlobStore and Cache. Like cascade invalidation, the flow is synchronous — the RPC blocks until all deletions complete, ensuring no concurrent operation can reference a partially-collected state.

The synchronous design ensures that after the RunGc RPC returns, all orphaned blobs and cache entries have been removed. No background cleanup, no eventual consistency — the caller receives a GcResult that reflects the completed state. This simplifies reasoning about storage capacity: the live_blobs count in the result is the exact number of blobs remaining in storage at the moment the response is sent.

7.6 Pin Set: Protecting Critical Addresses

The live set computed from the DAG captures addresses that are structurally reachable — they appear as recipe outputs or inputs. But some addresses need protection for reasons the DAG cannot express. A leaf blob referenced by an external database, a recipe output served as a stable API endpoint, or inputs being consumed by a multi-hour materialization — all of these are "live" from the system's perspective but may not appear in the current DAG if the referencing recipe has been replaced or not yet created.

The PinSet provides explicit GC roots. Any pinned address is added to the live set during the mark phase, guaranteeing it survives collection regardless of DAG reachability. The implementation is a RwLock<HashSet<CAddr>> — reads (which happen during GC's mark phase) take a shared lock, while pins and unpins take an exclusive lock. Since GC runs infrequently and pin/unpin operations are rare, contention is negligible. 

pub struct PinSet {
    pinned: RwLock<HashSet<CAddr>>,
}

impl PinSet {
    pub fn pin(&self, addr: CAddr) -> bool {
        self.pinned.write().unwrap().insert(addr)
    }

    pub fn unpin(&self, addr: CAddr) -> bool {
        self.pinned.write().unwrap().remove(&addr)
    }

    pub fn is_pinned(&self, addr: &CAddr) -> bool {
        self.pinned.read().unwrap().contains(addr)
    }

    pub fn count(&self) -> usize {
        self.pinned.read().unwrap().len()
    }

    pub fn as_set(&self) -> HashSet<CAddr> {
        self.pinned.read().unwrap().clone()
    }
}

The as_set() method clones the entire pin set under a read lock, producing a snapshot that the GC can iterate without holding the lock for the duration of collection. This prevents a deadlock scenario where GC holds the pin set lock while a concurrent operation tries to pin an address. The clone cost is proportional to the pin set size, which is typically small (tens of addresses, not thousands).

  • Stable endpoints: Pin a recipe output served as a public API endpoint — prevents GC from removing it if the recipe is temporarily restructured
  • External references: Pin a leaf referenced by external systems (e.g., a dataset ID stored in a database) that the DAG does not know about
  • Long-running computations: Pin inputs during multi-hour materializations to prevent mid-computation removal
  • Debugging: Pin addresses under investigation to prevent cleanup during analysis

7.7 Orphaned Recipe Detection

After removing orphaned blobs in step 6, the GC performs a second pass to find recipes that became broken — recipes whose input lists reference blobs that were just deleted. A recipe with a missing input can never be materialized: attempting to resolve inputs will fail with "address not found." Leaving these broken recipes in the DAG wastes storage and creates confusing errors for clients who attempt to materialize them. The GC proactively removes them, keeping the DAG in a consistent state where every recipe's inputs are resolvable. 

let removed_set: HashSet<CAddr> = orphaned_blobs.iter().copied().collect();
let mut orphaned_recipes = Vec::new();

for addr in dag.all_addrs() {
    if pins.is_pinned(&addr) { continue; }  // Protected by pin

    if let Ok(Some(inputs)) = dag.inputs(&addr) {
        // If ANY input was removed, recipe is broken
        if inputs.iter().any(|i| removed_set.contains(i)) {
            orphaned_recipes.push(addr);
        }
    }
}

The detection uses the removed_set HashSet built from the just-deleted blobs, making each input check O(1). The full scan is O(R × I_avg) where R is the number of recipes and I_avg is the average input count — typically fast because most recipes have 1–5 inputs. Pinned recipes are skipped even if their inputs were removed, preserving the pin contract: a pinned address survives GC unconditionally, even if it becomes unmaterializable. The operator can unpin and re-run GC later if the broken recipe is no longer needed.

7.8 GC Configuration and Dry-Run

The GcConfig struct controls three aspects of GC behavior. The dry_run flag runs the full mark-and-sweep logic but replaces all destructive operations with size measurements, producing a GcResult that accurately predicts what a real run would do. The max_removals limit caps the number of blobs removed per cycle, preventing I/O storms when draining a large backlog of orphans. The detail_addrs flag controls whether the full list of removed CAddrs is included in the result — useful for auditing but potentially large for big collections. 

pub struct GcConfig {
    pub dry_run: bool,
    pub max_removals: usize,  // 0 = unlimited
    pub detail_addrs: bool,
}

// Dry-run to assess impact
let config = GcConfig {
    dry_run: true,
    max_removals: 0,
    detail_addrs: true,
};
let result = run_gc(&dag, &blobs, &cache, &pins, &config).await?;

println!("Would remove {} blobs, {} recipes",
         result.blobs_removed, result.recipes_removed);
println!("Would reclaim {} bytes", result.total_bytes_reclaimed);

// If acceptable, run actual GC
let config = GcConfig { dry_run: false, ..config };
let result = run_gc(&dag, &blobs, &cache, &pins, &config).await?;

The dry-run → actual-run pattern mirrors the cascade invalidation workflow (§6.9). Operators preview the blast radius, verify it matches expectations, then commit. For automated GC schedules, the max_removals limit provides a different safety mechanism: rather than previewing, it bounds the per-cycle impact, allowing the system to drain orphans gradually over multiple runs without risking latency spikes from large batch deletions.

7.9 GC Result Metrics

The GcResult struct provides comprehensive accounting of every GC operation. The removal counts (blobs_removed, recipes_removed, cache_entries_removed) and byte counts (bytes_reclaimed_blobs, bytes_reclaimed_cache, total_bytes_reclaimed) give exact measurements of what was collected. The liveness counts (live_blobs, live_recipes, pinned_count) describe the state of the system after collection. Together, these enable trend analysis: if blobs_removed is growing each cycle, the orphan creation rate exceeds the GC frequency and the schedule should be tightened. 

pub struct GcResult {
    pub blobs_removed: u64,
    pub recipes_removed: u64,
    pub cache_entries_removed: u64,
    pub bytes_reclaimed_blobs: u64,

    pub live_recipes: u64,

7.10 Performance Characteristics

GC performance is dominated by two operations: iterating the forward tree to build the live set (mark phase), and listing all blob addresses to compute the orphan set (sweep phase). Both are sequential scans over sorted B-tree structures in sled, which are I/O-efficient due to locality. The mark phase is O(R + I) where R is recipes and I is total inputs; the sweep phase is O(B) where B is total blobs plus O(O) for the actual deletions where O is orphans. 

Blobs    Recipes    Orphans    Mark    Sweep    Total
──────────────────────────────────────────────────────
1,000      500        100      2ms     8ms      10ms
10,000    2,000       500      5ms     35ms     40ms
100,000   10,000     5,000     15ms    180ms    195ms

For typical workloads (1,000–10,000 blobs), GC completes in tens of milliseconds — fast enough to run on every request if needed, though weekly or daily scheduling is more practical. The persistent DAG's forward edge index makes live set computation fast: there is no need to scan the entire blob store to determine liveness, only the DAG's forward tree. At 100,000 blobs, the sweep phase dominates because listing all blob addresses requires a full scan of the blob store's key space, but even this completes in under 200ms.

7.11 Comparison with Other GC Strategies

Deriva's GC is most similar to Git's git gc: both use mark-and-sweep over an immutable content-addressed store with explicit dependency edges. Git traverses from refs (branches, tags) to find reachable commits and blobs; Deriva traverses from recipe outputs and inputs to find reachable CAddrs. The key structural difference is that Deriva's DAG is bidirectional — forward edges enable materialization, reverse edges enable invalidation and GC — while Git's commit graph is unidirectional (parent pointers only).

  • Git: Mark-and-sweep via git gc — traverses from refs to find reachable objects. Also packs objects for compression. Most similar to Deriva's approach.
  • HDFS: Time-based expiration — files older than N days are deleted. No dependency tracking. Simple but cannot distinguish actively-used old data from truly abandoned data.
  • Redis: LRU eviction — removes least-recently-used keys when memory limit is reached. Reactive (triggered by memory pressure) rather than proactive. Cannot guarantee that evicted data is truly unreachable.
  • Java/Python GC: Tracing GC with generational collection. More complex than Deriva's GC because mutable objects require write barriers and generational tracking. Deriva's immutable data simplifies collection to a single-generation mark-and-sweep.
  • Build systems (Bazel, Ninja): No persistent storage GC — build outputs are ephemeral. Bazel has bazel clean but it removes everything, not just orphans. No selective reachability-based collection.

The combination of content-addressing (immutable data), explicit dependency edges (DAG), and bidirectional traversal (forward + reverse) gives Deriva's GC precise reachability information that most storage systems lack. This precision is what enables the 97% storage reduction shown in §7.1 — the GC knows exactly which addresses are live and can safely remove everything else.

1. Introduction

In the introduction paper, Deriva's core model was introduced: a storage system where data is addressed by how it's computed rather than by name or location. A CAddr (Computation Address) is the BLAKE3 hash of either raw data (a leaf) or a Recipe — a canonical encoding of a function identifier, input CAddrs, and parameters. This model enables structural properties that external metadata cannot provide: provenance is intrinsic (every derived result carries its computation history), recomputability is guaranteed (recipes can always be re-executed), and eviction becomes cost-aware (the system knows which cached results are expensive to recompute).

The initial implementation demonstrated the viability of this model with a single-node implementation: 244 tests, 6 crates, a complete gRPC API, and a CLI client. However, it had three critical limitations that prevented practical deployment:

The Robustness Gap

  • Sequential materialization: The original Executor resolved dependencies one at a time. For a recipe with two unmaterialized inputs, it would compute the first, wait for completion, then compute the second — even when both inputs were independent and could run concurrently. This left CPU cores idle and made deep DAGs prohibitively slow.
  • No invalidation mechanism: When a leaf changed (e.g., updated source data), all transitive dependents had stale cached materializations. The system had no way to identify or evict these stale entries. Users had to manually track dependencies and clear caches — defeating the purpose of a computation-addressed store.
  • No garbage collection: Orphaned blobs (leaf data no longer referenced by any recipe), orphaned recipes (whose inputs had been GC'd), and stale cache entries accumulated indefinitely. The only cleanup mechanism was EvictableCache::maybe_evict, which handled cache pressure but not storage reclamation.

This work addresses all three limitations through a single architectural foundation: the persistent DAG. Unlike The original in-memory DagStore, the persistent DAG is backed by sled (an embedded transactional key-value store) and survives process restarts. More importantly, it provides the structural information needed for three interlocking robustness features:

  • Async materialization (§4): The AsyncExecutor uses the DAG to detect independent branches and spawn concurrent Tokio tasks. When multiple clients request the same unmaterialized CAddr simultaneously, a broadcast channel deduplicates the computation — only one task executes, and all subscribers receive the result. A semaphore bounds concurrency, but critically, it's acquired only at the compute step (not during input resolution) to prevent deadlock in diamond DAGs.
  • Cascade invalidation (§6): The CascadeInvalidator walks the DAG's reverse edges (dependents) to find all transitive consumers of a changed leaf. A CascadePolicy controls depth limits and whether to invalidate or just report. Dry-run mode shows what would be evicted without actually clearing caches — essential for understanding blast radius before triggering a cascade.
  • Garbage collection (§7): The GC engine performs mark-and-sweep: the live set is dag.live_addr_set() (all CAddrs reachable from recipes) ∪ pinned addrs (protected via gRPC Pin/Unpin RPCs). Orphaned blobs are swept first, then orphaned recipes, then stale cache entries, then dangling DAG edges. A grace period prevents premature deletion of recently-added data.

Why DAG-Aware Parallelism is Different

Generic task parallelism (e.g., Tokio's spawn, Rayon's par_iter) treats tasks as independent units. DAG-aware parallelism exploits structural dependencies: the system knows that recipe R depends on inputs A and B, and if both are unmaterialized, it can compute them concurrently. This is fundamentally different from:

  • Build systems (Bazel, Ninja): They parallelize action execution but don't own the storage. Cached results live in external systems (remote cache servers, S3), so the build system can't perform structural invalidation or GC. When a source file changes, Bazel must recompute all transitive dependents — it has no way to selectively evict only the affected cached actions.
  • Dataflow engines (Spark, Dask, Ray): They parallelize task graphs but treat storage as external. Spark's RDD lineage enables recomputation but doesn't provide cache invalidation (if an input DataFrame changes, Spark doesn't know which cached RDDs are stale). Dask's task graph is ephemeral — it doesn't persist across sessions, so there's no way to GC orphaned intermediate results.

Deriva's persistent DAG is the single enabler for all three robustness features. It's not just a performance optimization (parallelism) or a convenience feature (invalidation) — it's the structural foundation that makes computation-addressed storage viable for real-world workloads.

Contributions

This paper presents:

  1. A persistent DAG design (§3) that survives restarts and provides O(1) forward/reverse edge lookups via sled's transactional key-value interface.
  2. An async materialization engine (§4) with broadcast-channel deduplication and deadlock-free semaphore placement — combining DAG-aware parallelism with in-storage computation.
  3. A verification mode (§5) that detects non-deterministic functions at runtime via dual-compute and hash comparison — essential for maintaining CAS invariants in practice.
  4. A cascade invalidation mechanism (§6) that walks the DAG to find all transitive dependents of a changed leaf — no external orchestrator needed.
  5. A mark-and-sweep GC (§7) that reclaims orphaned blobs, recipes, and cache entries while respecting pin-based protection — the first computation-aware storage GC.
  6. A comprehensive observability layer (§8) with 40+ Prometheus metrics across all subsystems, enabling operators to monitor cache hit rates, materialization latency, GC efficiency, and cascade blast radius.
  7. Empirical evaluation (§9) showing that parallel materialization achieves near-linear speedup on independent branches, cascade invalidation is 100-1000× faster than manual cache clearing, and GC sweep time scales linearly with blob count (I/O-bound, dominated by list_addrs).

The rest of this paper is organized as follows: §2 provides background on DAG scheduling and compares Deriva with related systems. §3–§7 describe the five core robustness features. §8 covers observability. §9 presents evaluation results. §10 discusses tradeoffs. §11 concludes.

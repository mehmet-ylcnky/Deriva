3. Persistent DAG Store

The persistent DAG is the foundation of The robustness features presented here. Unlike The original in-memory DagStore, the persistent DAG survives process restarts and provides transactional guarantees for concurrent inserts. This section describes its design, implementation, and the key operations that enable parallel materialization, cascade invalidation, and garbage collection.

3.1 Design: Sled-Backed Adjacency Lists

The DAG is represented as two adjacency lists stored in sled, an embedded transactional key-value store written in Rust. Each adjacency list is a separate sled Tree (a sorted key-value namespace within the same database file):

  • Forward tree (CAddr → Vec<CAddr>): Maps each recipe to its inputs. Given recipe R, a lookup returns the list of content addresses that must be materialized before R can execute. This tree drives the materialization engine (§4).
  • Reverse tree (CAddr → Vec<CAddr>): Maps each content address to the recipes that depend on it. Given input A, a lookup returns every recipe whose output would be invalidated if A changes. This tree drives cascade invalidation (§6) and garbage collection (§7).

Keys are 32-byte BLAKE3 hashes (CAddr); values are bincode-serialized vectors of CAddr. Maintaining both directions avoids expensive full-table scans: forward lookups are O(1) for materialization, and reverse lookups are O(1) for invalidation — neither requires iterating the entire DAG.

A critical invariant ties the two trees together: every forward entry R → [A, B] must have corresponding reverse entries A → [..., R] and B → [..., R]. If this invariant is violated (e.g., by a partial write during a crash), cascade invalidation would silently miss dependents. The transactional insert (§3.2) enforces this invariant atomically.

Figure 2: Forward and reverse edge indices in the persistent dependency DAG.

Why sled? Three options were evaluated:

  • RocksDB: Industry-standard embedded KV store, but requires C++ bindings and has a complex API. Overkill for the required access patterns (mostly sequential scans for GC, point lookups for materialization).
  • SQLite: Mature and widely deployed, but SQL overhead is unnecessary — only key-value operations are needed. Also, SQLite's locking model (database-level write lock) would serialize concurrent recipe inserts.
  • sled: Pure Rust, embedded, transactional, and optimized for concurrent writes. Supports ACID transactions across multiple trees (critical for atomic forward+reverse edge updates). Chosen for simplicity and Rust ecosystem integration.

3.2 Transactional Insert

When a recipe is registered via PutRecipe, the DAG must atomically update both forward and reverse edges. A partial write — where the forward edge is persisted but the corresponding reverse entries are not — would silently break cascade invalidation: a changed input would fail to propagate to all its dependents, producing stale results without any error signal.

The insert operation proceeds through three logical stages before touching the database:

  1. Idempotency check: If the forward tree already contains the recipe's address, the insert returns immediately. Because content addresses are deterministic (the same recipe definition always produces the same CAddr), duplicate registrations are harmless and expected — especially when multiple clients submit overlapping workloads.
  2. Self-cycle guard: If the recipe lists its own address as an input, the insert is rejected with a CycleDetected error. Transitive cycles (A → B → C → A) are not checked here; they are caught at materialization time (§4) when the async executor detects that a request is already in flight.
  3. Atomic dual-tree write: A sled transaction opens both the forward and reverse trees, writes the forward edge, then iterates over each input to append the recipe's address to that input's reverse entry. If any step fails — including a process crash mid-transaction — sled's write-ahead log ensures the entire transaction is rolled back. No partial state is ever visible to concurrent readers.

The implementation: 

pub fn insert(&self, recipe: &Recipe) -> Result<CAddr> {
    let addr = recipe.addr();
    let addr_bytes = addr.as_bytes();

    // Idempotent: already exists
    if self.forward.contains_key(addr_bytes)? {
        return Ok(addr);
    }

    // Self-cycle check
    if recipe.inputs.contains(&addr) {
        return Err(DerivaError::CycleDetected(
            format!("recipe {} lists itself as input", addr)
        ));
    }

    // Serialize inputs
    let inputs_bytes = bincode::serialize(&recipe.inputs)?;

    // Atomic transaction: write forward + update all reverse entries
    (&self.forward, &self.reverse).transaction(|(fwd_tx, rev_tx)| {
        // Write forward edge: addr → inputs
        fwd_tx.insert(addr_bytes, inputs_bytes.as_slice())?;
        
        // Update reverse edges: for each input, add addr to its dependents
        for input in &recipe.inputs {
            let key = input.as_bytes();
            let mut deps: Vec<CAddr> = match rev_tx.get(key)? {
                Some(bytes) => bincode::deserialize(&bytes)?,
                None => Vec::new(),
            };
            if !deps.contains(&addr) {
                deps.push(addr);
                let bytes = bincode::serialize(&deps)?;
                rev_tx.insert(key, bytes.as_slice())?;
            }
        }
        Ok(())
    })?;

    self.db.flush()?;  // Ensure durability
    Ok(addr)
}

Figure 3: Atomic multi-key transaction ensuring forward/reverse edge consistency.

Key properties:

  • Atomicity: The transaction closure either commits all writes or rolls back all writes. If the process crashes during the transaction, sled's write-ahead log ensures the DAG remains consistent.
  • Idempotency: Inserting the same recipe twice is a no-op (early return if forward.contains_key). This is critical for distributed scenarios where multiple nodes might register the same recipe concurrently.
  • Cycle detection: Self-cycles (recipe lists itself as input) are rejected immediately. Transitive cycles (A → B → C → A) are not detected at insert time — they're prevented by the materialization engine (§4), which tracks in-flight requests and rejects circular dependencies.

3.3 Querying the DAG

The persistent DAG exposes four query operations. Each is consumed by a different subsystem, and the separation reflects a deliberate design choice: the DAG itself has no knowledge of materialization, invalidation, or garbage collection. It is a pure data structure; the subsystems that read it impose their own semantics.

The first two operations are point lookups — single-key reads against the forward or reverse tree:

  • inputs(addr) reads the forward tree. Given a recipe's content address, it returns the list of inputs that must be materialized before the recipe can execute. This is the primary query used by the async materialization engine (§4): when a GetBlob request arrives, the engine calls inputs() to discover what needs to be resolved first, then fans out concurrent sub-requests for each input.
  • dependents(addr) reads the reverse tree. Given any content address, it returns every recipe that lists that address as an input. This is the primary query used by cascade invalidation (§6): when a blob is deleted or overwritten, dependents() identifies all downstream recipes whose cached outputs are now stale.

The remaining two operations are full scans — they iterate the entire forward tree and are used exclusively by garbage collection (§7):

  • all_addrs() enumerates every recipe address in the DAG. The GC sweep phase uses this to compare the set of known recipes against the set of blobs in the content store, identifying orphaned blobs that are no longer reachable.
  • live_addr_set() computes the transitive closure of all live addresses: every recipe output and every input referenced by any recipe. Any content address not in this set is unreachable and safe to delete. This is the mark phase of the mark-and-sweep GC described in §7.
// 1. Get inputs of a recipe (for materialization)
pub fn inputs(&self, addr: &CAddr) -> Result<Option<Vec<CAddr>>> {
    match self.forward.get(addr.as_bytes())? {
        Some(bytes) => Ok(Some(bincode::deserialize(&bytes)?)),
        None => Ok(None),
    }
}

// 2. Get dependents of an addr (for cascade invalidation)
pub fn dependents(&self, addr: &CAddr) -> Result<Vec<CAddr>> {
    match self.reverse.get(addr.as_bytes())? {
        Some(bytes) => Ok(bincode::deserialize(&bytes)?),
        None => Ok(Vec::new()),
    }
}

// 3. Get all addrs in the DAG (for GC enumeration)
pub fn all_addrs(&self) -> Vec<CAddr> {
    self.forward.iter()
        .filter_map(|r| r.ok())
        .filter_map(|(k, _)| <[u8; 32]>::try_from(k.as_ref()).ok()
                              .map(CAddr::from_raw))
        .collect()
}

// 4. Compute live set (for GC mark phase)
pub fn live_addr_set(&self) -> HashSet<CAddr> {
    let mut live = HashSet::new();
    for (key, val) in self.forward.iter().flatten() {
        // Recipe output is live
        if let Ok(arr) = <[u8; 32]>::try_from(key.as_ref()) {
            live.insert(CAddr::from_raw(arr));
        }
        // Recipe inputs are live
        if let Ok(inputs) = bincode::deserialize::<Vec<CAddr>>(&val) {
            for input in inputs {
                live.insert(input);
            }
        }
    }
    live
}

An important asymmetry: inputs() returns Option (the address might not be a recipe at all — it could be a raw blob), while dependents() returns an empty Vec for unknown addresses. This distinction matters because a missing forward entry is a normal condition (leaf blobs have no inputs), whereas a missing reverse entry simply means nothing depends on that address yet.

Figure 4: DAG query operations — inputs(), dependents(), and ancestry traversal.

Performance characteristics:

  • inputs(addr) and dependents(addr): O(1) point lookups in sled (B-tree index). Typical latency: <10μs for warm cache, <100μs for cold disk read.
  • all_addrs(): O(N) sequential scan of the forward tree, where N = number of recipes. Used only during GC (infrequent). Typical throughput: ~100K addrs/sec on SSD.
  • live_addr_set(): O(N + E) where E = total number of edges. Scans both keys (recipe outputs) and values (recipe inputs). Used only during GC mark phase. Typical time: <5ms for 1000 recipes.

3.4 Restart Recovery

A key motivation for persisting the DAG is crash recovery. The original in-memory DagStore was rebuilt from scratch on every server restart: every recipe in the content store was re-read, its inputs were re-parsed, and both forward and reverse edges were re-inserted into a fresh hash map. For a DAG with N recipes and E total edges, this reconstruction was O(N + E) — dominated by deserialization and hash-map insertion. In practice, a DAG with 10,000 recipes took 2–5 seconds to rebuild, during which the server could not accept requests.

The persistent DAG eliminates this overhead entirely. Because sled stores the forward and reverse trees on disk, the server startup path reduces to a single sled::open() call, which memory-maps the database file and returns immediately. No iteration, no deserialization, no reconstruction: 

// In deriva-server/src/state.rs
pub fn new(config: ServerConfig) -> Result<Self> {
    let db = sled::open(&config.dag_path)?;
    let dag = PersistentDag::open(&db)?;
    
    // DAG is immediately available — no rebuild needed
    log::info!("Loaded DAG with {} recipes", dag.len());
    
    // ... initialize other components
}

The dag.len() call (which counts entries in the forward tree) is the only scan performed at startup — it serves as a sanity check logged for operational visibility. Actual query operations begin serving immediately.

This difference is most significant in deployment scenarios where restarts are frequent: rolling updates, auto-scaling events, or crash-loop recovery. A 5-second rebuild window means 5 seconds of unavailability per restart; with the persistent DAG, the window drops to under 100ms regardless of DAG size.

3.5 Why Not a Graph Database?

Using a dedicated graph database (Neo4j, TigerGraph, DGraph) was considered but rejected for three reasons:

  1. Operational complexity: Graph databases are separate services that require deployment, monitoring, and backup. Sled is embedded — zero operational overhead.
  2. Query complexity mismatch: Graph databases excel at complex traversals (shortest path, PageRank, community detection). Deriva's queries are simple: "get inputs," "get dependents," "enumerate all nodes." A KV store is sufficient.
  3. Rust ecosystem integration: Sled is pure Rust with no external dependencies. Graph databases require network clients (gRPC, HTTP) and serialization overhead. For a storage system where the DAG is accessed on every materialization, the overhead would be prohibitive.

The persistent DAG is intentionally minimal: two adjacency lists, four query operations, and transactional inserts. This simplicity is a feature — it makes the system easier to reason about, test, and debug. The next section shows how this simple foundation enables sophisticated parallel materialization.

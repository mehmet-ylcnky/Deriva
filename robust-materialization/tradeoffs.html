10. Tradeoffs

The robustness features presented here involve deliberate tradeoffs between correctness, performance, complexity, and operational cost. This section examines the design decisions and their implications for real-world deployments.

10.1 Persistent DAG: Storage vs Speed

The original Deriva DAG was an in-memory HashMap<CAddr, Vec<CAddr>> — zero storage cost, instant startup, and trivial implementation. But every restart discarded the entire dependency graph, forcing full recomputation of all recipes. Worse, without reverse edges (child → parent), there was no way to answer "what depends on this address?" — making cascade invalidation and garbage collection impossible.

The persistent DAG (sled-backed B-trees with forward and reverse edge indices) trades insert latency and storage for crash recovery, invalidation, and GC. The cost is measurable: 0.5–5ms per insert versus ~0.1ms in-memory, and ~12 MB storage for 10,000 recipes. But inserts are rare compared to reads (roughly 1:1000 in typical workloads), so the latency penalty applies to a small fraction of operations. The storage cost is negligible relative to the blob data it indexes.

Figure 26: Persistent DAG tradeoff — in-memory speed vs crash-safe durability.

The decision to persist is not primarily about performance — it is about enabling capabilities that are impossible without reverse edges and crash recovery. Cascade invalidation (§6), garbage collection (§7), and restart-safe materialization (§3.4) all depend on the persistent DAG. The insert latency cost is paid once per recipe creation; the benefits are realized on every invalidation, every GC cycle, and every server restart.

10.2 Async Executor: Complexity vs Parallelism

The original synchronous executor materialized DAG nodes in a simple recursive loop: resolve inputs, compute, store, return. The control flow was straightforward, stack traces were readable, and there was no async runtime overhead. But sequential execution meant that independent branches of the DAG were materialized one at a time, leaving CPU cores idle while waiting for I/O or compute on a single branch.

The async executor (§4) replaces the recursive loop with tokio::spawn per DAG node, a semaphore for concurrency control, and broadcast channels for deduplication. The complexity cost is real: Send + 'static bounds propagate through the codebase, async stack traces are harder to read, and the tokio runtime adds a dependency. But the parallelism gains are substantial — 1.5× to 8.5× speedup depending on DAG shape — because independent branches execute concurrently and duplicate requests are coalesced rather than recomputed.

Figure 27: Async executor tradeoff — synchronous simplicity vs concurrent parallelism gains.

The speedup depends on DAG shape. A linear chain (A → B → C → D) sees no benefit because each node depends on the previous one — there is nothing to parallelize. A wide DAG with many independent branches (A depends on B, C, D, E independently) sees near-linear speedup up to the semaphore limit. Real-world DAGs fall between these extremes, typically achieving 2–4× speedup. The broadcast-channel deduplication (§4.2) adds further value: when multiple requests materialize overlapping subgraphs, the second request subscribes to the first's broadcast rather than recomputing, turning redundant work into a channel receive.

10.3 Verification Mode: Cost vs Confidence

Verification (§5) detects non-determinism by re-executing compute functions and comparing outputs. The tradeoff is direct: more verification means more confidence but higher CPU cost. The three modes — Off, DualCompute, and Sampled — represent distinct points on this curve, and the right choice depends on the environment.

Figure 28: Verification tradeoff — confidence level vs CPU cost across three modes.

The recommended deployment strategy uses DualCompute in CI/CD (where the 2× CPU cost is acceptable and full coverage catches regressions before release) and Sampled(0.05) in deployed systems (where the 0.3% wall-clock overhead is negligible and statistical detection catches persistent bugs over time). Off mode is appropriate only for compute functions that are provably deterministic by construction — pure arithmetic, deterministic serialization — where verification adds cost without value.

10.4 Semaphore Placement: Deadlock Prevention vs Throughput

The async executor uses a semaphore to limit concurrent compute operations (§4.3). The critical design question is when the semaphore permit is acquired. Acquiring before input resolution is simpler — one acquire/release pair wraps the entire operation — but creates a deadlock risk: a parent node holds a permit while waiting for child nodes to complete, but the children cannot acquire permits because the parent is holding one. In a deep DAG with a semaphore limit of N, N+1 levels of nesting guarantees deadlock.

The chosen design acquires the permit only during the CPU-bound compute phase, releasing it before any I/O or recursive materialization. This eliminates deadlock entirely (a node never holds a permit while waiting for another node) and improves throughput by 20–30% (permits are held for milliseconds of compute rather than seconds of I/O wait).

Figure 29: Semaphore placement tradeoff — full-scope holding vs compute-only acquisition.

The complexity cost of the chosen design is an RAII guard pattern that acquires the permit, executes compute, and releases on drop — roughly 5 lines of code. The alternative (deadlock in deep DAGs) is an operational catastrophe that is difficult to diagnose and impossible to recover from without restarting the server. The tradeoff is unambiguous: a small increase in code complexity eliminates an entire class of failure.

10.5 Cascade Invalidation: Correctness vs Performance

When a leaf blob changes, the original Deriva invalidated only that single address — fast (0.1ms) but incorrect. Any recipe that depended on the changed blob, directly or transitively, retained its cached output computed from the old data. Clients reading those recipes got stale results with no indication that the underlying data had changed. This is the "silent stale data" problem described in §6.1.

Full cascade invalidation (§6) traverses the reverse edge index to find all transitive dependents and evicts their cached outputs. The cost scales with the number of dependents: 0.3ms for 10 dependents, 31ms for 1,000. This is slower than single-address invalidation, but the operation is rare (invalidation happens when source data changes, not on every read) and the alternative — serving stale data — violates the fundamental correctness guarantee of computation addressing.

Figure 30: Cascade invalidation tradeoff — single-address speed vs full transitive correctness.

The DryRun mode (§6.9) mitigates the "may evict more than necessary" concern. Before committing a cascade, operators can preview exactly which addresses would be invalidated and how many cache entries would be evicted. If the blast radius is unexpectedly large, the invalidation can be scoped or deferred. This gives operators control over the performance cost without compromising the correctness guarantee — every committed cascade is complete.

10.6 Garbage Collection: Throughput vs Storage

Without garbage collection, every blob ever written persists indefinitely — even after the recipe that produced it has been replaced or deleted. Over time, orphaned blobs accumulate and storage grows without bound: benchmarks show 5.2 GB after one year of typical usage. GC (§7) reclaims this storage by removing blobs that are unreachable from any recipe in the DAG, but each GC cycle imposes a brief pause (~200ms) while the mark-and-sweep runs.

The tradeoff is between GC frequency and storage overhead. Aggressive collection (hourly) keeps storage minimal (~160 MB stable) but imposes 200ms pauses 24 times per day. Conservative collection (weekly) allows slightly more storage (~320 MB) but pauses only once per week. The 97% storage reduction compared to no GC is achieved by either schedule — the difference between hourly and weekly is marginal because most orphans accumulate over days, not hours.

Figure 31: GC frequency tradeoff — storage overhead vs collection pause frequency.

The pin set (§7.6) provides a safety valve: any address that must survive GC regardless of DAG reachability can be pinned. This decouples the GC schedule from data retention requirements — operators choose the GC frequency based on storage budget, and pin critical addresses independently.

10.7 Observability: Overhead vs Insight

The original Deriva had no metrics, no structured tracing, and no monitoring endpoints. Debugging required reading raw logs and guessing. The observability layer (§8) adds 40+ Prometheus metrics and structured tracing spans to every operation, at a measured cost of ~2% CPU overhead (atomic counter increments, histogram observations) and ~1% latency overhead (span creation and context propagation).

Figure 32: Observability tradeoff — zero overhead vs full metrics and tracing at ~2% CPU cost.

The 2% overhead is a conservative upper bound measured under sustained load. In practice, most of the cost comes from histogram observations (which require finding the correct bucket), not counter increments (which are single atomic adds). The tracing overhead is dominated by span creation, not propagation — once a span exists, attaching it to an async future via .instrument() adds negligible cost. The alternative — operating blind — turns every incident into hours of log archaeology.

10.8 Broadcast Channels: Memory vs Deduplication

When multiple concurrent requests materialize the same address, the broadcast-channel deduplication mechanism (§4.2) ensures the compute runs once and all waiters receive the result. The channel capacity determines how many messages can be buffered before subscribers must consume them. A large capacity (1,024) guarantees no message drops but costs ~32 MB per 1,000 in-flight addresses. A small capacity (16) uses ~0.5 MB but risks dropping messages if a subscriber is slow to receive.

The chosen capacity of 16 reflects the observation that subscribers are fast — they clone a Bytes handle (a reference-counted pointer, not a deep copy) and return immediately. Drops are therefore rare in practice. When a drop does occur, the subscriber receives a RecvError::Lagged and retries the materialization, which will either hit the cache (if the first computation completed) or join a new broadcast. The retry cost is low, and the 64× memory savings over large-capacity channels is significant for high-concurrency workloads.

Figure 33: Broadcast channel capacity — large buffers vs small capacity with retry on lag.

10.9 Sled vs Alternatives

The persistent DAG requires an embedded key-value store that supports ordered iteration (for live set computation during GC), ACID transactions (for atomic multi-key inserts), and efficient prefix scans (for reverse edge lookups). Three candidates were evaluated: sled, RocksDB, and SQLite.

Figure 34: Embedded store comparison — sled, RocksDB, and SQLite tradeoffs for the persistent DAG.

The decision is pragmatic rather than permanent. Sled's pure-Rust implementation eliminates cross-compilation issues and keeps the dependency tree simple — important for a research system where build reproducibility matters. If Deriva evolves toward a distributed architecture requiring multi-node replication, RocksDB's mature ecosystem (used by CockroachDB, TiKV, and others for distributed storage) would justify the build complexity tradeoff. The DAG abstraction layer isolates the storage backend, so switching requires changing one module without affecting the rest of the system.

10.10 Design Principles

These tradeoffs follow three principles that guided every design decision. These principles emerged from the fundamental requirements of computation-addressed storage: the system must be correct (computation addressing breaks if results are non-deterministic or stale), it must be practical (operators need to tune overhead for their specific workloads), and it must be observable (deployed systems require monitoring and debugging capabilities).

  • Correctness over performance: Cascade invalidation, verification mode, and deadlock-free semaphore placement prioritize correctness even when it costs performance. Silent bugs are unacceptable in a computation-addressed store because they violate the fundamental invariant: the same CAddr must always resolve to the same data. A stale cache entry or a non-deterministic function doesn't just produce wrong results — it undermines the entire addressing model. Every design decision that trades performance for correctness is justified by this principle.
  • Pay-as-you-go: Features like verification and GC are optional and configurable. Deployments can tune overhead (Sampled vs DualCompute, weekly vs daily GC, max concurrency) based on requirements. This principle recognizes that different workloads have different robustness needs: a CI/CD pipeline benefits from DualCompute verification (catch bugs before deployment), while a latency-sensitive serving system should use Sampled mode (minimize overhead). The system provides knobs, not mandates.
  • Operational visibility: Observability is non-negotiable. The 2% overhead is justified by the ability to debug, monitor, and alert in deployed systems. Without metrics, operators cannot diagnose cache hit rate degradation, identify non-deterministic functions, measure GC effectiveness, or plan capacity. Every feature in This implementation exposes Prometheus metrics and structured tracing spans, enabling operators to understand system behavior without reading source code.

10.11 When NOT to Use These Features

Scenario: Ephemeral workloads (CI builds, one-off scripts)
  → Skip persistent DAG (use the original in-memory DagStore)
  → Skip GC (process exits before orphans accumulate)
  → Use DualCompute verification (catch bugs early)

Scenario: Latency-critical serving (p99 < 10ms)
  → Disable verification (6% overhead may exceed budget)
  → Use aggressive caching (minimize compute)
  → Monitor with metrics but disable tracing (1% overhead)

Scenario: Memory-constrained environments (embedded, edge)
  → Use in-memory DAG (save 12 MB)
  → Disable observability (save 2% CPU)
  → Manual invalidation instead of cascade

Scenario: Trusted compute functions (pure math, cryptography)
  → Disable verification (functions are deterministic by design)
  → Use cascade invalidation (still need correctness)
  → Use GC (still need storage management)

10.12 Future Tradeoff Considerations

a future distributed extension will introduce new tradeoffs:

  • Consistency vs availability: Strong consistency (linearizable reads) vs eventual consistency (lower latency, higher availability).
  • Replication factor: Higher replication (better durability, availability) vs lower storage cost.
  • Network overhead: Cross-node materialization (better load balancing) vs local-only (lower latency).
  • Coordination: Centralized coordinator (simpler) vs distributed consensus (more scalable).

The single-node design avoids these tradeoffs, focusing on robustness within a single process. The persistent DAG and async executor provide a solid foundation for distributed extension.

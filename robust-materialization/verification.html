5. Verification Mode

Deriva's core invariant is computation addressing: the same recipe (function + inputs + params) MUST always produce the same output. The CAddr IS the identity of the data. If a compute function is non-deterministic (uses random numbers, timestamps, unordered iteration), the system silently produces wrong results: the "same" address returns different data depending on when it's computed. Verification Mode detects this by executing recipes twice and comparing outputs byte-for-byte.

5.1 The Problem: Silent Non-Determinism

Computation addressing relies on a single invariant: the same recipe always produces the same output. The content address is the identity of the data — if a compute function returns different bytes on re-execution with identical inputs, the address becomes a lie. A client that retrieves 0xabc123 from cache gets one result; if the cache is evicted and the recipe is recomputed, the same address returns different data. No error is raised, no warning is logged — the system silently returns wrong results.

This failure mode is particularly dangerous because it is invisible. The system continues to operate normally — cache hits return stale data, cache misses trigger recomputation that produces different data, and downstream recipes that depend on the non-deterministic output propagate the inconsistency through the DAG. The longer the non-determinism goes undetected, the wider the blast radius.

The danger is not that non-determinism is hard to fix — replacing HashMap with BTreeMap or sorting output before serialization is straightforward. The danger is that non-determinism is hard to detect. A function may pass all unit tests (which run deterministically on a single thread) and only exhibit non-determinism under concurrent execution or after a cache eviction cycle. Verification Mode (§5.2) addresses this by executing recipes twice and comparing outputs byte-for-byte, turning a silent correctness bug into an explicit error.

5.2 Three Verification Modes

Verification is configurable per-executor via VerificationMode. The design offers three points on the cost-vs-coverage tradeoff: zero overhead with no detection, full detection with 2× CPU cost, and probabilistic detection with tunable overhead. The choice depends on the deployment context — development, CI, or runtime auditing. 

#[derive(Debug, Clone, Copy, PartialEq)]
pub enum VerificationMode {
    /// No verification — single compute (default)
    Off,
    /// Dual compute — execute twice in parallel, compare byte-for-byte
    DualCompute,
    /// Deterministic sampling — verify `rate` fraction of recipes (0.0-1.0)
    Sampled { rate: f64 },
}

The key insight behind DualCompute is that parallel execution on separate blocking threads costs ~1× wall-clock time despite 2× CPU usage — the two executions overlap completely. Sampled mode uses the CAddr itself as the sampling seed, making the decision deterministic: the same address is always verified or never verified, which makes failures reproducible rather than intermittent.

The three modes form a deployment progression: DualCompute during development and CI to catch non-determinism early, Off as the default for trusted runtime workloads, and Sampled for ongoing auditing where a small fraction of recipes are continuously verified. The rate parameter in Sampled mode controls the tradeoff directly — 0.05 adds 5% CPU overhead while verifying 5% of all materializations, enough to detect systematic non-determinism within minutes of deployment.

5.3 Parallel Dual Compute

When verification is enabled, execute_verified() runs the compute function twice in parallel using tokio::join!. The two executions are fully independent: inputs are cloned, the Arc<dyn ComputeFunction> is cloned (cheap reference count), and each execution runs on a separate blocking thread via spawn_blocking. This isolation ensures that any shared mutable state inside the compute function (the most common source of non-determinism) manifests as different outputs rather than being masked by sequential execution.

The comparison is byte-for-byte: output1 == output2. This is stricter than hash comparison — it catches cases where different byte sequences happen to collide under a weaker hash. On match, the result is returned and a pass is recorded. On mismatch, a DeterminismViolation error is constructed with both output hashes, lengths, the CAddr, and the function ID — enough information to reproduce and debug the failure.

The following visualization shows the dual-compute data flow. Click to toggle between the pass and fail outcomes:

Figure 14: Dual-compute data flow — pass and fail outcomes when outputs match or diverge.

The choice of spawn_blocking over spawn is deliberate. Compute functions are CPU-bound — they transform input bytes into output bytes, often involving serialization, hashing, or numerical computation. Running CPU-bound work on the async runtime's cooperative threads would starve I/O tasks (cache lookups, network calls, semaphore operations). spawn_blocking moves each execution to a dedicated OS thread from Tokio's blocking thread pool, keeping the async runtime responsive. The tokio::join! combinator awaits both blocking tasks concurrently without consuming an async thread while waiting. 

async fn execute_verified(
    &self, addr: &CAddr, func: Arc<dyn ComputeFunction>,
    input_bytes: Vec<Bytes>, params: &BTreeMap<String, Value>,
) -> Result<Bytes> {
    let should_verify = match self.config.verification {
        VerificationMode::Off => false,
        VerificationMode::DualCompute => true,
        VerificationMode::Sampled { rate } => {
            let sample = (addr.as_bytes()[0] as f64) / 255.0;
            sample < rate
        }
    };

    if !should_verify {
        return tokio::task::spawn_blocking(move || {
            func.execute(input_bytes, ¶ms)
        }).await??;
    }

    // Clone inputs for independent execution
    let (func1, func2) = (Arc::clone(&func), Arc::clone(&func));
    let (input1, input2) = (input_bytes.clone(), input_bytes);
    let (params1, params2) = (params.clone(), params.clone());

    let (result1, result2) = tokio::join!(
        tokio::task::spawn_blocking(move || func1.execute(input1, ¶ms1)),
        tokio::task::spawn_blocking(move || func2.execute(input2, ¶ms2))
    );

    let output1 = result1??;
    let output2 = result2??;

    if output1 == output2 {
        self.verification_stats.record_pass();
        Ok(output1)
    } else {
        let error = DerivaError::DeterminismViolation {
            addr: format!("{:?}", addr),
            function_id: func.id().to_string(),
            output_1_hash: blake3::hash(&output1).to_hex().to_string(),
            output_2_hash: blake3::hash(&output2).to_hex().to_string(),
            output_1_len: output1.len(),
            output_2_len: output2.len(),
        };
        self.verification_stats.record_fail(error.clone()).await;
        Err(error)
    }
}

5.4 Verification Decision Flow

The execute_verified() function follows a three-way branch based on the configured VerificationMode. The decision is made once per materialization, before any compute work begins. For Sampled mode, the branch further depends on whether the CAddr falls within the sampling threshold — a deterministic check that always produces the same result for the same address.

The following visualization traces each path through the decision tree. Click to step through the four possible execution paths:

Figure 15: Verification decision tree — four execution paths based on cache state and mode.

The four paths cover every possible execution outcome. The Off path is the fast path — no cloning, no comparison, minimal overhead. The DualCompute path always forks, and its outcome depends on whether the function is deterministic. The Sampled path adds a threshold check that deterministically routes each CAddr to either the dual or single path. The threshold uses the first byte of the content address, which is uniformly distributed (BLAKE3 output), ensuring unbiased sampling without external RNG state.

5.5 Deterministic Sampling

Full dual-compute verification doubles CPU cost for every materialization. In practice, most compute functions are deterministic — only a small fraction contain hidden non-determinism from floating-point ordering, HashMap iteration, or timestamp embedding. Sampled { rate } mode addresses this by verifying only a fraction of materializations, chosen deterministically from the CAddr itself.

The sampling decision uses the first byte of the CAddr as a seed: addr.as_bytes()[0] as f64 / 255.0. Because Blake3 output bytes are uniformly distributed across 0–255, this produces an unbiased sample without any external random number generator. No Mutex<RNG>, no thread-local state, no coordination between threads — the CAddr alone determines whether verification occurs.

The critical property is reproducibility: the same CAddr always produces the same sampling decision. If sampling were random per-request, a flaky non-deterministic function might pass verification sometimes and fail other times for the same CAddr, making failures impossible to reproduce. With deterministic sampling, once a CAddr triggers a verification failure, it fails every time — the bug is pinned to a specific content address and can be debugged reliably. 

let sample = (addr.as_bytes()[0] as f64) / 255.0;
let should_verify = sample < rate;

  • Reproducibility: The same CAddr is always verified or never verified — not random per-request. A non-deterministic function that fails verification for CAddr 0x0C… will fail every time that CAddr is materialized, making the bug reproducible and debuggable.
  • Uniform distribution: Blake3 output bytes are uniformly distributed across 0–255, so the sampling fraction closely matches the configured rate across any population of CAddrs.
  • No external RNG: No Mutex<RNG>, no thread-local state, no coordination between threads. The CAddr is the only input to the sampling decision, eliminating an entire class of concurrency concerns.

5.6 Parallel Execution: Wall-Clock vs CPU Cost

tokio::join! runs both executions concurrently on separate blocking threads. This means verification adds ~2× CPU cost but only ~1× wall-clock time — the two executions overlap completely. On multi-core systems, the wall-clock overhead is just the comparison plus tokio::join! coordination, roughly 5% in benchmarks. The critical resource to monitor is CPU utilization, not latency.

This distinction matters for capacity planning. A service running at 40% CPU can enable DualCompute verification and rise to ~80% CPU without any increase in response latency. A service already at 80% CPU would need to scale horizontally first, or use Sampled mode to limit the CPU multiplier to 1 + rate instead of 2×.

5.7 Verification Statistics

Verification is only useful if failures are visible. The executor maintains a VerificationStats struct with three atomic counters — total_verified, total_passed, and total_failed — plus a mutex-guarded last_failure holding the most recent DerivaError. All counters use AtomicU64 with Ordering::Relaxed, which is sufficient because these are monotonic counters read only for observability — no correctness decisions depend on their exact values at any instant.

Every verification outcome calls either record_pass() or record_fail(), both of which increment total_verified first, then the outcome-specific counter. The failure_rate() method computes the ratio on demand, returning 0.0 when no verifications have occurred to avoid division by zero. This rate is the primary signal for alerting: a failure rate above zero in a system expected to be fully deterministic indicates a bug, while a sustained low rate in a system with known non-deterministic functions confirms that sampling is catching violations at the expected frequency. 

pub struct VerificationStats {
    pub total_verified: AtomicU64,
    pub total_passed: AtomicU64,
    pub total_failed: AtomicU64,
    pub last_failure: tokio::sync::Mutex<Option<DerivaError>>,
}

impl VerificationStats {
    pub fn record_pass(&self) {
        self.total_verified.fetch_add(1, Ordering::Relaxed);
        self.total_passed.fetch_add(1, Ordering::Relaxed);
    }

    pub async fn record_fail(&self, error: DerivaError) {
        self.total_verified.fetch_add(1, Ordering::Relaxed);
        self.total_failed.fetch_add(1, Ordering::Relaxed);
        *self.last_failure.lock().await = Some(error);
    }

    pub fn failure_rate(&self) -> f64 {
        let verified = self.total_verified.load(Ordering::Relaxed);
        if verified == 0 { return 0.0; }
        let failed = self.total_failed.load(Ordering::Relaxed);
        failed as f64 / verified as f64
    }
}

The last_failure field uses a tokio::sync::Mutex rather than a standard mutex because it is accessed from async contexts (the gRPC Status RPC handler). Storing only the most recent failure rather than a full history keeps memory bounded — in practice, the first failure is the most diagnostic, and subsequent failures for the same function tend to carry identical information.

These statistics feed into Prometheus metrics (§8) and the gRPC Status RPC, enabling monitoring dashboards to track verification health at runtime. A typical alert configuration fires when failure_rate() exceeds zero for more than one reporting interval, since any non-zero rate in a deterministic system warrants investigation.

5.8 Error Details for Debugging

A verification failure is only useful if it carries enough context to reproduce and diagnose the root cause. When execute_verified() detects a mismatch, it constructs a DeterminismViolation error containing the CAddr, the function identifier, both output hashes, and both output lengths. This combination uniquely identifies what was computed, which function computed it, and how the two outputs differed — without requiring the full output bytes to be retained in the error. 

DeterminismViolation {
    addr: "caddr(0xabc123...)",
    function_id: "transform/v1",
    output_1_hash: "0xdef456...",
    output_2_hash: "0x789abc...",
    output_1_len: 1024,
    output_2_len: 1024,
}

The relationship between output length and hash provides the first diagnostic signal. When both lengths are identical but hashes differ, the content itself varies — the most common causes are HashMap iteration order, floating-point non-associativity from parallel reductions, or timestamp embedding. When lengths differ, the divergence is structural: conditional logic with side effects, buffer allocation differences, or serialization format variations. This distinction narrows the search space before any code inspection begins.

The function_id field pins the failure to a specific compute function registration, which is critical in systems with hundreds of registered functions. Combined with the CAddr, it provides a fully reproducible test case: fetch the inputs for that CAddr, invoke that function twice, and compare. No log correlation or request tracing is needed — the error itself contains the reproduction recipe.

Failure patterns also carry diagnostic information:

  • Same length, different hash: Content differs — likely HashMap ordering or floating-point non-associativity
  • Different length: Structural divergence — conditional logic with side effects, or serialization format variation
  • Reproducible on every run: The function is fundamentally non-deterministic for this input — run with DualCompute to confirm
  • Intermittent: Race condition or timing-dependent behavior — use Sampled { rate: 1.0 } with many iterations to increase detection probability and collect multiple failure instances

5.9 Verification Scope

Verification is scoped to exactly one step in the materialization pipeline: func.execute(). This is a deliberate design choice, not a limitation. Every other step in the pipeline — cache lookup, leaf store reads, DAG traversal, recursive input resolution — is deterministic by construction. Cache lookups return bytes keyed by content address; leaf reads return immutable raw data; DAG traversal follows fixed edges. None of these can produce different outputs for the same inputs, so verifying them would consume CPU without catching any bugs. 

materialize(addr)
    │
    ├── cache.get()        ← NOT verified (no computation)
    ├── leaf_store.get()   ← NOT verified (raw data)
    ├── dag.get_recipe()   ← NOT verified (graph lookup)
    ├── resolve inputs     ← NOT verified (recursive, each node verified individually)
    │
    ├── func.execute()     ← VERIFIED (this is where non-determinism can occur)
    │
    └── cache.put()        ← NOT verified (stores verified result)

The compute function is the only point where arbitrary user-supplied code runs, and arbitrary code is the only source of non-determinism in a content-addressed system. HashMap iteration order, floating-point reduction order, thread-local caches, timestamp calls, random number generators — all of these live inside func.execute(). By focusing verification on this single boundary, the system achieves maximum detection coverage with minimum overhead.

Recursive input resolution does not need separate verification because each node in the DAG is verified individually during its own materialization. If node A depends on nodes B and C, then B and C are each verified when they are materialized. By the time A's func.execute() runs, its inputs are already verified outputs — the verification guarantee composes transitively through the DAG without any additional mechanism.

5.10 Configuration and Deployment

Verification mode is a field on ExecutorConfig, defaulting to Off. This means existing deployments gain no overhead and require no code changes — verification is purely opt-in. The default concurrency limit of num_cpus::get() * 2 provides enough thread headroom for dual-compute without starving other work, since each verification doubles the blocking threads for a single materialization but the semaphore still bounds total concurrency. 

pub struct ExecutorConfig {
    pub max_concurrency: usize,
    pub dedup_channel_capacity: usize,
    pub verification: VerificationMode,
}

impl Default for ExecutorConfig {
    fn default() -> Self {
        Self {
            max_concurrency: num_cpus::get() * 2,
            dedup_channel_capacity: 16,
            verification: VerificationMode::Off,
        }
    }
}

The recommended deployment strategy varies by environment. In development and CI/CD, DualCompute catches non-determinism before code reaches any shared environment — test suites that exercise compute functions will fail immediately if any function produces inconsistent outputs. After function updates, temporarily enabling DualCompute verifies that the new version maintains determinism for the same inputs, catching regressions that unit tests might miss because they typically test with a single invocation.

In deployed environments, Sampled { rate: 0.05 } provides continuous auditing at roughly 5% additional CPU cost. This catches non-determinism that only manifests under real workload conditions — thread contention patterns, memory pressure, or input distributions that differ from test data. The rate can be tuned per-environment: higher in staging where CPU headroom is plentiful, lower where utilization is tight.

  • Development: DualCompute — catch bugs before commit
  • CI/CD: DualCompute — gate merges on determinism
  • Deployed (default): Off — zero overhead until opted in
  • Deployed (auditing): Sampled { rate: 0.05 } — continuous 5% sampling
  • After function updates: DualCompute temporarily to verify new versions

5.11 Comparison with Build Systems

Build systems (Bazel, Ninja) also require deterministic build actions, but they detect non-determinism differently:

  • Bazel: Sandboxing + hermetic builds — prevents non-determinism by restricting access to timestamps, network, etc. Does not verify outputs.
  • Ninja: Assumes determinism — no verification. Relies on developers to ensure build rules are deterministic.
  • Deriva: Active verification — executes twice and compares. Catches non-determinism that sandboxing misses (e.g., HashMap iteration, floating-point).

Deriva's approach is more expensive (2× compute) but provides stronger guarantees: it detects non-determinism in the actual output, not just in the environment. This is critical for computation addressing, where the CAddr IS the identity — any non-determinism breaks the system's fundamental invariant.

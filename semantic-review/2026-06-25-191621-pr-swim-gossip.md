# SWIM gossip membership for the deriva cluster

This change introduces a full SWIM-protocol gossip layer in `deriva-network`: memberlist management with incarnation-based conflict resolution, UDP probe/ack/ping-req messaging, background failure detection, suspicion/dead promotion, and a cleanup loop. A `SwimRuntime` wires it together with tokio tasks, and the server exposes a `ClusterStatus` gRPC endpoint so the CLI can show live cluster state. The design is clean, well-tested, and structurally sound for a first implementation.

Watch for: **(1)** NodeId's `Hash`/`Eq` include `incarnation`, so the `HashMap<NodeId, Member>` in `MemberList` silently creates duplicate entries when a node's incarnation increments (confirmed). **(2)** The probe cycle uses `recent_acks` keyed by `SocketAddr`, which is shared across concurrent probe cycles and can give false positive acks if two cycles overlap (confirmed). **(3)** `declare_dead` in the cleanup loop hardcodes `incarnation: 0` in the queued update, making the gossipped dead-update stale relative to any node with incarnation > 0 (confirmed).

## High-level view

The membership list tracks nodes in a `HashMap<NodeId, Member>`, but `NodeId` derives `Hash` and `Eq` over all fields including `incarnation`. Every incarnation bump inserts a *new* key rather than updating the existing entry. This is the most consequential bug — it means a single node accumulates multiple entries in the map, the `alive_count()` over-counts, `random_peer()` can return stale phantom entries, and cleanup never matches the dead entry against the alive entry.

The runtime's probe loop uses a shared `HashSet<SocketAddr>` called `recent_acks` to correlate acks to the current probe target. Because the probe interval can be shorter than twice the probe timeout, two probe cycles can overlap and one cycle's ack can satisfy another cycle's liveness check, masking failures. This is a concurrency design issue rather than a data race — the lock discipline is correct, but the logical correlation is not.

The PingReq relay path forwards a Ping to the target but sends the Ack back to the relay node, not to the original requester. The original prober never receives an indirect ack, so indirect probing is effectively inoperative — every PingReq-based round falls through to suspicion.

The cleanup loop constructs a `MemberUpdate` with `incarnation: 0` when broadcasting a dead declaration. Nodes that already know the target at incarnation > 0 will discard this update as stale, preventing dead-state gossip convergence across the cluster.

<details>
<summary>Issues (5)</summary>

1. **NodeId as HashMap key includes incarnation** — `HashMap<NodeId, Member>` creates duplicate entries on incarnation bumps. Key by `SocketAddr` or a separate stable identifier instead.
2. **Probe-cycle ack correlation is global** — `recent_acks` is shared across overlapping probe cycles, causing false-positive liveness. Use per-probe-cycle sequence-based tracking.
3. **Indirect probe ack never reaches requester** — PingReq relay sends Ping with its own identity as sender; the Ack returns to the relay, not the original prober. Relay must forward the Ack back.
4. **Dead-update incarnation hardcoded to 0** — cleanup loop's `queue_update` passes `incarnation: 0` for dead declarations, making them stale at receivers with higher recorded incarnation.
5. **No message size cap before send** — metadata with a 12 KB bloom filter plus 8 piggyback entries could exceed UDP's 65507-byte limit without pre-send validation.

## NodeId as HashMap key includes incarnation

`NodeId` derives `Hash` and `Eq` over `(addr, incarnation, name)`:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct NodeId {
    pub addr: SocketAddr,
    pub incarnation: u64,
    pub name: String,
}
```

`MemberList` stores `members: HashMap<NodeId, Member>`. When `apply_update` receives an update with a higher incarnation for an existing node, it looks up the old key (with the old incarnation), finds it, and mutates the `Member`'s `id.incarnation` in place — but it does *not* re-key the map. The HashMap still holds the entry under the old key. Meanwhile, any path that constructs a `NodeId` at the new incarnation and calls `members.get(&new_id)` will miss the entry. More critically, `refute()` updates `self.local` (the key used for self-lookups), creating a divergence between the map key and the logical identity.

The test suite happens to pass because tests only check `alive_count()` which iterates values, and the short-lived property tests don't trigger multiple incarnation bumps for the same node within one map. In production with real refutation cycles, this will cause phantom entries.

Fix: key the map by `SocketAddr` (or a `(SocketAddr, String)` tuple if names can differ), and store incarnation solely inside the `Member` value.

## Probe-cycle ack correlation uses a shared set

```rust
recent_acks: Arc<Mutex<HashSet<std::net::SocketAddr>>>,
```

The probe loop clears the target's address from this set at the start of a cycle, sends a Ping, sleeps for `probe_timeout`, then checks the set. If two probe cycles are in flight simultaneously (possible if `probe_interval < 2 * probe_timeout`), an ack from cycle N can land in the set while cycle N+1 is checking it. Default config has `probe_interval = 1s` and `probe_timeout = 500ms`, so the two sleeps in one cycle total 1s, exactly matching the interval — the next cycle starts immediately as the prior one finishes, giving a race window.

More importantly, the receive loop inserts *every* ack sender into `recent_acks` unconditionally. An ack from node X to any prior Ping will mask a failure of node X in the current cycle, because the set tracks only addresses, not sequence numbers.

Fix: track `(SocketAddr, sequence)` pairs, or use per-cycle one-shot channels.

## Indirect probe path is broken

When node A sends PingReq to relay R for target T, R's handler does:

```rust
SwimMessage::PingReq { target, sender, sequence, .. } => {
    let ping = SwimMessage::Ping {
        sender: local_id.clone(),  // R's own identity
        sequence: *sequence,
        piggyback,
    };
    let _ = socket.send_to(&data, target.addr).await;
    ...
}
```

R sends a Ping to T *as if from R*. T will Ack back to R, not to A. A never receives an ack for its indirect probe. The `recent_acks` set on A never gets `target.addr` inserted via the indirect path, so indirect probing always falls through to suspicion.

Fix: R should either (a) forward T's Ack back to A, or (b) include A's address in the PingReq so T can Ack directly to A.

## Dead-update gossip carries incarnation 0

In the cleanup loop:

```rust
ml.queue_update(MemberUpdate {
    node: suspect_id,
    state: MemberState::Dead,
    incarnation: 0, // Will use recorded incarnation
    metadata: None,
});
```

The comment says "will use recorded incarnation," but nothing replaces the `0` — it's queued verbatim. When other nodes receive this via piggyback, their `apply_update` checks `update.incarnation < existing_inc` and discards it if the target's recorded incarnation is > 0. Dead declarations will not converge across the cluster.

Fix: look up the recorded incarnation from `ml.get_member(&suspect_id)` before queuing.

## No pre-send size validation

`NodeMetadata` contains `cache_bloom: Vec<u8>` (default 12,000 bytes at 96,000 bits). A Ping with 8 piggybacked updates each carrying full metadata could exceed UDP's 65,507-byte limit. The message test asserts this fits, but only with a 1,200-byte bloom stub. The production default of 12,000 bytes times 8 updates is 96 KB of bloom alone.

Mitigation options: cap metadata in piggyback (only attach metadata to a subset), or fragment, or simply validate encoded size before send and strip metadata from piggyback entries that would push over the limit.

## Test coverage observations

The test suite is thorough for the memberlist state machine — property tests verify idempotence and commutativity. Integration tests cover two-node discovery, three-node convergence, failure detection, and incarnation refutation.

Not tested: the indirect probe path end-to-end (which would have caught the relay bug), message size under production-sized metadata, behavior when multiple nodes refute simultaneously, and the metadata refresh loop (which doesn't exist in this diff — metadata is never populated in production).

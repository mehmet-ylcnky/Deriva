Abstract

Computation-addressed storage systems identify derived data by the computation that produces it, enabling content-addressing, caching, and deterministic recomputation. However, existing implementations — including the batch materialization model described in the companion papers — require full input assembly before computation begins and full output generation before any byte reaches the client. This forces O(input size) memory consumption and sequential stage execution, making large-pipeline workloads impractical.

This paper presents a streaming materialization layer for the Deriva computation-addressed store. Data flows through multi-stage compute pipelines as 64 KB chunks over bounded tokio::mpsc channels, enabling concurrent stage execution with predictable memory usage of O(stages × channel_capacity × chunk_size). A StreamChunk protocol (Data/End/Error) provides end-of-stream and error propagation semantics. Twenty built-in streaming functions span four categories — chunk-by-chunk transforms, rolling accumulators, multi-input combiners, and pipeline utilities — while a batch-to-stream adapter allows existing batch functions to participate in streaming pipelines without modification. The StreamPipeline builder constructs pipelines from the persistent DAG via topological sort, with a tee mechanism that simultaneously streams results to the client and collects them for caching.

All content-addressing invariants are preserved: streaming pipelines produce the same CAddr as batch materialization, cached results bypass the pipeline entirely, and cascade invalidation applies uniformly to streaming and batch outputs. Benchmarks on Criterion.rs measure time-to-first-byte, pipeline throughput, auto-wrap overhead, and memory consumption across varying pipeline depths and input sizes.

Keywords: streaming dataflow, computation-addressed storage, backpressure, chunk-level processing, pipeline parallelism, async Rust

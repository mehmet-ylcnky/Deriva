5. Auto-Wrapping: Batch-to-Stream Adaptation

Not every compute function has a streaming implementation. Deriva's function registry contains both batch functions (ComputeFunction) and streaming functions (StreamingComputeFunction). When the pipeline builder encounters a recipe whose function has no streaming variant, it automatically wraps the batch function — collecting inputs, executing batch, and re-chunking the output. This section describes the mechanism and its implications.

5.1 Registry Resolution

The StreamingExecutor resolves each recipe's function in two steps: 

if let Some(streaming_fn) = registry.get_streaming(func_id) {
    // Native streaming — wire directly into pipeline
    pipeline.add_streaming_stage(addr, streaming_fn, params, input_indices);
} else if let Some(batch_fn) = registry.get(func_id) {
    // Batch fallback — auto-wrap with collect + re-chunk
    pipeline.add_batch_stage(addr, batch_fn, params, input_indices);
} else {
    return Err(DerivaError::FunctionNotFound(...));
}

Streaming-native functions are preferred. The batch path is a transparent fallback — the pipeline builder handles the adaptation internally, and the rest of the pipeline sees a standard Receiver<StreamChunk> output regardless of which path was taken.

5.2 The Batch Stage Wrapper

When a BatchStage node executes inside the pipeline, it performs three steps:

  1. Collect. For each input, call collect_stream(rx) to gather all chunks into a single Bytes value. This blocks until the upstream sends End.
  2. Execute. Call the batch function's execute(input_bytes, &params) — the standard synchronous batch path.
  3. Re-chunk. Pass the result through batch_to_stream(result, chunk_size) to slice it back into 64 KB chunks for downstream consumers.
PipelineNode::BatchStage { function, params, input_indices, .. } => {
    let mut input_bytes = Vec::new();
    for &i in &input_indices {
        let rx = outputs[i].take().expect("input already consumed");
        let bytes = collect_stream(rx).await?;
        input_bytes.push(bytes);
    }
    let result = function.execute(input_bytes, &params)?;
    let rx = batch_to_stream(result, config.chunk_size);
    outputs.push(Some(rx));
}

5.3 Implications

The auto-wrap mechanism has three costs:

  • Memory. collect_stream buffers the entire input in memory. For a 100 MB input, the batch stage holds 100 MB while executing — compared to 512 KB for a streaming stage.
  • Latency. No output is produced until the batch function completes. Downstream stages stall, and the client sees no data until the batch stage finishes and batch_to_stream begins emitting chunks.
  • Cancellation granularity. A streaming function can be cancelled mid-chunk. A batch function either runs to completion or doesn't run at all — there is no partial cancellation once execute() begins.

5.4 When Auto-Wrapping Is Acceptable

Despite these costs, auto-wrapping is the right default for several cases:

  • Small inputs. For values under 1 MB, the memory and latency overhead is negligible.
  • Inherently batch operations. Some computations (sorting, global deduplication) require seeing all input before producing any output. A streaming wrapper would just buffer internally anyway.
  • Third-party functions. User-registered compute functions that predate the streaming API work without modification. The registry resolution ensures backward compatibility.

The key design principle: every function participates in the streaming pipeline, whether natively or through auto-wrapping. The pipeline builder never needs to distinguish between the two — it always gets a Receiver<StreamChunk> from each stage.

5. Auto-Wrapping: Batch-to-Stream Adaptation

Not every compute function has a streaming implementation. Deriva's function registry contains both batch functions (ComputeFunction) and streaming functions (StreamingComputeFunction). When the pipeline builder encounters a recipe whose function has no streaming variant, it automatically wraps the batch function — collecting inputs, executing batch, and re-chunking the output. This section describes the mechanism and its implications.

The auto-wrapping layer is what makes the streaming pipeline a universal execution engine rather than a parallel system that only works for streaming-aware functions. Without it, introducing the streaming path would create a fragmentation problem: every existing batch function would need a manual streaming port before it could participate in a pipeline. With auto-wrapping, the 4 existing batch functions (see §4) and any future user-registered functions work immediately — the pipeline builder handles the adaptation transparently, and downstream stages see a standard Receiver<StreamChunk> regardless of whether the upstream stage was native-streaming or auto-wrapped. This design follows the same principle as Unix pipes: every program reads stdin and writes stdout, whether it processes data incrementally or buffers everything internally.

5.1 Registry Resolution

The StreamingExecutor resolves each recipe's function in a strict two-step priority order. For every node in the topological order of the recipe DAG, it first checks whether a streaming implementation exists, then falls back to batch: 

if let Some(streaming_fn) = registry.get_streaming(func_id) {
    // Native streaming — wire directly into pipeline
    pipeline.add_streaming_stage(addr, streaming_fn, params, input_indices);
} else if let Some(batch_fn) = registry.get(func_id) {
    // Batch fallback — auto-wrap with collect + re-chunk
    pipeline.add_batch_stage(addr, batch_fn, params, input_indices);
} else {
    return Err(DerivaError::FunctionNotFound(...));
}

Streaming-native functions are always preferred. The FunctionRegistry maintains two separate maps — get_streaming(func_id) looks up the Arc<dyn StreamingComputeFunction> map, while get(func_id) looks up the Arc<dyn ComputeFunction> map. A function can be registered in both maps simultaneously (e.g., identity has both IdentityFn and StreamingIdentity), in which case the streaming variant wins. This means that adding a streaming implementation for an existing batch function is a pure upgrade — no configuration change, no pipeline rebuild, just register the streaming variant and it takes priority automatically.

The resolution happens once during pipeline construction, not per-chunk at runtime. By the time StreamPipeline::execute() runs, every node is already classified as either a StreamingStage or a BatchStage in the pipeline's node list. This means the per-chunk hot path has zero branching overhead from the resolution logic — the decision is baked into the pipeline topology.

The third branch — FunctionNotFound — is a hard error. The pipeline builder does not attempt to synthesize a function from its name or parameters. If a recipe references sha512@1.0.0 and neither a streaming nor batch implementation is registered, the entire pipeline construction fails immediately rather than producing a partial result. This fail-fast behavior is essential for a Computation-Addressed Store where every output must be deterministically reproducible: a pipeline that silently skips a missing function would produce a different CAddr than one that includes it.

5.2 The Batch Stage Wrapper

When a BatchStage node executes inside the pipeline, it performs three steps that bridge the streaming world (channels of StreamChunk) and the batch world (a single Vec<Bytes> input, single Bytes output):

  1. Collect. For each input, call collect_stream(rx) to gather all chunks into a single Bytes value. This drains the upstream receiver until it sees StreamChunk::End, accumulating all Data chunks into a Vec<Bytes> and then coalescing them into a contiguous Bytes. If the upstream sends StreamChunk::Error, the error propagates immediately — the batch function never executes. If the channel closes without an End marker (sender dropped), collect_stream returns DerivaError::ComputeFailed("stream closed without End marker"). As an optimization, if only a single Data chunk was received (common for small inputs), collect_stream returns it directly without copying — avoiding an unnecessary allocation.
  2. Execute. Call the batch function's execute(input_bytes, &params) — the standard synchronous batch path. This is the same code path that the non-streaming Executor uses, so batch functions require zero modification to work inside a streaming pipeline. The function receives a Vec<Bytes> (one entry per input) and returns Result<Bytes, ComputeError>. Any ComputeError is mapped to DerivaError::ComputeFailed and propagated.
  3. Re-chunk. Pass the result through batch_to_stream(result, chunk_size) to slice it back into chunks for downstream consumers. This function delegates to value_to_stream, which spawns a Tokio task that iterates over the Bytes in chunk_size increments using Bytes::slice(offset..end) — a zero-copy operation since Bytes is reference-counted. Each slice is sent as StreamChunk::Data, followed by StreamChunk::End. If the downstream receiver is dropped (cancellation), the spawned task detects the send failure and exits cleanly.
PipelineNode::BatchStage { function, params, input_indices, .. } => {
    let mut input_bytes = Vec::with_capacity(input_indices.len());
    for &i in &input_indices {
        let rx = outputs[i].take().expect("input already consumed");
        let bytes = collect_stream(rx).await?;
        input_bytes.push(bytes);
    }
    let result = function.execute(input_bytes, &params)
        .map_err(|e| DerivaError::ComputeFailed(e.to_string()))?;
    let rx = batch_to_stream(result, self.config.chunk_size);
    outputs.push(Some(rx));
}

The three-step pattern — collect, execute, re-chunk — is the minimal bridge between the two execution models. It preserves the StreamChunk protocol invariants from §3.2: the output always follows the Data* · End grammar, errors propagate correctly, and the downstream receiver is indistinguishable from one produced by a native streaming stage. The cost is that the collect step breaks the bounded-memory property: the batch stage must hold the entire input in memory simultaneously. For a 512 MB input with 64 KB chunks, this means the batch stage allocates ~512 MB during collection, compared to ~512 KB (8 chunks × 64 KB) for a streaming stage with the default channel capacity of 8.

The auto-wrap mechanism has three costs, each a direct consequence of breaking the streaming invariant that data flows through the pipeline without full materialization:

  • Memory. collect_stream buffers the entire input in memory. For a 100 MB input, the batch stage holds 100 MB while executing — compared to ~512 KB for a streaming stage with the default channel capacity of 8 (see §8.7). In a pipeline with multiple batch stages, the memory cost compounds: each batch stage independently buffers its full input. A 5-stage pipeline where 3 stages are auto-wrapped could hold 3× the input size simultaneously. The memory_budget field in PipelineConfig exists to cap this, but is not yet enforced (see future improvements §4).
  • Latency. No output is produced until the batch function completes. Downstream stages stall, and the client sees no data until the batch stage finishes and batch_to_stream begins emitting chunks. For a streaming-native stage, time-to-first-byte (TTFB) is approximately the time to process one chunk (~microseconds for simple transforms). For an auto-wrapped stage, TTFB equals the full collection time plus the batch execution time. On a 512 MB input, the batch vs streaming benchmarks in §8 show this difference is 744 ms (batch) vs 564 ms (streaming) — but the TTFB difference is far larger: the streaming path emits its first chunk in under 1 ms, while the batch path emits nothing for the full 744 ms.
  • Cancellation granularity. A streaming function can be cancelled mid-chunk: dropping the output receiver triggers backward propagation through the channel chain (see §3.5), and each upstream stage detects the closed channel on its next send(). A batch function either runs to completion or doesn't run at all — there is no partial cancellation once execute() begins. If a downstream consumer decides it only needs the first 1 MB of a 512 MB result, the streaming path stops after ~16 chunks; the batch path computes the entire 512 MB result, then the re-chunking task detects the dropped receiver after sending the first 16 chunks. The wasted computation is proportional to the input size.

5.4 When Auto-Wrapping Is Acceptable

Despite these costs, auto-wrapping is the right default for several cases. The key insight is that not all functions can benefit from streaming — some are inherently batch by nature, and wrapping them is not a compromise but the correct execution strategy:

  • Small inputs. For values under ~3 MB, the batch path is actually faster than streaming. The batch vs streaming sweep in §8 shows batch winning by 9.3× at 1 MB and 5.7× at 2 MB, because streaming's fixed overhead (channel creation, Tokio task spawning, chunk framing) dominates when there are only a few chunks to process. Auto-wrapping a batch function for small inputs is not just acceptable — it is optimal. The future size-aware execution mode (§1 in future improvements) will formalize this by automatically selecting batch for inputs below a configurable threshold.
  • Inherently batch operations. Some computations require seeing all input before producing any output. Sorting requires the complete dataset. Global deduplication needs to compare every element against every other. JSON schema validation must parse the entire document tree. PDF page extraction requires random access to the page tree via the cross-reference table. For these functions, a streaming wrapper would just call collect_stream internally anyway — the auto-wrap mechanism does exactly the same thing, but without requiring the function author to know about the streaming API. The 283 format-aware functions catalogued for future development include 148 batch-only functions (52%) that fall into this category.
  • Third-party and user-registered functions. The ComputeFunction trait predates the streaming API. Any function registered before the streaming system was introduced — or any function written by users who don't want to learn the streaming protocol — works without modification. The registry resolution ensures backward compatibility: existing register(Arc::new(MyCustomFn))) calls continue to work, and the function participates in streaming pipelines via auto-wrapping. This is critical for adoption — requiring all function authors to implement StreamingComputeFunction would create a barrier that discourages extending the function library.
  • Accumulator-like functions without streaming implementations. The 3 built-in accumulators (StreamingSha256, StreamingByteCount, StreamingChecksum) have native streaming implementations because they are high-value and straightforward. But a batch-only Sha512Fn or Blake3Fn would work perfectly well through auto-wrapping — the collect step gathers the input, the hash is computed in one call, and the 32–64 byte result is re-chunked (trivially, as a single chunk). The performance difference between streaming and auto-wrapped hashing is negligible for inputs under ~100 MB because the hash computation itself dominates the cost.

The key design principle: every function participates in the streaming pipeline, whether natively or through auto-wrapping. The pipeline builder never needs to distinguish between the two — it always gets a Receiver<StreamChunk> from each stage. This uniformity is what makes the pipeline composable: any function can be placed at any position in the DAG, and the type system guarantees that the connections work. A pipeline of [StreamingUppercase → BatchSha256 → StreamingBase64Encode] is perfectly valid — the middle stage auto-wraps, and the downstream base64 encoder sees the same StreamChunk protocol it would see from any streaming stage.

The auto-wrapping mechanism also provides a natural migration path for the function library. New functions can be introduced as batch-only implementations first — simpler to write, simpler to test — and later upgraded to streaming implementations when the performance benefit justifies the additional complexity. The registry resolution ensures that the upgrade is transparent: the moment a streaming variant is registered, it takes priority over the batch variant, and all pipelines that use that function automatically benefit without any configuration change.

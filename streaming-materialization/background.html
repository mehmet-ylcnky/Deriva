2. Background: Streaming Dataflow

Streaming dataflow — processing data incrementally as it arrives rather than waiting for the full input — is a well-studied paradigm in data processing systems. The core insight is simple: if a function can produce useful output before consuming its entire input, there is no reason to buffer the full input first. This observation drives systems from Unix pipes (1973) to modern distributed stream processors like Apache Flink.

Yet streaming has historically been confined to data transport systems — message brokers, event processors, ETL pipelines — where the storage layer is external and opaque. Content-addressed systems, which derive storage identifiers from computation structure, have remained firmly batch-oriented: the entire output must exist before its address can be computed. This section surveys both traditions and identifies the gap that Deriva's streaming materialization fills.

2.1 Dataflow Models

Two fundamental models govern how data moves through a computation pipeline. The choice between them determines latency characteristics, memory bounds, and the complexity of flow control.

  • Push-based. Producers emit data as it becomes available. Consumers must keep up or apply backpressure to avoid unbounded buffering. This model minimizes latency — the first output byte can arrive before the last input byte is produced — but requires explicit flow control to prevent fast producers from overwhelming slow consumers. Examples: Apache Flink, Kafka Streams, Unix pipes, Reactive Streams.
  • Pull-based. Consumers request data when ready. Producers block until a request arrives, naturally bounding memory usage. Latency is higher because the consumer must explicitly poll, but the model is simpler to reason about and compose. Examples: Rust iterators, database cursors, Volcano-style query engines, Python generators.

A third hybrid approach — push-with-backpressure — combines the latency benefits of push with the memory safety of pull. The producer pushes eagerly, but a bounded buffer between stages forces the producer to suspend when the buffer is full. This is the model adopted by Reactive Streams (JVM), Tokio channels (Rust), and Go channels.

Deriva uses this hybrid model. Producers push StreamChunk values into tokio::sync::mpsc channels with a bounded capacity (default: 8 chunks × 64 KB = 512 KB per stage). When the channel is full, the producer's send().await suspends until the consumer reads a chunk — providing natural flow control without explicit backpressure signaling or rate negotiation. The bounded buffer also caps the maximum memory footprint of a pipeline at stages × capacity × chunk_size, which is predictable and configurable via PipelineConfig.

This design has three important properties:

  1. Bounded memory. A 5-stage pipeline with default settings uses at most 5 × 512 KB = 2.5 MB of in-flight buffer, regardless of input size. Processing a 512 MB input uses the same buffer as processing a 4 MB input.
  2. Automatic flow control. No explicit backpressure protocol is needed. The Tokio runtime suspends and resumes tasks based on channel capacity, which means the slowest stage in the pipeline naturally becomes the throughput bottleneck — exactly the behavior we want.
  3. Composability. Each stage is a standalone Tokio task that reads from one channel and writes to another. Stages can be added, removed, or reordered without changing the flow control mechanism. This is what enables the auto-wrapping of batch functions described in §5.

2.2 Comparison with Existing Systems

To understand where Deriva sits in the landscape, we compare along two orthogonal axes: streaming capability (whether the system can process and emit data incrementally) and content-addressing / storage ownership (whether the system derives storage identifiers from computation structure and manages its own store). The scatter plot below maps eight systems across these axes.

The systems cluster into three groups: streaming-capable but storage-external (top-left), content-addressed but batch-only (bottom-right), and Deriva alone in the top-right quadrant. We examine each group below.

Apache Flink

True streaming with event-time processing, exactly-once semantics, and sophisticated windowing. Flink's dataflow model is the closest to Deriva's in terms of execution: both use push-based channels with backpressure, both support DAG topologies, and both process data in bounded chunks (Flink's network buffers are analogous to Deriva's StreamChunk). However, Flink is an external orchestrator — it does not own the storage layer. Intermediate state is managed through pluggable state backends (RocksDB, heap), and outputs are written to external sinks (Kafka, HDFS, S3). There is no content-addressing, no structural caching, and no computation-aware eviction. If the same Flink job runs twice with identical input, it recomputes from scratch unless the user builds an external caching layer.

Spark Structured Streaming

Micro-batch streaming built on the Spark SQL engine. Each micro-batch is a small batch job — the engine collects data for a configurable trigger interval, then processes it as a DataFrame. This provides streaming semantics at the cost of higher latency (typically 100 ms–10 s per micro-batch) compared to true streaming systems. Storage is external (HDFS, S3, Delta Lake), and there is no content-addressing of intermediate or final results. Spark's Catalyst optimizer can rewrite query plans, but it cannot skip computation when the result already exists in a content-addressed store.

Kafka Streams

Log-based stream processing embedded in the Kafka ecosystem. State is maintained in local RocksDB stores backed by Kafka changelog topics, providing fault tolerance through log compaction. While Kafka provides durable message storage, it has no concept of computation-addressing — messages are identified by topic-partition-offset, not by the computation that produced them. Two different Kafka Streams applications producing identical output from identical input will generate distinct messages with distinct offsets.

Unix Pipes

The original streaming dataflow (Thompson, 1973): processes connected by kernel-buffered pipes with natural backpressure. A pipe's buffer (typically 64 KB on Linux) suspends the writer when full — the same mechanism Deriva uses with tokio::mpsc channels. Simple and effective, but with no caching, no DAG awareness (pipes form linear chains, not arbitrary graphs), no content-addressing, and no ability to skip computation when the result is already known. Each invocation recomputes from scratch. Deriva's streaming model can be seen as "Unix pipes with content-addressed caching and DAG topology."

Bazel / Nix

Content-addressed build systems that cache computation results by deriving storage keys from the computation's inputs and parameters. Bazel uses action graphs with remote caching and remote execution; Nix uses derivation hashes for reproducible builds. Both are strictly batch — the entire output must be produced and hashed before it is stored and served. A Bazel action that produces a 500 MB artifact must write all 500 MB to disk before the downstream action can begin reading. Neither supports streaming intermediate results through a pipeline, which is precisely the limitation that motivated this work.

IPFS / DVC

IPFS (InterPlanetary File System) provides content-addressed storage for arbitrary data using Merkle DAGs, but has no computation model — it stores and retrieves data, it does not transform it. DVC (Data Version Control) adds content-addressed tracking to ML pipelines, using file hashes to detect when a pipeline stage can be skipped. However, DVC's execution model is batch: each stage runs to completion, writes its output, and only then does DVC hash the result and check the cache. Neither system supports streaming execution within the content-addressed framework.

2.3 The Gap

No existing system combines streaming execution with content-addressed storage. Streaming systems (Flink, Kafka Streams) treat storage as external. Content-addressed systems (Bazel, Nix, IPFS) are batch-only. Deriva's streaming materialization occupies the previously empty quadrant: streaming dataflow within a computation-addressed store, where intermediate chunks are processed incrementally while the final result is content-addressed, cached, and subject to the same invalidation and GC semantics as batch outputs.

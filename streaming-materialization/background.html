2. Background: Streaming Dataflow

Streaming dataflow — processing data incrementally as it arrives rather than waiting for the full input — is a well-studied paradigm in data processing systems. This section surveys the landscape and identifies the gap that Deriva's streaming materialization fills.

2.1 Dataflow Models

Two fundamental models govern how data moves through a pipeline:

  • Push-based. Producers emit data as it becomes available. Consumers must keep up or apply backpressure. Examples: Flink, Kafka Streams, Unix pipes.
  • Pull-based. Consumers request data when ready. Producers block until a request arrives. Examples: iterators, database cursors, Volcano-style query engines.

Deriva uses a push-based model with bounded-channel backpressure. Producers push StreamChunk values into tokio::mpsc channels. When the channel is full (default capacity: 8 chunks), the producer's send().await suspends until the consumer reads a chunk — providing natural flow control without explicit backpressure signaling.

2.2 Comparison with Existing Systems

The following comparison highlights where each system sits along two axes: whether it supports streaming execution, and whether it owns the storage layer with content-addressing semantics.

Apache Flink

True streaming with event-time processing, exactly-once semantics, and sophisticated windowing. However, Flink is an external orchestrator — it does not own the storage layer. Intermediate state is managed through pluggable state backends (RocksDB, heap), and outputs are written to external sinks. There is no content-addressing, no structural caching, and no computation-aware eviction.

Spark Structured Streaming

Micro-batch streaming built on the Spark SQL engine. Each micro-batch is a small batch job, providing streaming semantics at the cost of higher latency than true streaming systems. Storage is external (HDFS, S3, Delta Lake), and there is no content-addressing of intermediate or final results.

Kafka Streams

Log-based stream processing embedded in the Kafka ecosystem. State is maintained in local RocksDB stores backed by Kafka changelog topics. While Kafka provides durable message storage, it has no concept of computation-addressing — messages are identified by topic-partition-offset, not by the computation that produced them.

Unix Pipes

The original streaming dataflow: processes connected by kernel-buffered pipes with natural backpressure. Simple and effective, but with no caching, no DAG awareness, no content-addressing, and no ability to skip computation when the result is already known. Each invocation recomputes from scratch.

Bazel / Nix

Content-addressed build systems that cache computation results. Bazel uses action graphs with remote caching; Nix uses derivation hashes for reproducible builds. Both are strictly batch — the entire output must be produced before it is stored and served. Neither supports streaming intermediate results through a pipeline.

2.3 The Gap

No existing system combines streaming execution with content-addressed storage. Streaming systems (Flink, Kafka Streams) treat storage as external. Content-addressed systems (Bazel, Nix, IPFS) are batch-only. Deriva's streaming materialization occupies the previously empty quadrant: streaming dataflow within a computation-addressed store, where intermediate chunks are processed incrementally while the final result is content-addressed, cached, and subject to the same invalidation and GC semantics as batch outputs.

7. Integration with CAS Invariants

Streaming materialization must not break the invariants that make content-addressed storage useful. This section shows how each core CAS property — deterministic addressing, caching, invalidation, and verification — is preserved when computation happens incrementally through chunks rather than atomically on full values.

7.1 Deterministic CAddr

The fundamental CAS invariant: the same recipe always produces the same CAddr. Streaming does not change this because the CAddr is computed from the recipe structure — the function ID, the ordered list of input addresses, and the parameter map — not from the output bytes. Whether the output is produced in one batch call or flows through 1,000 chunks across five concurrent stages, the recipe is identical, so the CAddr is identical. This is by design: the addressing model defined in the foundational architecture deliberately decouples the address from the execution strategy.

A subtlety: the output bytes must also be identical regardless of execution path. If a streaming function produced different bytes than its batch equivalent for the same inputs, the cache-after-collect strategy (§7.2) would store a different value under the same CAddr — a silent correctness violation. The StreamChunk protocol (§3) guarantees this: concatenating all Data chunk payloads in order yields exactly the same Bytes as the batch function's return value. This equivalence is verified empirically for all 20 built-in streaming functions in the test suite.

7.2 Caching Strategy: Cache-After-Collect

Decision: Option A. The cache stores full Bytes values, exactly as in batch mode. Streaming is a compute-path optimization — the storage layer sees no difference. After a streaming pipeline completes, the executor uses tee_stream to fork the output: one branch delivers chunks to the client immediately, while the other feeds into collect_stream() which reassembles the chunks into a single Bytes value and stores it in the existing AsyncMaterializationCache. Future requests for the same CAddr get a cache hit and return the full value without recomputation — served as a batch response regardless of how it was originally produced.

The cost of this approach is one full-result buffer per pipeline execution (the collector must hold all chunks in memory before writing to cache). For a 512 MB result, this means 512 MB of temporary allocation. But this cost is incurred exactly once per unique computation — subsequent requests are served from cache with zero compute cost. Option B (chunk-level caching) would avoid this buffer but introduces cache fragmentation, chunk ordering complexity, and incompatibility with the existing GC and invalidation infrastructure. Option C (no caching) defeats the core value proposition of content-addressed storage.

7.3 Invalidation

Cascade invalidation (described in Robust Materialization, §6) traverses reverse edges in the persistent dependency DAG and evicts all transitive dependents of a changed input. Streaming does not alter the DAG structure — the same recipes produce the same forward and reverse edges, regardless of execution mode. When an input is invalidated:

  1. The cached Bytes for all transitive dependents are evicted via BFS over the reverse-edge index.
  2. The next request for any evicted CAddr triggers a fresh materialization. The StreamingExecutor checks the cache first (§6.3), finds a miss, and builds a new pipeline — which may use streaming, batch, or a mix depending on registry availability.

No streaming-specific invalidation logic is needed. The invariant holds: if any input changes, the cached output is evicted, and recomputation produces the correct new result. The only edge case is invalidation during an in-progress pipeline: if a source node's data is invalidated while chunks are in transit, the pipeline completes with the old data and caches the (now stale) result. The next invalidation pass will evict it again. This is the same race condition that exists in batch mode — the invalidation protocol is eventually consistent, not transactional.

7.4 Verification

The verification system (described in Robust Materialization, §5) detects non-determinism by re-executing a recipe and comparing the output against the cached result. With streaming, the verification path becomes:

  1. Build a streaming pipeline for the recipe (identical to the normal materialization path).
  2. Collect the output stream into Bytes via collect_stream().
  3. Hash the collected bytes and compare against the expected CAddr.

The critical property: the collected bytes must be identical whether produced by the batch path or the streaming path. This holds because each streaming function is required to produce the same logical output as its batch equivalent — the StreamChunk protocol simply slices that output into bounded pieces for transport. The collect_stream() function reassembles them by concatenating all Data payloads in order, recovering the original Bytes value. All three verification modes (Off, DualCompute, Sampled) work unchanged — the mode selection is orthogonal to the execution strategy.

7.5 Garbage Collection

GC (described in Robust Materialization, §7) operates on cached Bytes values identified by CAddr. Since streaming results are cached as full Bytes via cache-after-collect (§7.2), GC sees no difference between a value produced by batch and one produced by streaming. The three-phase mark-sweep-report cycle, cost-aware eviction scoring, and LRU policies all work unchanged.

The only streaming-specific consideration: if a streaming pipeline is in progress when GC runs, the intermediate chunks in transit through channels are not visible to GC. They exist only as Tokio task state and bounded channel buffers — not in the cache or the blob store. GC cannot and need not evict them. Once the pipeline completes and the result is cached, it becomes a normal cache entry subject to normal eviction. If GC evicts a value that a running pipeline depends on (e.g., a cached intermediate that was already loaded as a Cached node), the pipeline is unaffected — the node holds its own Bytes copy, independent of the cache.

7. Integration with CAS Invariants

Streaming materialization must not break the invariants that make content-addressed storage useful. This section shows how each core CAS property — deterministic addressing, caching, invalidation, and verification — is preserved when computation happens incrementally through chunks rather than atomically on full values.

7.1 Deterministic CAddr

The fundamental CAS invariant: the same recipe always produces the same CAddr. Streaming does not change this. The CAddr is computed from the recipe (function ID + input addresses + parameters), not from the output bytes. Whether the output is produced in one batch or 1,000 chunks, the recipe is identical, so the CAddr is identical.

Streaming is a compute optimization, not a storage optimization. The pipeline changes how bytes flow through the system, but the logical computation — and therefore the address — remains the same.

7.2 Caching Strategy: Cache-After-Collect

Decision: Option A. The cache stores full Bytes values, exactly as in batch mode. Streaming is a compute-path optimization — the storage layer is unchanged. After a streaming pipeline completes, the result is collected via tee_stream + collect_stream and cached normally. Future requests for the same CAddr get a cache hit and return the full value without recomputation.

7.3 Invalidation

Cascade invalidation (Paper 2, §4) traverses reverse edges in the dependency DAG and evicts dependents. Streaming does not change the DAG structure — the same recipes, the same edges, the same reverse-edge index. When an input is invalidated:

  1. The cached Bytes for all transitive dependents are evicted.
  2. The next request for any evicted CAddr triggers a fresh materialization — which may now use the streaming pipeline instead of batch.

No streaming-specific invalidation logic is needed. The invariant holds: if any input changes, the cached output is evicted, and recomputation produces the correct new result.

7.4 Verification

Verification re-executes a recipe and compares the output hash against the stored CAddr. With streaming, the verification path is:

  1. Build a streaming pipeline for the recipe.
  2. Collect the output stream into Bytes.
  3. Hash the collected bytes.
  4. Compare against the expected CAddr.

The collected bytes are identical whether produced by batch or streaming execution — the StreamChunk protocol guarantees that concatenating all Data chunks yields the same Bytes as the batch function's output. This is a testable invariant: the test suite (§8) verifies batch/streaming equivalence for all 20 built-in functions.

7.5 Garbage Collection

GC operates on cached Bytes values identified by CAddr. Since streaming results are cached as full Bytes (§7.2), GC sees no difference between a value produced by batch and one produced by streaming. The cost-aware eviction model, reference counting, and LRU policies all work unchanged.

The only streaming-specific consideration: if a streaming pipeline is in progress when GC runs, the intermediate chunks in transit through channels are not visible to GC. They exist only as Tokio task state and channel buffers — not in the cache. GC cannot and need not evict them. Once the pipeline completes and the result is cached, it becomes a normal cache entry subject to normal eviction.

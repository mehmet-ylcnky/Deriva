10. Conclusion

This paper presented the design and implementation of streaming materialization for a Computation-Addressed Store. Where Robust Materialization established fault-tolerant batch execution with content-addressed caching, the present work extends the execution model to support chunk-at-a-time processing through concurrent pipeline stages — without breaking the CAddr invariants that make the store trustworthy. The result is a system that can materialize arbitrarily large derived data with bounded memory, immediate first-byte delivery, and graceful cancellation, while preserving full backward compatibility with existing batch functions.

10.1 Summary of Contributions

This work makes five concrete contributions to the Deriva system, each building on the previous:

  • The StreamChunk protocol (§3) — a three-variant enum (Data, End, Error) carried over Tokio bounded mpsc channels. The protocol's regular language characterization (Data* · (End | Error)) guarantees that every stream terminates, every error propagates, and backpressure emerges naturally from channel bounds without an explicit variant. The Bytes-based payload provides O(1) clone via reference counting, enabling zero-copy fan-out.
  • 20 streaming built-in functions (§4) — organized into four categories (9 transforms, 3 accumulators, 3 combiners, 5 utilities) that demonstrate the four fundamental execution patterns for streaming computation. The spawn_map and spawn_accumulate helpers reduce per-function boilerplate to a single closure, making the library straightforward to extend. Each function implements the StreamingComputeFunction trait with stream_execute, providing a uniform interface regardless of internal complexity.
  • Transparent auto-wrapping (§5) — any existing batch ComputeFunction participates in a streaming pipeline via automatic collect-execute-rechunk adaptation. The registry resolution prefers streaming variants when available, ensuring that upgrading a function from batch to streaming is a transparent, zero-configuration change. This design eliminates the adoption barrier that would otherwise require porting all existing functions before the streaming path could be used.
  • The StreamPipeline executor (§6) — topological DAG traversal, concurrent stage execution via Tokio spawned tasks, and bounded mpsc channels with configurable capacity. The executor handles source nodes (leaf data), cached nodes (previously materialized results), streaming stages, and auto-wrapped batch stages through a single unified execution loop. Metrics instrumentation (chunk counts, byte totals, pipeline duration) is woven into the output wrapper for observability.
  • CAddr invariant preservation (§7) — the cache-after-collect strategy ensures that streaming is a pure compute optimization: the content-addressed store sees identical Bytes values regardless of whether the result was produced by the batch path or the streaming path. This is the critical correctness property — without it, the same recipe could produce different content addresses depending on the execution mode, breaking the fundamental guarantee of a Computation-Addressed Store.

10.2 Key Results

The empirical evaluation (§8) validates the design with Criterion.rs benchmarks on real hardware across multiple dimensions:

  • Crossover at ~3 MB. For a 5-stage identity pipeline, streaming overtakes batch on wall-clock time above 3–4 MB of input data. At 32 MB, streaming is 2.7× faster (14.4 ms vs 38.7 ms) — the peak advantage. The crossover point is determined by streaming's fixed overhead (~1.5 ms for channel creation and task spawning) versus batch's linear scaling with input size. Below the crossover, batch is up to 9.3× faster (0.17 ms vs 1.59 ms at 1 MB).
  • Constant time-to-first-byte. Streaming delivers the first 64 KB chunk in ~1.6 ms regardless of input size (1 MB or 512 MB) or pipeline depth. This property is independent of the computation being performed — it depends only on the channel setup time and the source chunking speed. For interactive applications, this means the user sees data almost immediately, even for multi-gigabyte materializations.
  • Bounded memory. A 5-stage pipeline with the default configuration (channel capacity 8, chunk size 64 KB) uses at most 2.5 MB of buffer memory — regardless of input size. The same pipeline in batch mode would buffer up to 2.5 GB for a 512 MB input. This 1,000× reduction is the most impactful result for production deployments, where memory is often the binding constraint on worker node density.
  • Channel capacity 8 is near-optimal. The capacity sweep across 7 values (1–64) and 6 input sizes (1 MB–512 MB) shows that capacity 8 captures 80–90% of available throughput gain while using only 2.5 MB per 5-stage pipeline. Capacity 16 offers 14–21% improvement for ≥256 MB workloads at 2× memory cost. Capacity 32+ shows diminishing returns that do not justify the memory increase.
  • 64 KB chunk size is the sweet spot. Chunk size sensitivity testing shows 64 KB at the knee of the overhead curve: 1 KB chunks incur 5.7× more overhead (8.56 ms vs 1.50 ms at 1 MB), while 256 KB chunks save only 8% with 4× the memory footprint per stage.

10.3 Future Directions

The streaming materialization system opens several avenues for future work, ranging from near-term optimizations to architectural extensions:

  • Size-aware execution mode selection. The benchmark crossover at ~3 MB provides a clear decision boundary that can be automated. A streaming_threshold parameter in PipelineConfig would let the executor check input size at pipeline construction time and select batch for small inputs, streaming for large ones. For intermediate computed nodes where the size is unknown, the system could default to streaming (safe for any size) or use size hints from function metadata.
  • Adaptive chunk sizing. The current uniform 64 KB chunk size works well for the common case, but stages with different throughput characteristics could benefit from per-stage tuning. A ChunkResizer stage inserted between mismatched stages could monitor send/recv timing and adjust chunk boundaries dynamically — increasing chunk size when the downstream is fast (reducing channel overhead) and decreasing it when the downstream is slow (improving backpressure granularity).
  • Pipeline fusion. Adjacent map stages (e.g., StreamingUppercase → StreamingBase64Encode) each require a channel hop (~100 ns per chunk). For simple byte transforms where the channel overhead exceeds the compute cost, fusing adjacent stages into a single task that applies both transforms per chunk would eliminate the intermediate channel. The optimizer would need to verify that both stages are pure map functions with no cross-chunk state.
  • Streaming-aware caching. The current cache-after-collect strategy requires buffering the full result before storing it. A chunk-level cache keyed by (CAddr, offset) could store individual chunks, enabling partial cache hits (serve bytes 0–64 KB from cache while computing the rest) and range reads without full materialization. This is architecturally significant but complex — it requires sub-addressing within the CAddr model and chunk-level garbage collection.
  • Cross-node streaming. The current pipeline runs within a single process. Extending the StreamChunk protocol across network boundaries — via gRPC streaming, QUIC, or a custom transport — would enable distributed computation graphs where stages run on different machines. The bounded channel model maps naturally to flow-controlled network streams, and the Error variant already carries enough information for remote error propagation.
  • Expanded function library. The current 20 streaming functions and 4 batch functions cover the core patterns. The function catalogs developed alongside this work identify 483 additional functions across cryptography, compression, format conversion, validation, analytics, and domain-specific operations (columnar data, images, audio/video, documents, geospatial, scientific, ML, bioinformatics). Implementing these in priority order — starting with high-value, low-complexity functions like BLAKE3 hashing, Zstandard compression, and format detection — would make the streaming pipeline a general-purpose data processing engine.

10.4 Closing Remarks

Streaming materialization does not replace batch — it complements it. The auto-wrapping layer (§5) ensures backward compatibility, the CAddr invariants (§7) ensure correctness, and the benchmark crossover at ~3 MB (§8) provides a clear decision boundary. Together, these properties mean that adding streaming to the Computation-Addressed Store is a pure upgrade: existing functions continue to work, existing content addresses remain valid, and the system automatically selects the better execution path as streaming implementations become available.

For the Computation-Addressed Store, streaming is the natural next step in the materialization story. The foundational architecture established that derived data can be addressed by the computation that produces it. Robust Materialization ensured that the computation succeeds reliably in the presence of failures. This work ensures that the computation executes efficiently — flowing through bounded pipelines, chunk by chunk, without sacrificing the content-addressing guarantees that make the system trustworthy. The three papers together describe a complete materialization stack: from addressing, through fault tolerance, to performance.

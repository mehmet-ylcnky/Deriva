10. Conclusion

This paper presented the design and implementation of streaming materialization for a computation-addressed store. Where Paper 2 established robust batch materialization with content-addressed caching, Paper 2b extends the execution model to support chunk-at-a-time processing through concurrent pipeline stages — without breaking the CAddr invariants that make the store trustworthy.

10.1 Summary of Contributions

  • StreamChunk protocol (§3) — a four-variant enum (Data, End, Error, Backpressure) that provides a typed, bounded communication contract between pipeline stages.
  • 20 streaming built-in functions (§4) — covering map transforms, accumulators, multi-input combiners, and utility operations, all implementing the StreamingFunction trait with spawn_streaming.
  • Auto-wrapping (§5) — any existing batch ComputeFunction can participate in a streaming pipeline via automatic collect-execute-rechunk wrapping, enabling incremental migration.
  • StreamPipeline executor (§6) — topological sort, concurrent stage execution via Tokio tasks, and bounded mpsc channels with configurable capacity.
  • CAddr invariant preservation (§7) — cache-after-collect strategy ensures that streaming is a compute optimization only; the content-addressed store sees identical Bytes values regardless of execution path.

10.2 Key Results

Benchmarks on real hardware (§8) demonstrate:

  • Crossover at ~3 MB — for a 5-stage pipeline, streaming overtakes batch on wall-clock time above 3 MB of input data. At 32 MB, streaming is 2.7× faster.
  • Constant TTFB — streaming delivers the first 64 KB chunk in ~1.6 ms regardless of input size (1 MB or 512 MB) or pipeline depth.
  • Bounded memory — a 5-stage pipeline uses at most D × N × chunk_size = 2.5 MB of buffer memory, compared to 2.5 GB for batch on 512 MB input.
  • 64 KB optimal chunk size — balances channel overhead (1.50 ms at 64 KB vs 8.56 ms at 1 KB) with streaming responsiveness.

10.3 Future Directions

  • Adaptive chunk sizing — dynamically adjust chunk size based on observed throughput and backpressure signals. Stages processing faster than their downstream could increase chunk size to reduce channel overhead; slow consumers could request smaller chunks for finer-grained backpressure.
  • Streaming-aware caching — the current cache-after-collect strategy requires buffering the full result before caching. A chunk-level cache could store individual chunks keyed by (CAddr, offset), enabling partial cache hits and range reads without full materialization.
  • Cross-node streaming — extend the pipeline model across network boundaries using gRPC streaming or QUIC, enabling distributed computation graphs where stages run on different machines with chunk-level flow control.
  • Pipeline fusion — adjacent map stages (e.g., uppercase → lowercase) could be fused into a single stage that applies both transforms per chunk, eliminating intermediate channel hops. The optimizer would need to prove that fusion preserves semantics.
  • Memory budget enforcement — the PipelineConfig::memory_budget field exists but is not yet enforced. A global memory controller could pause producers when total pipeline memory approaches the budget, providing system-wide backpressure beyond per-channel bounds.

10.4 Closing Remarks

Streaming materialization does not replace batch — it complements it. The auto-wrapping layer (§5) ensures backward compatibility, the CAddr invariants (§7) ensure correctness, and the benchmark crossover at ~3 MB (§8) provides a clear decision boundary. For the computation-addressed store, streaming is the natural next step: computations that are too large to materialize in one shot can now flow through bounded pipelines, chunk by chunk, without sacrificing the content-addressing guarantees that make the system trustworthy.

8. Evaluation

This section presents benchmark results for the streaming materialization layer. All benchmarks use Criterion.rs with 100 samples per benchmark (10 for the 100 MB test), reporting the median with 95% confidence intervals. The benchmark suite lives in crates/deriva-benchmarks/benches/streaming_pipeline.rs and is fully reproducible via cargo bench --bench streaming_pipeline.

The default 64 KB chunk size sits near the optimum. At 1 KB, the 1,000 channel send/recv round-trips dominate (8.56 ms — 5.7× slower). 8.5 Batch vs Streaming — Direct Comparison

The decisive benchmark: a 5-stage pipeline (uppercase → identity → uppercase → identity → uppercase) running the same computation in batch mode (sequential) vs streaming mode (concurrent stages). Both use real ComputeFunction and StreamingFunction implementations. Measured via cargo bench --bench sweep with 10 data sizes from 1 MB to 512 MB.

Figure 9: Batch vs streaming throughput — wall-clock time across input sizes with crossover at ~3 MB.

The crossover occurs at approximately 3 MB. Below that, batch wins because streaming's fixed overhead (channel creation, task spawning, Tokio runtime) dominates. Above 3 MB, streaming's pipeline parallelism — all 5 stages running concurrently — overtakes batch's sequential execution. The advantage peaks at 32 MB (2.7× faster) then gradually narrows as both approaches become memory-bandwidth-bound at larger sizes.

8.7 Channel Capacity Sweep

§3.4 claims that the default channel capacity of 8 balances throughput against memory usage. To validate this empirically, we swept 7 capacity values (1, 2, 4, 8, 16, 32, 64) across 6 input sizes (1 MB – 512 MB) using a 5-stage streaming identity pipeline. Each configuration was measured with Criterion.rs (median of 10–100 iterations depending on size).

Figure 10: Channel capacity sweep — throughput across 7 capacities and 6 input sizes showing capacity 8 as near-optimal.

Key findings:

  • Capacity 1 is consistently the worst — 1.6× to 3.1× slower than capacity 8 across all sizes. With only one slot, the producer suspends after every chunk, causing a task-switch round-trip (~100 ns) per 64 KB of data. At 512 MB (8,192 chunks), that adds up to ~800 µs of pure scheduling overhead on top of the data movement cost.
  • Capacity 1→8 shows steep improvement — each doubling roughly halves the overhead. Going from 1→2 cuts latency by 19–33%; from 4→8 cuts another 7–30%. This is the region where additional buffering absorbs producer/consumer speed mismatches.
  • Capacity 8→16 shows diminishing returns — improvement drops to 5–14% for most sizes. The exception is 512 MB, where 16 is 21% faster than 8, suggesting that very large inputs benefit from slightly deeper buffers.
  • Capacity 32→64 is nearly flat — within noise for small inputs (1–4 MB). For large inputs (256–512 MB), capacity 64 is only 5–14% faster than 32. The marginal gain no longer justifies the 8× memory increase over capacity 8.
  • Capacity 4 MB anomaly at cap=64 — at 4 MB, capacity 64 (2.82 ms) is slightly slower than capacity 32 (2.25 ms). With only 64 chunks total and capacity 64, the entire input fits in the channel buffer, eliminating pipelining — the producer fills the buffer before the consumer starts, effectively reverting to batch behavior with channel overhead.

Memory tradeoff: Each capacity unit buffers one 64 KB chunk per pipeline edge. A 5-stage pipeline has 5 edges, so total buffer memory is capacity × 5 × 64 KB:

  • Capacity 1: 320 KB
  • Capacity 8 (default): 2.5 MB
  • Capacity 32: 10 MB
  • Capacity 64: 20 MB

Verdict: Capacity 8 sits at the knee of the curve — it captures 80–90% of the throughput gain available from buffering, while using only 2.5 MB per pipeline. Higher capacities offer marginal latency improvements at linear memory cost. For workloads dominated by inputs ≥256 MB, capacity 16 is a reasonable alternative (14–21% faster, 2× the memory). Capacity 32+ is not justified by the data.

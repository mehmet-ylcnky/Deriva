1. Introduction

The Deriva computation-addressed store, introduced in the companion paper, addresses derived data by the computation that produces it — a cryptographic hash of the function, its input addresses, and its parameters. A CAddr encodes not where data lives, but how it was derived: the BLAKE3 hash of a function identifier, its input addresses, and its parameters. This makes caching automatic (identical computations produce identical addresses), deduplication free (two clients requesting the same derivation share one result), and provenance intrinsic (every value carries its full computation history).

The robustness paper extended this model with a persistent DAG backed by sled, async parallel materialization with broadcast-channel deduplication, cascade invalidation that walks reverse edges to find stale dependents, and mark-and-sweep garbage collection. Together, these features transformed Deriva from a prototype into a system that survives restarts, exploits multi-core parallelism, and reclaims storage from orphaned computations.

Both papers, however, share a fundamental limitation: batch materialization. Every ComputeFunction receives its inputs as fully materialized Vec<Bytes> and must produce the entire output as a single Bytes value before any downstream consumer — or the requesting client — sees a single byte. This all-or-nothing execution model creates three concrete problems as data sizes grow:

The Batch Ceiling

  • Memory amplification: A three-stage pipeline on 100 MB of data must hold up to 300 MB simultaneously — one full copy per stage — because each stage's output becomes the next stage's input and both must coexist during the handoff.
  • Sequential bottleneck: Stages execute one after another. The total wall-clock time is the sum of all stages, even when the stages could logically overlap — stage 2 could start processing the first megabyte while stage 1 is still producing the last.
  • Delayed first byte: The client must wait for the entire pipeline to complete before receiving any output. For a pipeline that takes 132 ms to process 100 MB (§8), the time-to-first-byte equals the total processing time — 132 ms of silence followed by a single bulk response.

This paper introduces streaming materialization: a chunk-at-a-time execution model that runs pipeline stages concurrently, bounds memory to a small constant regardless of data size, and delivers the first output chunk within milliseconds. The key design constraint is that streaming must not break the CAddr invariants that make the store trustworthy — the same computation address must always resolve to the same bytes, whether materialized via batch or streaming. We achieve this through a cache-after-collect strategy (§7): streaming is a compute-path optimization only, and the content-addressed store never sees partial results.

Contributions

This paper presents:

  1. A StreamChunk protocol (§3) with four variants — Data, End, Error, Backpressure — that provides typed, bounded communication between pipeline stages.
  2. A StreamingFunction trait and 20 built-in implementations (§4) covering map transforms, accumulators, multi-input combiners, and utility operations.
  3. An auto-wrapping layer (§5) that lets any existing batch ComputeFunction participate in streaming pipelines without modification — enabling incremental migration.
  4. A StreamPipeline executor (§6) with topological sort, concurrent Tokio task execution, and bounded mpsc channels with configurable capacity and backpressure.
  5. A cache-after-collect strategy (§7) that preserves CAddr invariants: streaming results are collected into Bytes and cached through the existing store — identical to batch from the cache's perspective.
  6. Empirical evaluation (§8) showing that streaming overtakes batch at ~3 MB for a 5-stage pipeline (2.7× faster at 32 MB), delivers constant ~1.6 ms time-to-first-byte regardless of data size, and bounds pipeline memory to D × N × chunk_size (2.5 MB for 5 stages vs 2.5 GB for batch on 512 MB input).

The rest of this paper is organized as follows: §2 provides background and compares Deriva with related streaming systems. §3–§7 describe the five core streaming components. §8 presents evaluation results with benchmarks against batch baselines. §9 discusses tradeoffs and design decisions. §10 concludes.

1.1 The Batch Bottleneck

In batch mode, every ComputeFunction receives its inputs as fully materialized Vec<Bytes> and must produce a complete Bytes output before the next stage can begin. This design is simple and correct — but it forces a rigid execution pattern that scales poorly with data size and pipeline depth.

Consider a concrete example: a three-stage pipeline that uppercases text, computes a checksum, and compresses the result, processing 100 MB of input. In batch mode, stage 1 allocates 100 MB to hold its output. Stage 2 cannot begin until stage 1 has finished writing all 100 MB — it then allocates another 100 MB for its own output. Stage 3 waits again. At peak, the system holds three full copies of the data (~300 MB), the wall-clock time is the sum of all three stages (132 ms in our benchmarks — §8), and the client sees nothing until the final stage completes. Every byte of output is delayed by the slowest stage in the chain, even though stage 2 could have started processing the first megabyte while stage 1 was still working on the ninety-ninth.

The problem compounds with pipeline depth. A five-stage pipeline on 512 MB of data requires up to 2.5 GB of intermediate buffers — five full copies coexisting in memory. On a 16 GB machine, that's 15% of total RAM consumed by a single materialization request. And because stages run sequentially, adding a stage adds its full latency to the total, with no opportunity for overlap. The batch model turns pipeline depth from a composability feature into a performance liability.

The following visualization contrasts this with streaming execution. In streaming mode, all stages run concurrently as independent Tokio tasks connected by bounded mpsc channels. Data flows through the pipeline chunk by chunk (64 KB default), and the client receives the first output chunk as soon as the first input chunk has traversed all stages — typically within ~1.6 ms regardless of total data size. Wall-clock time approaches the latency of the slowest stage (not the sum), and memory is bounded to D × N × chunk_size (e.g., 2.5 MB for a 5-stage pipeline — over 1,000× less than batch on the same 512 MB input).

1.2 Why Streaming Matters for CAS

A Computation-Addressed Store (CAS) derives its power from a simple invariant: every value is identified by the computation that produced it, and identical computations always yield identical results. This means the store can cache aggressively, deduplicate automatically, and invalidate precisely. But this invariant says nothing about how a computation executes — only that the final result must be deterministic. Streaming exploits this gap: it changes the execution strategy (chunk-at-a-time, concurrent stages) without changing the observable result (the same Bytes value, cached under the same CAddr).

This distinction is critical. A naive streaming implementation might cache partial results, expose intermediate chunks through the CAddr interface, or produce different byte sequences depending on chunk boundaries. Any of these would violate the CAS invariant and break the trust model. Deriva's streaming layer avoids all three pitfalls through the cache-after-collect strategy described in §7: the streaming pipeline runs internally, but the store only ever sees the fully collected result — identical to what batch would have produced.

With that invariant preserved, streaming changes the operational characteristics of the store in three fundamental ways:

  • Memory predictability. A streaming pipeline with D stages, channel capacity N, and chunk size C consumes at most D × N × C bytes of buffer memory, regardless of input size. A 5-stage pipeline with default settings (N = 8, C = 64 KB) uses at most 2.5 MB — whether the input is 1 MB or 10 GB. This transforms memory consumption from a function of data size (O(n) per stage in batch) to a fixed constant determined at pipeline construction time. For a multi-tenant store serving concurrent materialization requests, this predictability is the difference between stable operation and OOM kills.
  • Time-to-first-byte (TTFB). The client receives the first output chunk after only one chunk has traversed the pipeline, rather than waiting for the entire computation to complete. Our benchmarks (§8) show a constant ~1.6 ms TTFB regardless of input size — from 1 MB to 512 MB — while batch TTFB equals total processing time (132 ms for 100 MB through 5 stages). For interactive applications or downstream consumers that can begin processing partial results, this is the difference between perceived responsiveness and perceived stalling.
  • Pipeline parallelism. All stages execute concurrently as independent Tokio tasks. When stage 1 emits its second chunk, stage 2 is already processing the first — and stage 3 may be processing an even earlier chunk from a previous iteration. Backpressure propagates naturally through bounded channels: when a downstream stage is slow, upstream stages block on send(), preventing unbounded memory growth without any explicit flow-control protocol. The result is that wall-clock time approaches the latency of the slowest stage rather than the sum of all stages — a 5-stage pipeline on 32 MB runs in 14.4 ms (streaming) vs 38.7 ms (batch), a 2.7× improvement (§8).

1.3 Scope and Relationship to Companion Papers

This paper builds on the persistent DAG and async executor from the robustness paper and the core addressing model from the architecture paper. The streaming layer is additive — all existing batch functions continue to work unchanged, and the system automatically selects the best execution path for each node in the computation graph.

The selection mechanism is a per-node decision made during pipeline construction. When the StreamingExecutor walks the topological order of a recipe's dependency graph, it consults the FunctionRegistry for each node:

  1. Cache check first. If the node's CAddr is already cached, the executor inserts a Cached source node that emits the stored Bytes as a stream — no computation needed regardless of function type.
  2. Streaming preferred. If registry.get_streaming(func_id) returns a StreamingComputeFunction, the executor inserts a StreamingStage — a native streaming node that processes chunks concurrently with its neighbors.
  3. Batch fallback. If no streaming implementation exists but registry.get(func_id) returns a batch ComputeFunction, the executor inserts a BatchStage — an auto-wrapped node that collects all input chunks, runs the batch function, and re-chunks the output (§5).
  4. Error. If neither implementation is registered, the executor returns FunctionNotFound.

This means a single pipeline can mix streaming and batch stages freely. A 5-stage pipeline might have 3 native streaming stages and 2 auto-wrapped batch stages — the streaming stages run with bounded memory and pipeline parallelism, while the batch stages act as pipeline bubbles that collect-then-emit. As functions are migrated to native streaming implementations, the pipeline gets faster without any changes to the recipe graph or client code.

One important nuance: the current selection is purely capability-based — if a streaming implementation exists, the executor uses it. But our benchmarks (§8) reveal that this isn't always optimal. Below approximately 3 MB of input data, batch execution is actually faster — up to 9.3× at 1 MB — because streaming's fixed overhead (channel creation, Tokio task spawning, chunk framing) dominates the actual compute work. Above 3 MB, pipeline parallelism takes over and streaming wins decisively (2.7× faster at 32 MB). A future optimization could make the selection size-aware: when the executor knows the input size at construction time, it could fall back to the batch path for small inputs even when a streaming implementation is registered — combining the simplicity of batch for small data with the throughput of streaming for large data.

1.4 Contributions

  • A StreamChunk protocol with Data/End/Error semantics and bounded-channel backpressure (§3)
  • A StreamingComputeFunction trait with 20 built-in implementations across four categories (§4)
  • An auto-wrapping mechanism that allows batch functions to participate in streaming pipelines transparently (§5)
  • A DAG-aware pipeline builder that constructs streaming pipelines from recipe graphs with cache-aware stage selection (§6)
  • Preservation of all CAS invariants — identical CAddrs, cache behavior, and invalidation semantics for streaming and batch outputs (§7)

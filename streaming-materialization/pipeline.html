6. StreamPipeline & DAG-Aware Construction

The StreamPipeline is the execution engine that wires streaming stages together. Unlike traditional stream processors that operate on flat topologies, Deriva's pipeline is constructed directly from the dependency DAG — the same graph that drives batch materialization. This section describes the pipeline data structures, the topological construction algorithm, and the tee mechanism for fan-out.

6.1 Pipeline Configuration

pub struct PipelineConfig {
    pub chunk_size: usize,           // default: 64 KB
    pub channel_capacity: usize,    // default: 8 chunks
    pub cache_intermediates: bool,  // default: true
    pub memory_budget: usize,      // 0 = unlimited
}

chunk_size controls the granularity of data flow. channel_capacity sets the backpressure threshold per edge. cache_intermediates enables tee-to-cache for intermediate results. memory_budget is reserved for future use — currently all pipelines run with bounded-channel backpressure as the only memory control.

6.2 Pipeline Nodes

enum PipelineNode {
    Source   { addr: CAddr, data: Bytes },                  // leaf data
    Cached   { addr: CAddr, data: Bytes },                  // cache hit
    StreamingStage { addr: CAddr, function: ..., input_indices: ... },
    BatchStage     { addr: CAddr, function: ..., input_indices: ... },
}

Four node types. Source and Cached both emit their Bytes through value_to_stream — the distinction exists only for metrics and debugging. StreamingStage wires inputs directly. BatchStage collects inputs, runs batch, and re-chunks (§5). Each node references its inputs by index into the pipeline's node array — indices are assigned during topological construction.

6.3 DAG-to-Pipeline Construction

The executor walks the DAG depth-first from the target CAddr, producing a topological order where every node appears after all its dependencies. This is the same traversal used by the batch materializer — the streaming executor reuses the DagReader trait. 

fn resolve_order(dag: &dyn DagReader, addr: &CAddr) -> Result<Vec<CAddr>, DerivaError> {
    let mut order = Vec::new();
    let mut visited = HashSet::new();
    topo_visit(dag, addr, &mut visited, &mut order)?;
    Ok(order)
}

Node Creation Loop

For each address in topological order, the executor:

  1. Checks the cache. If the value is already materialized, insert a Cached node — no computation needed.
  2. Resolves leaf inputs. Any input not yet in the pipeline is looked up in the cache or leaf store and added as a Source or Cached node.
  3. Picks the function variant. Streaming-native if available, batch fallback otherwise (§5).
  4. Records the index. The node's position in the pipeline array becomes its identifier for downstream references.

6.4 Execution

The execute() method walks the node array in order. Each node take()s its input receivers from the outputs array, spawns its processing task, and stores its output receiver. Because nodes are in topological order, every input is guaranteed to be available when needed.

The take() semantics enforce single-consumer: each Receiver can only be consumed once. If a DAG has fan-out — one node's output feeds two downstream stages — the pipeline must use tee_stream (§3.6) to clone chunks to both consumers. The current implementation panics on double-take as a safety check; the pipeline builder is responsible for inserting tee nodes where needed.

6.5 Metrics Wrapper

The final output stream is wrapped in a metrics-collecting proxy that counts chunks, bytes, and pipeline duration: 

// Wrap output stream to track chunks, bytes, and pipeline duration.
while let Some(chunk) = raw.recv().await {
    match &chunk {
        StreamChunk::Data(d) => {
            metrics::STREAM_CHUNKS_TOTAL.inc();
            metrics::STREAM_BYTES_TOTAL.inc_by(d.len() as u64);
        }
        StreamChunk::End | StreamChunk::Error(_) => {
            metrics::STREAM_PIPELINE_DURATION.observe(start.elapsed());
        }
    }
    tx.send(chunk).await?;
}

This adds one extra channel hop but zero copying — the chunk is forwarded by move. The overhead is negligible compared to the compute work in each stage.

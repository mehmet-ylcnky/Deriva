6. StreamPipeline & DAG-Aware Construction

The StreamPipeline is the execution engine that wires streaming stages together. Unlike traditional stream processors that operate on flat topologies, Deriva's pipeline is constructed directly from the dependency DAG — the same graph that drives batch materialization. This section describes the pipeline data structures, the topological construction algorithm, and the tee mechanism for fan-out.

6.1 Pipeline Configuration

The PipelineConfig struct controls the four knobs that shape pipeline behavior. All four have sensible defaults; most pipelines run with zero configuration. 

pub struct PipelineConfig {
    pub chunk_size: usize,           // default: 64 KB
    pub channel_capacity: usize,    // default: 8 chunks
    pub cache_intermediates: bool,  // default: true
    pub memory_budget: usize,      // 0 = unlimited
}

chunk_size controls the granularity of data flow — the number of bytes per StreamChunk::Data payload. The default of 64 KB sits at the knee of the overhead curve (§8): 1 KB chunks incur 5.7× more overhead, while 256 KB chunks save only 8% at 4× the memory footprint per stage. channel_capacity sets the backpressure threshold per inter-stage channel. At the default of 8, each channel buffers up to 512 KB (8 × 64 KB) before the sender blocks — enough to absorb short throughput mismatches without unbounded growth. The capacity sweep (§8) shows that 8 captures 80–90% of available throughput gain. cache_intermediates enables tee-to-cache for intermediate results, allowing partial pipeline reuse on subsequent requests. memory_budget is reserved for future use — currently all pipelines rely on bounded-channel backpressure as the sole memory control mechanism.

6.2 Pipeline Nodes

The pipeline is a flat array of nodes, each representing one unit of work. The PipelineNode enum has four variants that cover every case the executor encounters: 

enum PipelineNode {
    Source   { addr: CAddr, data: Bytes },                  // leaf data from store
    Cached   { addr: CAddr, data: Bytes },                  // previously materialized
    StreamingStage { addr: CAddr, function: ..., input_indices: ... },
    BatchStage     { addr: CAddr, function: ..., input_indices: ... },
}

Source and Cached both hold a Bytes value and emit it through value_to_stream(), which slices the value into chunk_size pieces and sends them down a bounded channel. The two variants are functionally identical — the distinction exists for metrics and tracing, so operators can see how many pipeline nodes were served from cache vs fetched from the leaf store. StreamingStage holds an Arc<dyn StreamingComputeFunction> and a list of input indices pointing into the pipeline's node array. At execution time, it takes the input receivers and calls stream_execute(), which spawns a Tokio task internally. BatchStage holds an Arc<dyn ComputeFunction> and follows the auto-wrap path (§5): collect all input streams to Bytes, call execute(), then re-chunk the result via batch_to_stream(). The params field differs between the two: streaming functions take HashMap<String, String> (string-typed for channel-safe passing), while batch functions take BTreeMap<String, Value> (the original recipe parameter format).

6.3 DAG-to-Pipeline Construction

The StreamingExecutor builds the pipeline in three phases. Click through the diagram below to see each phase.

The executor walks the DAG depth-first from the target CAddr, producing a topological order where every node appears after all its dependencies. This is the same DagReader trait used by the batch materializer — the streaming executor reuses the existing DAG infrastructure rather than building a parallel graph representation. The visited set ensures each address is processed exactly once, even in diamond DAGs where multiple paths converge on the same node. 

fn resolve_order(dag: &dyn DagReader, addr: &CAddr) -> Result<Vec<CAddr>, DerivaError> {
    let mut order = Vec::new();
    let mut visited = HashSet::new();
    topo_visit(dag, addr, &mut visited, &mut order)?;
    Ok(order)
}

The recursive topo_visit calls dag.get_recipe(addr) for each address. If the address has a recipe (it's a computed node), the function recurses into its inputs before pushing the address onto the order vector — guaranteeing that dependencies appear before dependents. If the address has no recipe, it's a leaf blob and is skipped here; the node creation loop handles leaves separately.

Node Creation Loop

The materialize_streaming() method in StreamingExecutor iterates the topological order and builds the pipeline node array. For each address, it follows a four-step decision process:

  1. Check the cache. If cache.get(addr).await returns Some(data), insert a Cached node. This short-circuits the entire subtree — no function lookup, no input resolution needed.
  2. Resolve leaf inputs. For each input address in the recipe that isn't already in the pipeline (i.e., it's a raw blob, not a recipe), look it up in the cache first, then the leaf store. If an input is found in neither, the executor returns DerivaError::NotFound.
  3. Pick the function variant. The registry is queried streaming-first: registry.get_streaming(func_id). If a streaming implementation exists, a StreamingStage is created. Otherwise, registry.get(func_id) falls back to the batch variant, creating a BatchStage that will be auto-wrapped at execution time (§5). If neither exists, the executor returns DerivaError::FunctionNotFound.
  4. Record the index. Each add_* method returns the node's position in the pipeline array. This index is stored in an addr_to_idx: HashMap<CAddr, usize> so that downstream nodes can reference it in their input_indices vector.

6.4 Execution

The execute() method consumes the StreamPipeline and returns an mpsc::Receiver<StreamChunk> — the output stream of the final node. Internally, it maintains an outputs: Vec<Option<Receiver>> array, one slot per node. As it walks the node array in topological order, each node take()s its input receivers from the outputs array, processes them, and stores its own output receiver.

The take() semantics are critical: each Option<Receiver> can only be taken once. This enforces single-consumer — a receiver cannot be read by two stages simultaneously. If a DAG has fan-out (one node's output feeds two downstream stages), the pipeline builder must insert a tee_stream (§3) to clone chunks to both consumers. The current implementation panics on double-take as a safety check, making fan-out bugs immediately visible rather than silently producing incorrect results.

For Source and Cached nodes, execution is trivial: call value_to_stream(data, chunk_size, channel_capacity), which spawns a task that slices the Bytes value into chunks and sends them down a new channel. For StreamingStage nodes, the function's stream_execute() method receives the input receivers and returns an output receiver — the function is responsible for spawning its own processing task. For BatchStage nodes, each input receiver is collected to Bytes via collect_stream(), the batch function is called synchronously, and the result is re-chunked via batch_to_stream().

Because nodes are in topological order, every input receiver is guaranteed to exist in the outputs array when a node needs it. This invariant eliminates the need for runtime dependency tracking or readiness checks — the array index is sufficient.

6.5 Metrics Wrapper

The final output stream is not returned directly. Instead, execute() spawns a forwarding task that intercepts every chunk, updates three Prometheus metrics, and passes the chunk through to a new channel. The wrapper increments STREAM_PIPELINES_TOTAL at pipeline start, then per-chunk: 

// Wrap output stream to track chunks, bytes, and pipeline duration.
while let Some(chunk) = raw.recv().await {
    match &chunk {
        StreamChunk::Data(d) => {
            metrics::STREAM_CHUNKS_TOTAL.inc();
            metrics::STREAM_BYTES_TOTAL.inc_by(d.len() as u64);
        }
        StreamChunk::End | StreamChunk::Error(_) => {
            metrics::STREAM_PIPELINE_DURATION.observe(start.elapsed());
        }
    }
    tx.send(chunk).await?;
}

On each Data chunk, the wrapper increments the chunk counter and adds the payload length to the byte counter — both are atomic operations with negligible overhead. On End or Error, it records the total pipeline duration as a histogram observation, capturing the wall-clock time from pipeline start to stream termination. The chunk itself is forwarded by move (no copy), and the extra channel hop adds ~100 ns per chunk — negligible compared to the compute work in each stage. If the downstream consumer drops its receiver (e.g., client disconnects), the tx.send() returns an error and the forwarding task exits, which propagates cancellation upstream through the channel chain.

3. The StreamChunk Protocol

Every streaming pipeline in Deriva communicates through a single three-variant enum: StreamChunk. This type is the wire format between every pair of stages — sources, compute functions, adapters, and sinks all speak the same protocol. The design is deliberately minimal: three variants cover the entire lifecycle of a stream, from the first data byte to successful completion or failure.

The protocol draws inspiration from two traditions. From message-passing systems (Erlang, Akka), it borrows the idea of typed messages over bounded channels. From byte-stream protocols (HTTP chunked transfer, gRPC streaming), it borrows the framing model where a stream is a sequence of data frames terminated by a sentinel. The result is a protocol simple enough to implement in a single match arm, yet expressive enough to support backpressure, cancellation, error propagation, and zero-copy forwarding.

3.1 The Enum

The entire streaming protocol is captured in a single Rust enum with three variants. There are no configuration structs, no trait hierarchies, no generics — just three cases that every stage must handle: 

/// A chunk in a streaming computation pipeline.
/// Every inter-stage channel carries values of this type.
#[derive(Debug, Clone)]
pub enum StreamChunk {
    /// A data chunk — typically 64 KB of bytes.
    /// Size is advisory; consumers must not assume fixed sizes.
    Data(Bytes),
    /// End of stream. Exactly one per stream. No chunks follow.
    End,
    /// Stream error. Terminal. No chunks follow.
    Error(DerivaError),
}

Three variants, three roles:

  • Data(Bytes) — carries a payload. The Bytes type (from the bytes crate) is a reference-counted, cheaply cloneable byte buffer. Size is advisory: the default is 64 KB via DEFAULT_CHUNK_SIZE, but functions may emit smaller chunks (e.g., a filter that drops most input) or larger ones (e.g., a compression stage that buffers internally). Consumers must not assume fixed sizes — they should process whatever arrives.
  • End — signals successful completion. Exactly one End must terminate every successful stream. After End, no further chunks are sent. This is the sentinel that tells the consumer "you have all the data; finalize your output." For cache-after-collect stages, End triggers the hash computation and cache write.
  • Error(DerivaError) — signals failure. Terminal like End — no further chunks follow. Downstream stages propagate the error and stop processing. The DerivaError carries context (source stage, error kind, message) so that the pipeline can report which stage failed and why.

The choice of Bytes over Vec<u8> is deliberate. Bytes supports zero-copy slicing (bytes.slice(start..end) shares the underlying allocation) and cheap cloning (reference count increment, not memcpy). This matters for tee_stream (§3.6), where each chunk is sent to two consumers — with Bytes, the clone is O(1) regardless of chunk size.

3.2 Protocol Invariants

Every stream — whether produced by a source, a compute function, or the batch_to_stream adapter — must satisfy three invariants. These are not enforced at the type level (Rust's type system cannot express "exactly one terminal variant"), but they are upheld by convention and tested in the benchmark suite.

  1. Termination. Every stream ends with exactly one End or one Error. Never both, never neither (unless the receiver is dropped, which is treated as cancellation — see §3.5). This invariant ensures that consumers always know when the stream is complete. Without it, a consumer would need a timeout to distinguish "slow producer" from "producer crashed."
  2. Ordering. Zero or more Data chunks precede the terminal. No Data follows End or Error. This means a consumer can stop reading as soon as it sees a terminal variant — there is no need to drain remaining chunks. The ordering invariant also simplifies the collect_stream helper: it appends Data payloads to a buffer and returns when it sees End.
  3. Non-empty data. Data(Bytes) chunks should contain at least one byte. Empty chunks are harmless (they are simply ignored by consumers) but wasteful — they consume a channel slot without carrying payload. The batch_to_stream helper enforces this by skipping zero-length slices at the end of input.

Together, these invariants make the protocol a regular language over the alphabet {Data, End, Error}: the valid sequences are Data* End and Data* Error. Any other sequence (e.g., End Data, Error End, or an infinite stream of Data with no terminal) is a protocol violation.

3.3 Chunk Lifecycle

A chunk's journey from producer to consumer passes through four distinct phases. The interactive diagram below visualizes this lifecycle using a bounded tokio::mpsc channel with 8 slots. Click the diagram to advance through each phase:

  • Phase 1 — Data flows: The producer emits Data(Bytes) chunks into the channel. Each chunk occupies one slot in the 8-slot buffer. The animated dot represents chunks flowing from left (producer) to right (consumer). As long as the channel has room, the producer sends without blocking and the consumer reads and processes each chunk incrementally — this is the steady-state of a healthy pipeline.
  • Phase 2 — Backpressure: The channel fills to capacity (8/8 slots occupied, shown in orange). The producer's send().await suspends — the Tokio runtime parks the task until the consumer frees a slot by calling recv().await. No data is lost, no explicit backpressure signal is needed. The producer simply waits. This is the mechanism that keeps memory bounded regardless of input size.
  • Phase 3 — End of stream: The producer sends StreamChunk::End — exactly one per stream, shown in green. The consumer sees End, finalizes its output (e.g., flushes an accumulator, computes a hash for caching), and drops its receiver. The channel is deallocated.
  • Phase 4 — Error propagation: If the producer encounters a failure, it sends StreamChunk::Error(e) instead of End, shown in red. The consumer receives the error, propagates it downstream (causing the next stage to also emit Error), and the entire pipeline tears down stage by stage.

Backpressure — the mechanism by which a slow consumer slows down a fast producer — is the single most important property for bounded-memory streaming. Without it, a fast producer would fill memory with buffered chunks until the system runs out of RAM. Deriva's backpressure is implicit: it falls out of the channel semantics with zero additional code.

Each inter-stage connection is a tokio::mpsc::channel with a fixed capacity (default: 8 chunks). At the default chunk size of 64 KB, this means 512 KB of buffered data per edge. The mechanism is straightforward:

  1. Producer calls tx.send(chunk).await.
  2. If the channel has room, the send completes immediately — the chunk is moved into the buffer and the producer continues to the next chunk.
  3. If the channel is full (all 8 slots occupied), the .await suspends the producer's Tokio task. The runtime parks the task and schedules other work. No busy-waiting, no polling, no CPU wasted.
  4. When the consumer calls rx.recv().await and reads a chunk, a slot opens. The Tokio runtime wakes the suspended producer, which sends its next chunk into the freed slot.
  5. If the consumer drops the receiver (cancellation), send() returns Err(SendError), and the producer stops — see §3.5.

This gives bounded memory without explicit backpressure signals, flow-control frames, credit-based schemes, or watermarks. The total memory ceiling for a pipeline with N edges is N × capacity × chunk_size. For a 5-stage pipeline with default settings: 5 × 8 × 64 KB = 2.5 MB — whether the input is 4 MB or 512 MB. The benchmarks in §8 confirm this: streaming memory usage remains flat across all input sizes, while batch memory scales linearly.

The capacity of 8 was chosen based on the empirical sweep in §8.7, which tested capacities 1 through 64 across input sizes from 1 MB to 512 MB. The results show that capacity 8 sits at the knee of the throughput curve: it is 1.6–3.1× faster than capacity 1 (which suspends the producer after every chunk), while capacity 16–64 offers only 5–21% further improvement at 2–8× the memory cost. At capacity 8, a 5-stage pipeline buffers 2.5 MB total — enough to absorb producer/consumer speed mismatches without wasting memory. See §8.7 for the full data table and analysis.

3.5 Cancellation

Cancellation is receiver-driven and requires zero explicit coordination. When a consumer no longer needs data — because the client disconnected, a downstream error occurred, or a cache hit made the computation unnecessary — it simply drops its mpsc::Receiver. The next tx.send() in the producer returns Err(SendError), and the producer's spawned Tokio task exits cleanly.

This propagates backward through the pipeline automatically: each stage, upon receiving a send error, drops its own input receiver, causing the upstream stage to also receive a send error and exit. In a 5-stage pipeline, cancellation at the final consumer tears down all 5 stages in reverse order — typically within microseconds, since each stage is simply dropping a channel handle.

No explicit cancel tokens, no poison pills, no shutdown coordination protocol, no CancellationToken from tokio_util. Rust's ownership model (dropping a Receiver closes the channel) and Tokio's channel semantics (send() returns Err on a closed channel) provide graceful teardown for free. This is one of the key advantages of building on Rust's ownership system rather than a garbage-collected runtime where channel handles might linger.

3.6 Helper Functions

Four utility functions bridge between the batch world (a single Bytes buffer) and the streaming world (a channel of StreamChunk values). These are the glue that enables mixed pipelines where some stages are natively streaming and others are batch functions wrapped by the auto-wrapping mechanism described in §5. 

/// Wraps a complete Bytes value as a chunked stream.
/// Slices the input into chunk_size pieces, sends each as Data,
/// then sends End. Spawns a Tokio task for the sending loop.
fn batch_to_stream(data: Bytes, chunk_size: usize) -> Receiver<StreamChunk>

/// Same as batch_to_stream with explicit channel capacity.
/// Useful when the caller needs tighter or looser buffering.
fn value_to_stream(data: Bytes, chunk_size: usize, capacity: usize) -> Receiver<StreamChunk>

/// Collects a stream back into a single Bytes value.
/// Appends each Data chunk to a BytesMut buffer, returns on End,
/// returns Err on Error. This is the inverse of batch_to_stream.
async fn collect_stream(rx: Receiver<StreamChunk>) -> Result<Bytes, DerivaError>

/// Splits one stream into two identical streams (clone each chunk).
/// Both outputs receive every Data chunk and the terminal.
fn tee_stream(input: Receiver<StreamChunk>, capacity: usize)
    -> (Receiver<StreamChunk>, Receiver<StreamChunk>)

batch_to_stream is the entry point for injecting batch data into a streaming pipeline. It slices a Bytes buffer into chunks of the configured size (default 64 KB), sends each as Data, and finishes with End. The slicing uses Bytes::slice(), which is zero-copy — each chunk shares the underlying allocation with the original buffer. The sending loop runs in a spawned Tokio task, so the caller gets back a Receiver immediately and can begin consuming chunks before the entire input has been sliced.

collect_stream does the inverse: it reads chunks from a Receiver, appends each Data payload to a BytesMut accumulator, and returns the frozen Bytes when it sees End. If it encounters Error, it returns the error immediately without waiting for further chunks. This function is used by the cache-after-collect strategy (§7) and by the auto-wrapping adapter that feeds batch functions.

tee_stream is the most interesting helper. It spawns a task that reads from the input channel and clones each chunk to two output channels. Because Bytes::clone() is O(1) (reference count increment), the per-chunk overhead is negligible — roughly 10 ns per clone regardless of chunk size. This enables the pipeline builder to simultaneously deliver streaming output to the client and write the collected result to the cache, without buffering the entire output twice. The pipeline construction in §6 uses tee_stream for exactly this purpose.

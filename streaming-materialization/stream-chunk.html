3. The StreamChunk Protocol

Every streaming pipeline in Deriva communicates through a single three-variant enum. This section defines the protocol, its invariants, and the backpressure mechanism that keeps memory bounded.

3.1 The Enum

/// A chunk in a streaming computation pipeline.
#[derive(Debug, Clone)]
pub enum StreamChunk {
    /// A data chunk — typically 64 KB of bytes.
    Data(Bytes),
    /// End of stream. Exactly one per stream. No chunks follow.
    End,
    /// Stream error. Terminal. No chunks follow.
    Error(DerivaError),
}

Three variants, three roles:

  • Data(Bytes) — carries a payload. Size is advisory (default 64 KB via DEFAULT_CHUNK_SIZE), but functions may emit smaller or larger chunks. Consumers must not assume fixed sizes.
  • End — signals successful completion. Exactly one End must terminate every successful stream. After End, no further chunks are sent.
  • Error(DerivaError) — signals failure. Terminal like End. Downstream stages propagate the error and stop processing.

3.2 Protocol Invariants

Every stream — whether produced by a source, a compute function, or the batch_to_stream adapter — must satisfy:

  1. Termination. Every stream ends with exactly one End or one Error. Never both, never neither (unless the receiver is dropped, which is treated as cancellation).
  2. Ordering. Zero or more Data chunks precede the terminal. No Data follows End or Error.
  3. Non-empty data. Data(Bytes) chunks should contain at least one byte. Empty chunks are harmless but wasteful.

3.4 Backpressure via Bounded Channels

Backpressure is implicit. Each inter-stage connection is a tokio::mpsc::channel with a fixed capacity (default: 8 chunks × 64 KB = 512 KB of buffered data per edge). The mechanism:

  1. Producer calls tx.send(chunk).await.
  2. If the channel has room, the send completes immediately.
  3. If the channel is full, the .await suspends the producer's task until the consumer calls rx.recv().await and frees a slot.
  4. If the consumer drops the receiver (cancellation), send() returns Err, and the producer stops.

This gives bounded memory without explicit backpressure signals, flow-control frames, or watermarks. The total memory ceiling for a pipeline with N edges is N × capacity × chunk_size. For a 5-stage pipeline: 5 × 8 × 64 KB = 2.5 MB — regardless of input size.

3.5 Cancellation

Cancellation is receiver-driven. When a consumer no longer needs data — because the client disconnected, a downstream error occurred, or a cache hit made the computation unnecessary — it drops its mpsc::Receiver. The next tx.send() in the producer returns Err(SendError), and the producer's spawned task exits. This propagates backward through the pipeline: each stage drops its input receivers, causing upstream stages to terminate in turn.

No explicit cancel tokens, no poison pills, no shutdown coordination. Rust's ownership model and Tokio's channel semantics provide graceful teardown for free.

3.6 Helper Functions

Four utility functions bridge between batch and streaming worlds: 

/// Wraps a complete Bytes value as a chunked stream.
fn batch_to_stream(data: Bytes, chunk_size: usize) -> Receiver<StreamChunk>

/// Same as batch_to_stream with explicit channel capacity.
fn value_to_stream(data: Bytes, chunk_size: usize, capacity: usize) -> Receiver<StreamChunk>

/// Collects a stream back into a single Bytes value.
async fn collect_stream(rx: Receiver<StreamChunk>) -> Result<Bytes, DerivaError>

/// Splits one stream into two identical streams (clone each chunk).
fn tee_stream(input: Receiver<StreamChunk>, capacity: usize)
    -> (Receiver<StreamChunk>, Receiver<StreamChunk>)

batch_to_stream slices a Bytes buffer into 64 KB chunks and sends them followed by End. collect_stream does the inverse — accumulates all Data chunks, returns Err on Error, and returns the concatenated buffer on End. tee_stream clones each chunk to two outputs, enabling simultaneous cache-write and client-delivery (§6).

4. Streaming Compute Functions

Deriva ships 20 built-in streaming functions organized into four categories. Each implements the StreamingComputeFunction trait — accepting one or more Receiver<StreamChunk> inputs and returning a single Receiver<StreamChunk> output. This section describes the trait contract, the internal helper patterns, and all 20 functions.

4.1 The Trait

#[async_trait]
pub trait StreamingComputeFunction: Send + Sync {
    /// Process streaming inputs and produce streaming output.
    async fn stream_execute(
        &self,
        inputs: Vec<mpsc::Receiver<StreamChunk>>,
        params: &HashMap<String, String>,
    ) -> mpsc::Receiver<StreamChunk>;

    /// Preferred chunk size in bytes (default: 64 KB).
    fn preferred_chunk_size(&self) -> usize { DEFAULT_CHUNK_SIZE }

    /// Output channel capacity (default: 8 chunks).
    fn channel_capacity(&self) -> usize { DEFAULT_CHANNEL_CAPACITY }
}

The trait is intentionally minimal. A function receives its inputs as a vector of receivers (one per dependency in the DAG), a parameter map for configuration, and returns a single output receiver. The function spawns its own Tokio task internally — the caller never blocks.

4.2 Internal Helpers

Two helper patterns eliminate boilerplate across the 20 implementations: 

/// Map: read one input, transform each Data chunk, forward.
fn spawn_map(
    rx: Receiver<StreamChunk>,
    cap: usize,
    f: impl Fn(&[u8]) -> Result<Bytes, String> + Send + 'static,
) -> Receiver<StreamChunk>

/// Accumulate: fold all chunks into state, emit single result.
fn spawn_accumulate<S: Send + 'static>(
    rx: Receiver<StreamChunk>,
    init: S,
    fold: impl Fn(&mut S, &[u8]) + Send + 'static,
    finalize: impl FnOnce(S) -> Bytes + Send + 'static,
) -> Receiver<StreamChunk>

spawn_map handles the 9 transforms (category 1): it reads chunks, applies a closure, and forwards the result. Error from the closure becomes StreamChunk::Error. spawn_accumulate handles the 3 accumulators (category 2): it folds all chunks into a state value, then emits a single Data chunk followed by End.

4.3 The 20 Built-in Functions

Click a category in the diagram below to see its functions. Each category uses a different execution pattern.

Category 1: Transforms (9 functions)

All nine use spawn_map: read a chunk, apply a pure function, emit the result. Memory is O(chunk_size) — one chunk in, one chunk out. The transform closure is Fn(&[u8]) -> Result<Bytes, String>, so errors (e.g., invalid base64) become StreamChunk::Error and terminate the stream.

Notable: StreamingCompress and StreamingDecompress operate per-chunk, not across the full stream. This means each chunk is independently compressed — suitable for random-access decompression but with lower compression ratios than whole-stream compression.

Category 2: Accumulators (3 functions)

All three use spawn_accumulate: fold each chunk into a rolling state, then emit a single result on End. These are inherently non-streaming in output — they must see all input before producing any output — but they are streaming in input: they never buffer the full input, only the fixed-size state (32 bytes for SHA-256, 8 bytes for byte count, 4 bytes for CRC32).

Category 3: Combiners (3 functions)

These accept multiple input receivers. StreamingConcat drains inputs sequentially — all chunks from input 0, then input 1, etc. StreamingInterleave round-robins: one chunk from each input in turn. StreamingZipConcat reads one chunk from each of exactly two inputs and concatenates them pairwise.

Category 4: Utilities (5 functions)

Mixed patterns. StreamingChunkResizer maintains an internal BytesMut buffer and emits chunks of exactly the target size. StreamingTake and StreamingSkip track a byte offset and slice chunks at the boundary. StreamingRepeat is the only function that buffers the entire input — it must replay the full content N times. StreamingTeeCount combines pass-through with a rolling byte counter, appending the count as a decimal ASCII trailer after the final data chunk.

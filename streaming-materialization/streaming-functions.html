4. Streaming Compute Functions

Deriva ships 20 built-in streaming functions organized into four categories. Each implements the StreamingComputeFunction trait — accepting one or more Receiver<StreamChunk> inputs and returning a single Receiver<StreamChunk> output. The function spawns its own Tokio task internally, so the caller receives a Receiver immediately and can begin consuming output before the function has finished processing all input.

The 20 functions are not intended to be exhaustive — they are a foundation that demonstrates the four fundamental execution patterns (map, accumulate, combine, utility) and provides enough variety for realistic pipelines. Users register additional functions via the FunctionRegistry by implementing the same trait. The built-in set was chosen to cover common data processing operations (encoding, hashing, compression, slicing) while exercising every edge case in the protocol: single-input vs multi-input, chunk-preserving vs chunk-resizing, streaming-output vs batch-output, and stateless vs stateful.

4.1 The Trait

The entire streaming function contract is captured in a single trait with one required method and two optional configuration hooks: 

#[async_trait]
pub trait StreamingComputeFunction: Send + Sync {
    /// Process streaming inputs and produce streaming output.
    /// Called by the pipeline executor for each compute node.
    async fn stream_execute(
        &self,
        inputs: Vec<mpsc::Receiver<StreamChunk>>,
        params: &HashMap<String, String>,
    ) -> mpsc::Receiver<StreamChunk>;

    /// Preferred chunk size in bytes (default: 64 KB).
    /// The pipeline builder uses this to configure upstream chunking.
    fn preferred_chunk_size(&self) -> usize { DEFAULT_CHUNK_SIZE }

    /// Output channel capacity (default: 8 chunks).
    /// Higher values trade memory for reduced task switching.
    fn channel_capacity(&self) -> usize { DEFAULT_CHANNEL_CAPACITY }
}

The trait is intentionally minimal — three methods, two of which have defaults. A function receives its inputs as a vector of receivers (one per dependency in the recipe DAG), a parameter map for runtime configuration (e.g., {"key": "0x42"} for XOR, {"bytes": "1024"} for Take), and returns a single output receiver. The function is responsible for spawning its own Tokio task internally — the caller never blocks waiting for the function to complete.

The Send + Sync bounds are required because functions are stored in the FunctionRegistry as Arc<dyn StreamingComputeFunction> and may be invoked concurrently from multiple pipeline executions. The async_trait macro (from the async-trait crate) desugars the async method into a boxed future, which is the standard pattern for async trait methods in Rust prior to native async trait stabilization.

The two configuration hooks — preferred_chunk_size and channel_capacity — allow functions to override the pipeline defaults. For example, a compression function might prefer larger chunks (256 KB) for better compression ratios, while a hash function might prefer the default 64 KB since SHA-256 processes data in 64-byte blocks regardless of chunk size. The capacity sweep in §8.7 validates the default capacity of 8.

4.2 Internal Helpers

Implementing 20 streaming functions from scratch would involve significant boilerplate: each needs a channel pair, a spawned task, a recv loop, error propagation, and terminal forwarding. Two helper functions eliminate this repetition by capturing the two most common execution patterns: 

/// Map: read one input, transform each Data chunk, forward.
/// Handles End/Error propagation automatically.
fn spawn_map(
    rx: Receiver<StreamChunk>,
    cap: usize,
    f: impl Fn(&[u8]) -> Result<Bytes, String> + Send + 'static,
) -> Receiver<StreamChunk>

/// Accumulate: fold all chunks into state, emit single result.
/// Calls fold() for each Data chunk, finalize() on End.
fn spawn_accumulate<S: Send + 'static>(
    rx: Receiver<StreamChunk>,
    init: S,
    fold: impl Fn(&mut S, &[u8]) + Send + 'static,
    finalize: impl FnOnce(S) -> Bytes + Send + 'static,
) -> Receiver<StreamChunk>

spawn_map covers 9 of the 20 functions (all of Category 1). It creates a channel pair, spawns a Tokio task that reads chunks from the input receiver, applies the closure f to each Data chunk's bytes, and sends the result downstream. If the closure returns Err, the helper sends StreamChunk::Error and stops. On End, it forwards End. On Error, it propagates the error. The entire recv-transform-send loop is ~15 lines of code, but writing it 9 times would be error-prone — especially the edge cases around send failures (receiver dropped) and error propagation.

spawn_accumulate covers the 3 accumulators (Category 2). It takes an initial state value, a fold function that updates the state for each chunk, and a finalize function that converts the final state into a Bytes output. The pattern is analogous to Rust's Iterator::fold, but operating over an async stream of chunks. The key property is that the state is fixed-size — 32 bytes for SHA-256, 8 bytes for a counter, 4 bytes for CRC32 — so memory usage is O(1) regardless of input size, even though the function must see all input before producing output.

The remaining 8 functions (Categories 3 and 4) have custom implementations because their execution patterns don't fit either helper: combiners need multi-receiver loops, and utilities like StreamingChunkResizer maintain internal buffers that don't map to a simple map or fold.

4.3 The 20 Built-in Functions

The diagram below organizes all 20 functions into four categories based on their execution pattern. Click a category tab to expand its function list with descriptions. Each category uses a distinct pattern — map, accumulate, multi-input combine, or custom — that determines its memory profile, latency characteristics, and composability within a pipeline.

Category 1: Transforms (9 functions)

All nine use spawn_map: read a chunk, apply a pure function, emit the result. Memory is O(chunk_size) — one chunk in, one chunk out, with no accumulation across chunks. The transform closure is Fn(&[u8]) -> Result<Bytes, String>, so errors (e.g., invalid base64 input) become StreamChunk::Error and terminate the stream cleanly.

The simplest is StreamingIdentity — it forwards each chunk unchanged. Despite doing no useful work, it is essential for benchmarking: it isolates the overhead of the streaming infrastructure (channel creation, task spawning, chunk framing) from the cost of actual computation. The identity pipeline benchmarks in §8 use chains of StreamingIdentity stages for exactly this purpose.

StreamingUppercase and StreamingLowercase operate on ASCII bytes — they apply to_ascii_uppercase() / to_ascii_lowercase() per byte within each chunk. These are useful for text normalization pipelines and serve as simple examples of byte-level transforms.

StreamingBase64Encode and StreamingBase64Decode encode/decode each chunk independently. This means the base64 encoding is not continuous across chunk boundaries — each chunk is a self-contained base64 unit. This is a deliberate tradeoff: it enables random-access decoding (decode chunk N without reading chunks 0..N-1) at the cost of ~33% overhead per chunk boundary due to base64 padding.

StreamingCompress and StreamingDecompress apply zlib compression per chunk. Like base64, this is per-chunk rather than whole-stream — each chunk is independently compressed. This yields lower compression ratios than whole-stream compression (no cross-chunk dictionary sharing) but enables parallel decompression and random access. For workloads where compression ratio matters more than random access, a future whole-stream compressor could be added.

StreamingXor XORs each byte with a single-byte key from params["key"]. It demonstrates parameterized transforms — the same function struct handles different keys based on runtime configuration. StreamingReverse reverses the bytes within each chunk (not across the stream), useful for testing byte-order-sensitive pipelines.

Category 2: Accumulators (3 functions)

All three use spawn_accumulate: fold each chunk into a rolling state, then emit a single result on End. These are inherently non-streaming in output — they must see all input before producing any output — but they are streaming in input: they never buffer the full input, only the fixed-size state.

StreamingSha256 maintains a sha2::Sha256 hasher (32 bytes of internal state). Each Data chunk is fed into the hasher via update(). On End, the hasher is finalized and the 32-byte digest is emitted as a single Data chunk followed by End. This is functionally identical to computing sha256(entire_input), but without ever holding the entire input in memory. For a 512 MB input, the accumulator uses 32 bytes of state instead of 512 MB of buffer.

StreamingByteCount maintains a u64 counter (8 bytes). Each chunk adds chunk.len() to the counter. On End, the count is emitted as an 8-byte big-endian value. StreamingChecksum maintains a CRC32 state (4 bytes) and emits the 4-byte checksum on completion.

Accumulators are the only category where time-to-first-byte equals total latency — the consumer sees no output until the entire input has been processed. In a pipeline, an accumulator stage acts as a "barrier": all upstream stages must complete before any downstream stage can begin. The pipeline builder in §6 is aware of this and avoids placing accumulators in the middle of latency-sensitive paths when possible.

Category 3: Combiners (3 functions)

These accept multiple input receivers and merge them into a single output stream. Each combiner has a distinct merging strategy:

  • StreamingConcat drains inputs sequentially — all chunks from input 0 (until its End), then all chunks from input 1, and so on. The output is the concatenation of all inputs in order. This is the streaming equivalent of cat file1 file2 file3. Memory is O(chunk_size) since only one input is active at a time.
  • StreamingInterleave round-robins: one chunk from input 0, one from input 1, one from input 2, back to input 0, etc. When an input ends, it is removed from the rotation. This is useful for merging multiple data sources into a single stream with fair scheduling. Memory is O(N × chunk_size) where N is the number of active inputs, since each input may have one chunk buffered in its channel.
  • StreamingZipConcat reads one chunk from each of exactly two inputs and concatenates them pairwise — chunk 0 from input A + chunk 0 from input B, then chunk 1 from A + chunk 1 from B, etc. If one input ends before the other, the remaining chunks from the longer input are passed through. This is useful for combining parallel computation results (e.g., data + metadata).

All three combiners handle the End and Error protocol correctly: if any input produces Error, the combiner propagates it downstream and drops all remaining input receivers (triggering upstream cancellation via the mechanism in §3.5).

Category 4: Utilities (5 functions)

The utility category groups functions with mixed execution patterns that don't fit the map or accumulate helpers:

  • StreamingChunkResizer re-chunks the input stream to a target size specified by params["target_size"]. It maintains an internal BytesMut buffer: incoming chunks are appended to the buffer, and whenever the buffer reaches the target size, a chunk is sliced off and sent downstream. On End, any remaining bytes in the buffer are flushed as a final (possibly smaller) chunk. This is essential when a downstream function requires a specific chunk size — for example, a block cipher that needs exact 16-byte blocks.
  • StreamingTake emits the first params["bytes"] bytes of the input, then sends End and drops the input receiver (cancelling upstream). It tracks a running byte offset and slices the boundary chunk at the exact cutoff point. Memory is O(chunk_size).
  • StreamingSkip is the complement of Take: it discards the first params["bytes"] bytes and passes the rest through. Like Take, it tracks a byte offset and slices the boundary chunk. Together, Take and Skip enable range reads: Skip(offset) → Take(length) extracts an arbitrary byte range from a stream.
  • StreamingRepeat replays the entire input params["count"] times. This is the only built-in function that buffers the entire input — it collects all Data chunks into a Vec<Bytes>, then replays them N times. Memory is O(input_size), making it unsuitable for very large inputs. It exists for testing and for use cases where repetition is needed (e.g., generating test data from a template).
  • StreamingTeeCount combines pass-through with a rolling byte counter. All Data chunks are forwarded unchanged, and after the last data chunk, the function appends an additional Data chunk containing the total byte count as a decimal ASCII string (e.g., "1048576"), followed by End. This demonstrates a function that augments the stream with computed metadata without modifying the original data.

The utility category is the most likely to grow as new use cases emerge. Candidates for future additions include a StreamingFilter (drop chunks matching a predicate), StreamingRateLimit (throttle output to a target bytes/sec), and StreamingEncrypt / StreamingDecrypt (AES-GCM with streaming authenticated encryption).

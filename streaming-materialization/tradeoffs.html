9. Tradeoffs & Design Decisions

Streaming materialization is not a free upgrade over batch. It introduces new complexity axes that system designers must navigate. This section maps the five key tradeoffs, explains when each matters, and provides concrete guidance for choosing between batch and streaming paths.

9.1 When to Use Batch

Batch materialization remains the right choice when:

  • Data is small (<3 MB) — streaming's fixed overhead (channel creation, Tokio task spawning, chunk framing) exceeds the benefit of pipeline parallelism. At 1 MB, batch is 9.3× faster.
  • Single-stage computation — with no pipeline parallelism to exploit, streaming adds overhead with no upside. A direct blob_store_get returns 1 MB in 398 µs; streaming identity takes 1.59 ms.
  • Result will be immediately cached — if the output is small and will be served from cache on subsequent reads, the streaming infrastructure cost is wasted.

9.2 When to Use Streaming

Streaming materialization wins when:

  • Multi-stage pipelines on data >3 MB — pipeline parallelism overlaps stages. At 32 MB with 5 stages, streaming is 2.7× faster than batch.
  • Time-to-first-byte matters — streaming delivers the first 64 KB chunk in ~1.6 ms regardless of data size or pipeline depth. Batch must complete all stages first.
  • Memory must be bounded — a 5-stage streaming pipeline on 512 MB data uses at most 2.5 MB of buffer memory (D × N × chunk_size = 5 × 8 × 64 KB). Batch would buffer 5 × 512 MB = 2.5 GB.
  • Graceful cancellation — streaming pipelines can be cancelled mid-stream by dropping the receiver. Batch is all-or-nothing.

9.3 Chunk Size Selection Guide

The default 64 KB chunk size balances three concerns:

  • Channel overhead — each chunk requires one mpsc::send + recv (~100 ns uncontended). At 1 KB chunks, a 1 MB input generates 1,000 round-trips (8.56 ms). At 64 KB, only 16 round-trips (1.50 ms).
  • First-byte latency — smaller chunks mean faster TTFB. At 64 KB, the first chunk arrives after ~94 µs of compute. At 1 MB, TTFB equals total latency.
  • Memory footprint — per-stage buffer = channel_capacity × chunk_size. At 64 KB with 8 slots: 512 KB per stage. At 1 MB: 8 MB per stage.

Override the default only when you have a specific reason: use smaller chunks (16 KB) for latency-sensitive interactive responses, larger chunks (256 KB+) for bulk ETL where throughput dominates.

9.4 The Auto-Wrap Tax

Auto-wrapping (§5) lets any batch ComputeFunction participate in a streaming pipeline by collecting all input chunks, running the batch function, and re-chunking the output. This preserves API compatibility but defeats two key streaming benefits:

  • Bounded memory — the wrapper must buffer the entire input before calling the batch function, reverting to O(n) memory for that stage.
  • Pipeline parallelism — the wrapped stage cannot overlap with its predecessor because it waits for all input before starting. Downstream stages are blocked until the batch function completes.

Auto-wrap is a migration tool, not a long-term solution. For performance-critical pipelines, implement native StreamingFunction variants. The 20 built-in streaming functions (§4) cover the most common operations.

9.5 Accumulator Stages

Accumulators (SHA-256, byte count, checksum) are inherently non-streaming in their output — they must consume all input before emitting a result. However, they still benefit from streaming input: instead of buffering the full input in memory, they update internal state chunk-by-chunk (e.g., Sha256::update(chunk)) and emit one summary chunk on StreamChunk::End.

An accumulator in the middle of a pipeline creates a pipeline bubble: downstream stages stall until the accumulator finishes. Place accumulators at the end of pipelines when possible, or use StreamingTeeCount / StreamingTeeChecksum to compute metadata in parallel with the main data flow without blocking downstream.

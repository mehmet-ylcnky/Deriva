9. Tradeoffs & Design Decisions

Streaming materialization is not a free upgrade over batch. It introduces new complexity axes that system designers must navigate: additional infrastructure (channels, spawned tasks, chunk framing), new failure modes (backpressure stalls, partial stream errors), and performance characteristics that depend on input size, pipeline depth, and function category. The batch path remains the right choice for a significant class of workloads — the benchmarks in §8 show batch winning by up to 9.3× for small inputs.

This section maps the five key tradeoffs, explains when each matters, and provides concrete guidance for choosing between batch and streaming paths. 9.1 When to Use Batch

Batch materialization remains the right choice when the overhead of the streaming infrastructure exceeds the benefit of pipeline parallelism. The decision is not about capability — every batch function can participate in a streaming pipeline via auto-wrapping (§5) — but about whether the streaming path actually improves performance for the specific workload.

  • Data is small (<3 MB). The batch vs streaming sweep in §8 shows a clear crossover point around 3–4 MB for a 5-stage identity pipeline. At 1 MB, batch completes in 0.17 ms while streaming takes 1.59 ms — a 9.3× penalty. At 2 MB, the gap narrows to 5.7× (0.41 ms vs 2.36 ms). The streaming overhead is fixed: creating 5 channel pairs (~5 × 200 ns), spawning 5 Tokio tasks (~5 × 1 µs), and framing the input into chunks (~16 chunks at 64 KB). For small inputs, this fixed cost dominates the actual computation. The future size-aware execution mode (§1 in future improvements) will automate this decision by checking input size against a configurable threshold before selecting the execution path.
  • Single-stage computation. Streaming's primary advantage is pipeline parallelism — overlapping the execution of multiple stages so that stage N+1 begins processing before stage N finishes. With only one stage, there is nothing to overlap. A direct blob_store_get → execute → return path avoids all channel overhead. The streaming path still provides bounded memory and cancellation, but if the function is a simple transform on small data, these benefits are irrelevant.
  • Result will be immediately cached and is small. If the output is under a few MB and will be served from the CAS cache on all subsequent reads, the streaming infrastructure cost is amortized over exactly one execution. The cache stores the final Bytes value, not the stream — so the streaming path must collect the output into Bytes for caching anyway (see §7). For a 500 KB result that will be read 1,000 times from cache, the one-time streaming overhead of ~1 ms is negligible, but it is also unnecessary.
  • The function is inherently batch. Some operations require the full input before producing any output: sorting, global deduplication, JSON schema validation, PDF page extraction, Parquet column projection. These functions would call collect_stream internally even if given a streaming interface. Using them through auto-wrapping is not a compromise — it is the correct execution strategy. The 148 batch-only functions identified in the format-aware function catalog fall into this category.

9.2 When to Use Streaming

Streaming materialization wins when the workload can exploit pipeline parallelism, when memory must be bounded, or when the consumer benefits from seeing partial results early. The advantage grows with input size and pipeline depth:

  • Multi-stage pipelines on data >3 MB. Pipeline parallelism overlaps the execution of adjacent stages: while stage 3 processes chunk N, stage 2 processes chunk N+1, and stage 1 processes chunk N+2. The more stages and the larger the data, the greater the overlap. At 32 MB with 5 stages, streaming is 2.7× faster than batch (14.4 ms vs 38.7 ms) — the peak advantage in the benchmark sweep. Even at 512 MB, streaming maintains a 1.3× advantage (564 ms vs 744 ms). The diminishing advantage at very large sizes reflects the fact that the pipeline eventually becomes compute-bound rather than latency-bound, and the overlap benefit saturates.
  • Time-to-first-byte matters. Streaming delivers the first 64 KB chunk in ~1.6 ms regardless of data size or pipeline depth. This is the time to create the channel chain, spawn the tasks, read the first chunk from the source, and propagate it through one stage. Batch must complete all stages before the client sees any data — for a 512 MB, 5-stage pipeline, that is 744 ms of silence followed by the full result. For interactive applications (file previews, progressive rendering, HTTP range requests), the streaming path provides a qualitatively different user experience: the client begins receiving data almost immediately.
  • Memory must be bounded. A 5-stage streaming pipeline on 512 MB data uses at most 2.5 MB of buffer memory: 5 stages × 8 channel slots × 64 KB chunk size (see the capacity sweep in §8.7). The same pipeline in batch mode buffers the full input at each stage: 5 × 512 MB = 2.5 GB in the worst case (though in practice, stages execute sequentially and release memory, so the peak is closer to 2 × 512 MB). The 1,000× memory difference is the most compelling argument for streaming on large data — it determines whether a pipeline can run on a constrained worker node or requires a large-memory instance.
  • Graceful cancellation. Streaming pipelines can be cancelled mid-stream by dropping the output receiver. The cancellation propagates backward through the channel chain (see §3.5): each stage detects the closed channel on its next send() and exits. For a StreamingTake(1_000_000) on a 512 MB input, only ~1 MB is actually read and processed — the remaining 511 MB is never touched. Batch is all-or-nothing: the function runs to completion even if the consumer only needs the first byte of the result.
  • Backpressure is needed. When a fast producer feeds a slow consumer (e.g., a compression stage feeding a network writer), the bounded channel automatically throttles the producer. Without streaming, the producer would generate the full output in memory, potentially exhausting available RAM before the consumer finishes. The channel capacity acts as an automatic flow-control valve — no explicit rate-limiting logic is needed.

9.3 Chunk Size Selection Guide

The default 64 KB chunk size was chosen empirically from the chunk size sensitivity benchmarks and balances three competing concerns. Changing it affects every stage in the pipeline, so the choice has system-wide implications:

  • Channel overhead. Each chunk requires one mpsc::send() + recv() pair per stage (~100 ns uncontended, more under contention). The total channel operations for a pipeline are (input_size / chunk_size) × num_stages. At 1 KB chunks, a 1 MB input through 5 stages generates 5,000 channel round-trips, costing 8.56 ms in overhead alone. At 64 KB, the same pipeline generates only 80 round-trips (1.50 ms). At 256 KB, 20 round-trips (1.62 ms) — the marginal improvement from 64 KB to 256 KB is only 8%, because channel overhead is no longer the bottleneck. This is why 64 KB sits at the knee of the curve: further increases yield diminishing returns.
  • First-byte latency. TTFB is approximately the time to fill one chunk at the source plus one channel hop per stage. Smaller chunks mean faster TTFB: at 1 KB, the first chunk arrives in ~50 µs; at 64 KB, ~94 µs; at 1 MB, TTFB equals total latency because the entire input is a single chunk and there is no pipeline overlap. For interactive use cases where the consumer begins rendering or transmitting as soon as the first bytes arrive, smaller chunks provide a measurably better experience — but below 16 KB, the channel overhead penalty outweighs the TTFB benefit.
  • Memory footprint. Per-stage buffer memory is channel_capacity × chunk_size. With the default capacity of 8: at 64 KB chunks, each stage buffers up to 512 KB; at 256 KB, 2 MB; at 1 MB, 8 MB. For a 5-stage pipeline, the total buffer ranges from 2.5 MB (64 KB chunks) to 40 MB (1 MB chunks). On memory-constrained workers, this 16× difference determines whether the pipeline fits in the available headroom.

Override the default only when you have a specific reason:

  • 16 KB — for latency-sensitive interactive responses where TTFB dominates (e.g., streaming a file preview to a web client). Accept the ~17% throughput penalty (1.76 ms vs 1.50 ms at 1 MB) for faster first-byte delivery.
  • 256 KB — for bulk ETL pipelines where throughput dominates and TTFB is irrelevant. Reduces channel operations by 4× compared to 64 KB, with a modest memory increase (2 MB per stage instead of 512 KB).
  • 1 MB+ — rarely justified. At this size, each "chunk" is large enough that the streaming benefit (pipeline overlap, bounded memory) is minimal. The pipeline degenerates toward batch behavior. Only consider this for single-stage pipelines on very large inputs where you want to minimize channel overhead to near zero.

The chunk size is set globally in PipelineConfig::chunk_size and applies to all stages uniformly. A future improvement (adaptive chunk sizing, §2 in future improvements) would allow per-stage chunk sizes based on observed throughput — but the uniform default works well for the common case because most streaming functions have similar per-chunk processing costs.

9.4 The Auto-Wrap Tax

Auto-wrapping (§5) lets any batch ComputeFunction participate in a streaming pipeline by collecting all input chunks into a single Bytes, running the batch function, and re-chunking the output via batch_to_stream. This preserves API compatibility — the 4 existing batch functions and any user-registered functions work without modification — but it defeats two key streaming benefits:

  • Bounded memory. The collect_stream step buffers the entire input before calling the batch function, reverting to O(n) memory for that stage. In a pipeline where 3 of 5 stages are auto-wrapped, the memory profile is a hybrid: the 2 streaming stages use O(chunk_size) memory, but the 3 batch stages each hold the full input. The worst case is a pipeline where every stage is auto-wrapped — at that point, the streaming infrastructure adds overhead (channels, tasks) with none of the memory benefits. This is strictly worse than running the pipeline in pure batch mode.
  • Pipeline parallelism. The wrapped stage cannot overlap with its predecessor because collect_stream waits for all input before starting the batch function. Downstream stages are blocked until the batch function completes and batch_to_stream begins emitting chunks. In a 5-stage pipeline where stage 3 is auto-wrapped, stages 1–2 run with full streaming overlap, then the pipeline stalls at stage 3 while it collects and executes, then stages 4–5 resume streaming from the re-chunked output. The stall duration equals the collection time plus the batch execution time — for a 64 MB input through a moderately expensive function, this could be 50–100 ms of dead time where no downstream progress occurs.

Auto-wrap is a migration tool, not a long-term solution. It exists to ensure that the streaming pipeline is a universal execution engine from day one — every function works, even if not optimally. For performance-critical pipelines, the path forward is implementing native StreamingComputeFunction variants. The 20 built-in streaming functions (§4) cover the most common operations, and the spawn_map / spawn_accumulate helpers make adding new ones straightforward (~15 lines per function). The registry resolution in §5.1 ensures that upgrading a function from batch to streaming is transparent: register the streaming variant, and all pipelines automatically use it on the next execution.

9.5 Accumulator Stages

Accumulators (SHA-256, byte count, checksum) are inherently non-streaming in their output — they must consume all input before emitting a result. A SHA-256 hash cannot produce any output bytes until the last input byte has been processed, because every input byte affects every output byte. However, accumulators still benefit from streaming input: instead of buffering the full input in memory, they update fixed-size internal state chunk-by-chunk (e.g., Sha256::update(chunk) updates 32 bytes of hasher state) and emit one summary chunk on StreamChunk::End. For a 512 MB input, the accumulator uses 32 bytes of state instead of 512 MB of buffer — a 16-million-fold reduction.

An accumulator in the middle of a pipeline creates a pipeline bubble: all downstream stages stall until the accumulator finishes consuming its entire input. Consider a pipeline [Uppercase → SHA-256 → Base64Encode] on a 64 MB input. The Uppercase stage streams chunks to SHA-256 with full overlap — while SHA-256 processes chunk N, Uppercase processes chunk N+1. But Base64Encode sees nothing until SHA-256 emits its 32-byte digest after all ~1,000 chunks have been processed. The pipeline's effective throughput is gated by the accumulator: the total latency is approximately time(Uppercase + SHA-256) + time(Base64Encode) rather than max(time(Uppercase), time(SHA-256), time(Base64Encode)).

Two strategies mitigate the bubble effect:

  • Place accumulators at the end. If the pipeline's purpose is to compute a hash or checksum, make the accumulator the final stage. This way, no downstream stage is blocked — the accumulator's output is the pipeline's output, and the consumer simply waits for the single summary chunk. The pipeline [Uppercase → Compress → SHA-256] has full streaming overlap between Uppercase and Compress, and the SHA-256 stage adds only its per-chunk update cost (~2 µs per 64 KB chunk) to the total latency.
  • Use tee-style functions for parallel metadata. StreamingTeeCount demonstrates the pattern: it passes all data chunks through unchanged while maintaining a rolling byte counter, then appends the count as an extra chunk after the last data chunk. This avoids the bubble entirely — downstream stages receive data immediately, and the metadata arrives at the end as a bonus. The same pattern can be extended to StreamingTeeChecksum or StreamingTeeSha256 (not yet implemented) for computing integrity metadata without blocking the data flow.

The accumulator tradeoff is fundamental and cannot be eliminated by engineering — it is an information-theoretic constraint. A hash function that could emit partial output before seeing all input would not be a cryptographic hash. The streaming system's contribution is making the input side efficient (bounded memory, backpressure) while accepting that the output side is necessarily delayed. This is strictly better than the batch alternative, which buffers both input and output.

12. References

  • [1] Ghemawat, S., Gobioff, H., & Leung, S.-T. (2003). The Google File System. SOSP '03. — The foundational distributed file system paper. Treats all data as opaque bytes with path-based addressing.
  • [2] Shvachko, K., Kuang, H., Radia, S., & Chansler, R. (2010). The Hadoop Distributed File System. MSST '10. — Open-source implementation of GFS principles. No concept of derived data or computation.
  • [3] Benet, J. (2014). IPFS - Content Addressed, Versioned, P2P File System. arXiv:1407.3561. — Content-addressed distributed storage using Merkle DAGs. Addresses data by content hash but has no computation model.
  • [4] Dolstra, E., de Jonge, M., & Visser, E. (2004). Nix: A Safe and Policy-Free System for Software Deployment. LISA '04. — Computation-addressed package management. Packages are addressed by hash(build_recipe + all_inputs). Single-node, eager evaluation.
  • [5] Zaharia, M., Chowdhury, M., Das, T., Dave, A., Ma, J., McCauley, M., Franklin, M. J., Shenker, S., & Stoica, I. (2012). Resilient Distributed Datasets: A Fault-Tolerant Abstraction for In-Memory Cluster Computing. NSDI '12. — RDD lineage for fault tolerance via recomputation. Ephemeral, scoped to a single Spark job.
  • [6] Mokhtar, S. A. & Vlassov, V. (2019). A Survey on Content-Addressable Storage. — Comprehensive survey of CAS systems, deduplication techniques, and hash-based addressing schemes.
  • [7] Bazel Build System. Google. https://bazel.build/ — Content-addressed build caching with remote cache support. Scoped to build artifacts.
  • [8] Buck2 Build System. Meta. https://buck2.build/ — Next-generation build system with content-addressed outputs and remote execution.
  • [9] Pachyderm. https://www.pachyderm.com/ — Data versioning and pipeline platform with provenance tracking. Full platform, not a pure storage layer.
  • [10] Apache Atlas. https://atlas.apache.org/ — Data governance and metadata management framework. Lineage as external metadata, not integrated with storage.
  • [11] OpenLineage. https://openlineage.io/ — Open standard for data lineage collection. Metadata-only, no storage integration.
  • [12] Dagster. https://dagster.io/ — Data orchestrator with asset materialization model. Memoization at the orchestrator level.
  • [13] Tonic — gRPC for Rust. https://github.com/hyperium/tonic — The gRPC framework used by Deriva's server and client.
  • [14] Sled — Embedded Database. https://github.com/spacejam/sled — Pure Rust embedded database used for recipe storage.
  • [15] Tokio — Async Runtime for Rust. https://tokio.rs/ — The async runtime powering Deriva's server.
  • [16] Wasmtime — WebAssembly Runtime. https://wasmtime.dev/ — Planned WASM sandbox for user-defined compute functions in Phase 4.
  • [17] Das, A., Gupta, I., & Motivala, A. (2002). SWIM: Scalable Weakly-consistent Infection-style Process Group Membership Protocol. DSN '02. — The gossip protocol planned for Deriva's Phase 3 node discovery.
  • [18] Deriva Source Code. https://github.com/mehmet-ylcnky/Deriva — The complete implementation described in this paper.
